# Руководство по настройке вебхуков

> **Аудитория**: Администраторы, интегрирующие внешние сервисы с shittim-chest.
> **Последнее обновление**: 2026-05-25

## Обзор

Вебхуки позволяют внешним сервисам (GitHub, GitLab, Gitee) отправлять события в реальном времени в shittim-chest. События проверяются, разбираются и пересылаются в scepter, который направляет их соответствующему агенту.

```text
Внешний сервис → shittim_chest → scepter → Агент
```

`shittim_chest` также поддерживает пользовательские конечные точки вебхуков для сервисов, не поддерживаемых нативно.

## Настройка вебхука GitHub

### Шаг 1: Настройка окружения

Установите секрет вебхука в вашем `.env`:

```bash
WEBHOOK_GITHUB_SECRET=your-hmac-secret-here
WEBHOOK_PUBLIC_URL=https://your-domain.com
```

Сгенерируйте надёжный секрет:

```bash
openssl rand -hex 32
```

### Шаг 2: Создание вебхука в GitHub

1. Перейдите в ваш репозиторий → **Settings** → **Webhooks** → **Add webhook**
1. Установите **Payload URL** в `https://your-domain.com/api/webhook/github`
1. Установите **Content type** в `application/json`
1. Установите **Secret** в то же значение, что и `WEBHOOK_GITHUB_SECRET`
1. Выберите события: `push`, `pull_request`, `issues`, `issue_comment`
1. Убедитесь, что **Active** отмечен
1. Нажмите **Add webhook**

### Шаг 3: Проверка

GitHub немедленно отправит событие `ping`. Проверьте вкладку **Recent Deliveries**, чтобы подтвердить ответ `200`.

## Настройка вебхука GitLab

### Шаг 1: Настройка окружения

```bash
WEBHOOK_GITLAB_SECRET=your-gitlab-secret-token
```

### Шаг 2: Создание вебхука в GitLab

1. Перейдите в ваш проект → **Settings** → **Webhooks**
1. Установите **URL** в `https://your-domain.com/api/webhook/gitlab`
1. Установите **Secret token** в то же значение, что и `WEBHOOK_GITLAB_SECRET`
1. Выберите триггеры: `Push events`, `Merge request events`, `Issue events`
1. Убедитесь, что **Enable SSL verification** отмечен (для HTTPS)
1. Нажмите **Add webhook**

### Шаг 3: Проверка

Используйте кнопку **Test** в GitLab для отправки тестового события. Подтвердите, что доставка успешна.

## Настройка вебхука Gitee

Вебхуки Gitee (码云) также поддерживаются.

### Шаг 1: Настройка окружения

Gitee использует тот же `WEBHOOK_GITLAB_SECRET` для проверки HMAC (с токеном как запасным вариантом). Альтернативно, установите `WEBHOOK_GITEE_PASSWORD` при использовании аутентификации на основе пароля.

### Шаг 2: Создание вебхука в Gitee

1. Перейдите в ваш репозиторий → **Management** → **Webhooks**
1. Установите **URL** в `https://your-domain.com/api/webhook/gitee`
1. Установите **Password/Signing Key** в тот же секрет
1. Выберите события: `Push`, `Pull Request`, `Issues`
1. Нажмите **Add**

## Пользовательские вебхуки

`shittim_chest` поддерживает общую пользовательскую конечную точку вебхука на `/api/webhook/custom/{name}`. Чтобы добавить пользовательский источник вебхука:

1. Установите `WEBHOOK_PUBLIC_URL` в `.env`
1. Настройте ваш внешний сервис на POST по адресу `https://your-domain.com/api/webhook/custom/{name}`
1. События пересылаются в scepter с именем вебхука в качестве источника события

Для интеграции новых провайдеров вебхуков на уровне кода:

1. Добавьте обработчик в `packages/core/src/webhook.rs`
1. Реализуйте проверку HMAC или токена для нового провайдера
1. Разберите пользовательский формат события и перешлите в scepter через Unix-сокет

## Белый список IP

`shittim_chest` поддерживает белый список IP для источников вебхуков для отклонения запросов от неизвестных источников:

```bash
# .env
WEBHOOK_IP_WHITELIST=140.82.112.0/20,192.30.252.0/22  # IP GitHub
```

Настройте диапазоны CIDR для каждого провайдера вебхуков. Запросы от IP вне белого списка отклоняются.

## Типы событий

Поддерживаемые события и их сопоставление с триггерами scepter:

| Источник | Событие | `event_type` scepter |
| --- | --- | --- |
| GitHub | `push` | `github.push` |
| GitHub | `pull_request` | `github.pull_request` |
| GitHub | `issues` | `github.issues` |
| GitHub | `issue_comment` | `github.issue_comment` |
| GitLab | `push` | `gitlab.push` |
| GitLab | `merge_request` | `gitlab.merge_request` |
| GitLab | `issues` | `gitlab.issues` |
| Gitee | `push` | `gitee.push` |
| Gitee | `pull_request` | `gitee.pull_request` |
| Gitee | `issues` | `gitee.issues` |

## Журнал доставок

`shittim_chest` ведёт журнал доставок событий вебхуков. Дублирующиеся доставки обнаруживаются с помощью LRU-кеша (до 10,000 ID доставок). Доступ к журналам доставок через:

- **REST API**: `GET /api/webhook/deliveries`
- Панель администратора: **Webhooks** → **Delivery Log**

## Безопасность

Все вебхуки должны проходить проверку подписи:

- **GitHub**: Использует заголовок `X-Hub-Signature-256`. Проверяется по `WEBHOOK_GITHUB_SECRET`.
- **GitLab**: Использует заголовок `X-Gitlab-Token`. Проверяется по `WEBHOOK_GITLAB_SECRET`.
- **Gitee**: Использует подпись HMAC-SHA256 с запасным вариантом токена.

Запросы без действительных подписей отклоняются с `401 Unauthorized`. Никогда не раскрывайте секреты вебхуков в клиентском коде или логах.

## Тестирование

Используйте панель администратора для тестирования интеграции вебхуков:

1. Войдите в панель администратора (по умолчанию `:3000`)
1. Перейдите в **Webhooks** в боковой панели
1. Просмотрите журналы доставок и конфигурацию
1. Протестируйте конечные точки через функциональность тестирования внешнего сервиса

Вы также можете протестировать вручную с помощью curl:

```bash
curl -X POST https://your-domain.com/api/webhook/github \
  -H "Content-Type: application/json" \
  -H "X-Hub-Signature-256: sha256=<computed-hmac>" \
  -d '{"action":"push","ref":"refs/heads/main"}'
```

## Устранение неполадок

### 401 Unauthorized

**Причина**: Несовпадение подписи HMAC или IP не в белом списке.
**Исправление**: Убедитесь, что секрет в `.env` совпадает с секретом, настроенным на исходной платформе. Проверьте наличие пробелов в конце или проблем с кодировкой. Проверьте конфигурацию белого списка IP.

### 502 Bad Gateway

**Причина**: scepter недоступен.
**Исправление**: Проверьте `ENTELECHEIA_SCEPTER_URL` и `ENTELECHEIA_TUI_SOCK` в `.env`. Убедитесь, что экземпляр scepter работает и путь Unix-сокета доступен.

### События не достигают агентов

**Причина**: Тип события не сопоставлен или агент не настроен на его обработку.
**Исправление**: Проверьте логи бэкенда на наличие разобранного `event_type`. Убедитесь, что целевой агент имеет зарегистрированный обработчик для этого события. Проверьте журнал доставок через API или панель администратора.

### Дублирующиеся доставки

**Причина**: Внешний сервис повторяет попытку из-за таймаута. `shittim_chest` автоматически обнаруживает дубликаты через LRU-кеш.
**Исправление**: Если действительные повторные попытки блокируются, увеличьте размер кеша ID доставок. Убедитесь, что `shittim_chest` отвечает в пределах окна таймаута сервиса (GitHub: 10 секунд).
