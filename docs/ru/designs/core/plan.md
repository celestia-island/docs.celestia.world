# План автономной интеграции промышленного коридора

> **Цель**: Система должна демонстрировать **автономное само-взаимодействие** с полностью
> неизвестным промышленным демонстрационным коридором — обнаруживая оборудование, выводя модели данных,
> генерируя конфигурацию мониторинга и замыкая цикл тревога→ответ — без
> ручного инжиниринга для каждого устройства.
> **Государственный жёсткий дедлайн**: эта возможность привязана к вехе государственного проекта.

-----------------------------------------------------------------------------

## Оставшаяся работа

Полная цепочка обнаружение → вывод → мониторинг → тревога → **подтверждение записи**
реализована (Фазы A.1–A.3, B, C, D.1, **D.2 ✓**). Единственная оставшаяся работа — это
**сквозная проверка dogfood (Фаза E)** — эксплуатационная, не кодовая.

### D.2 — Полный цикл подтверждения записи (human-in-the-loop) ✓

```text
Агент решает, что нужна запись
  → verify_write_safety → Отказано
    → orexis.request_write_approval → WriteApprovalRequest broadcast
      → shittim-chest показывает диалог подтверждения (industrial.approveWrite)
        → [одобрено] → временная запись в белом списке → выполнить + проверка read-back
        → [отклонено] → агент получает отказ, корректирует план
```

**Реализовано:**

| # | Задача | Файл | Статус |
| --- | --- | --- | --- |
| A.2.4.1 | Инструмент MCP `orexis.request_write_approval` — создаёт `WriteApprovalRequest`, транслирует `TuiMessage::IndustrialWriteApprovalPush`, приостанавливает (oneshot + таймаут) до ответа оператора | `packages/agents/orexis/src/mcp/tools/industrial_write_tools.rs` | ✓ |
| A.2.4.2 | Обработчик WS `industrial.approveWrite` — разрешает ожидающий запрос через общий `WriteApprovalRegistry`; при одобрении добавляет временную запись в белом списке, чтобы последующая запись прошла `verify_write_safety` | `packages/scepter/src/tui_connection/mod.rs` | ✓ |

Производитель/разрешитель развязаны через общепроцессный общий
`WriteApprovalRegistry` (`_shared_security_policy::write_approval_registry`),
внедряемый в orexis при запуске и используемый scepter при ответе оператора.

## 1. Адресация Sub-Badge + Параллельное выполнение навыков

**Вердикт**: Стоит реализовать. Инфраструктура готова ~80%, не хватает только последних 20%.

**Текущее состояние**:

- `BadgeRegistry` (`packages/scepter/src/state_machine/badge_registry.rs:92-120`) уже поддерживает родительско-дочерние `link_sessions()`.
- Парсинг синтаксиса sub-badge `#001.005` существует в `find_by_container_id_or_sub()`, но отбрасывает sub-номер вместо разрешения в отдельный дочерний контейнер.
- Поля `SnowflakeContainer.parent_id` и `branch_level` существуют, но только как метаданные — никогда не используются для маршрутизации.
- Приоритетная очередь граничных узлов (`edge_node_registry.rs:73-126`) готова для тонкой блокировки ресурсов.
- Цепочка навыков строго **последовательная** — `pipeline.rs:68-226` выполняет один навык за раз. Навыки координатора с независимыми `next_targets` выполняются последовательно, когда могли бы выполняться параллельно.

**Чего не хватает**:

1. ✅ Сделать так, чтобы `find_by_container_id_or_sub()` разрешала `#001.005` → самый глубокий

активный форкнутый дочерний элемент родительского контейнера, с откатом к родителю, когда
форка не существует (обратно совместимо).

1. ✅ Добавить поиск дочерних/потомков в `SnowflakeManager`: `children_of`,

`children_of_badge`, `most_recent_child_of`, `deepest_descendant`
(`parent_id` → обратный индекс).

1. ✅ Параллельное выполнение `next_targets` на основе `FuturesUnordered`:

`dispatch_parallel_targets` разветвляет независимые **листовые**
цели координатора конкурентно через `parallel_dispatch::fan_out` (ограничено
`Semaphore`). Два глобально-синглтонных блокировщика в последовательном
пути `invoke_skill_with_retries` обрабатываются следующим образом:

- **Общее локальное пространство имён cosmos** → каждая цель форкается в свой **собственный

контейнер cosmos** в Фазе 1 (`fork_container_for_skill` +
`assign_container_id` + `register_container_badge_in_registry`), поэтому
`dump/restore_cosmos_namespace` — это no-op для каждой ветки, и конкурентное выполнение
изолировано. `MAX_BRANCH_DEPTH` (пункт 4) ограничивает цепочку форков.

- **Гонка `active_streaming_skill` UI** → допустима (последний писатель побеждает на

`Option`; сброс в `None` после каждой ветки).

- **Потоковая безопасность `&mut SkillChainInput`** → `BranchOwner` зеркалирует мутабельные

части для каждой ветки; `as_input` заимствует их обратно в короткоживущий
`SkillChainInput`, чтобы неизменённые помощники конвейера использовались повторно.
Фаза 1 (форк + подготовка + построение промптов + белый список инструментов) **сериализована**
для избежания гонок `rag_buffer`; только Фаза 2 (доминирующие по задержке вызовы
LLM) выполняется параллельно; Фаза 3 очищает и объединяет отчёты
(`merge_branch_reports`) в родительский контекст. Закрыто за
`SKILL_CHAIN_PARALLEL_TARGETS` (по умолчанию **выкл**) +
`parallel_targets_eligible` (контейнеризованные + все листовые цели). Последовательная
размотка стека в `route_to_next_skill` остаётся по умолчанию.

1. ✅ Принудительно применять `MAX_BRANCH_DEPTH` (`COSMOS_MAX_BRANCH_DEPTH`, по умолчанию 4) в обоих

путях форка; дочерние элементы теперь регистрируются на `source.branch_level + 1` вместо
жёстко закодированного `1`.

**Ожидаемое влияние**: Параллельная запись файлов, параллельный анализ от навыков координатора, таких как `industrial_discover`, значительно снизили бы сквозную задержку.

## 2. Конвейер седиментации памяти

**Вердикт**: Множитель качества, не критичен. Зарезервирован для долгосрочной дорожной карты.

**Текущее состояние**:

- `PhiliaMemoryService` — это плоский граф «сохранить → эмбеддинг → извлечь» без метаболизма.
- `memory_consolidate` тривиален — просто создаёт узел эпизода, без абстракции/суммаризации.
- Нет затухания памяти, старения, оценки устаревания или градиента качества между узлами.
- Все узлы — недифференцированные `MemoryNode` — без разделения на эпизодические/процедурные/атомарные.
- Векторный поиск в памяти — O(n) полный перебор (не будет масштабироваться в долгосрочной перспективе).
- `KnowledgeStore` (отдельная система) имеет стадии жизненного цикла (Created→Vectorized→Searchable→Consolidated→Deprecated) и проверку консенсусом — это ближайший существующий аналог седиментации.

**Почему это не срочно**:

- Внедрение контекста RAG (`RagContextBuffer` → перезапись запроса LLM → `bundle_search`) обеспечивает достаточный контекст для текущих агентов вызова инструментов.
- Индекс pgvector HNSW обрабатывает поиск производственного масштаба.
- Система работает как «сохранить и извлечь» — седиментация сделала бы это «метаболизмом», но это инкрементальное качество, а не функциональный пробел.

**Будущая работа** (без временной шкалы):

- Автоконсолидация: периодическая суммаризация связанных узлов в более высокоуровневые «эпизоды» с помощью LM.
- Градиент качества: счётчики доступа, временное затухание, оценка уверенности.
- Трёхканальный прототип (эпизодический/процедурный/атомарный) с дифференцированными стратегиями извлечения.

## 3. Межагентные переговоры

**Вердикт**: Низкий приоритет. Примитивы существуют как низкоуровневые строительные блоки; нет непосредственного сценария использования.

**Текущее состояние**:

- `deliver_message(message_type="Question")` существует (`epieikeia/src/mcp/tools/deliver_message.rs:63`) — может отправлять вопросы в почтовый ящик другого агента.
- `inject_user_prompt` / `consume_injected_prompts` существуют, но **на основе опроса** — без интеграции в конвейер. Агенты должны явно вызывать `consume_injected_prompts` для проверки почты.
- `Haplotes` имеет типы маршрутизации разговоров `AskAgent` / `ReplyAgent` / `Escalated` — но все они являются no-op ACK с нулевой бизнес-логикой.
- Переменные окружения `NEGOTIATION_ROUND_TIMEOUT_SECS` / `NEGOTIATION_TOTAL_TIMEOUT_SECS` определены в `RuntimeTuningConfig`, но **никогда не используются** — мёртвый код.

**Почему это низкий приоритет**:

- Текущая последовательная диспетчеризация цепочек навыков + передача контекста как строки обрабатывает все текущие сценарии использования.
- Конфликты слияния обрабатываются диспетчеризацией одного навыка (`resolve_merge_conflict`), что достаточно.
- Цикл переговоров (перехват цепочки навыков → спросить агента → ждать ответа → включить) был бы сложным в построении и тестировании. Ни один производственный сценарий пока этого не требует.

**Когда вернуться**: Если агентам когда-либо понадобится динамически договариваться о решениях в середине цепочки (а не просто диспетчеризовать и ждать), примитивы готовы на 40%. Пробел — в цикле интеграции конвейера.
