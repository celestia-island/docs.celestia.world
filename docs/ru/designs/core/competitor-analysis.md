## 1. CrewAI

**Репозиторий**: [crewAIInc/crewAI](https://github.com/crewAIInc/crewAI)
**Язык**: Python
**Лицензия**: MIT
**Размер**: ~23k+ звёзд. Независим от LangChain.

### Архитектура

- **Агенты**: Определяются через YAML (роль, цель, предыстория) или класс Python `Agent`. Каждый агент оборачивает LLM с доступом к инструментам.
- **Оркестровка**: Два режима:
  - **Crews**: Команда агентов с последовательным или иерархическим процессом. Последовательный запускает задачи по порядку; иерархический назначает агента-"менеджера" для делегирования.
  - **Flows**: Управляемый событиями DAG с декораторами `@start`, `@listen`, `@router`. Состояние с типизацией Pydantic. Поддерживает комбинаторы условий `and_`/`or_`.
- **Коммуникация**: Передача сообщений через среду выполнения Crew/Flow. Агенты производят структурированный вывод (`output_pydantic`, `output_json`).
- **Типы процессов**: `Process.sequential` и `Process.hierarchical`.

### Предоставление Инструментов

- Встроенные инструменты через пакет `crewai[tools]` (SerperDev и др.). Пользовательские инструменты как функции Python.
- Поддержка MCP (Model Context Protocol) задокументирована.
- Инструменты назначаются на агента во время определения.
- Нет явного ограничения на количество инструментов на вызов LLM — все назначенные инструменты предоставляются в каждом ходе.

### Модель Безопасности

- **Нет изоляции**. Агенты работают в том же процессе Python, что и оркестровка.
- Корпоративный "AMP Suite" предлагает плоскость управления с наблюдаемостью и контролем доступа (проприетарный).
- Человек-в-цикле через `human_input=True` на задачах.
- Изоляция выполнения кода не упоминается.

### Память/Контекст

- Краткосрочная: Память агента через историю разговора.
- Долгосрочная: Опциональные хранилища памяти, включаемые через `memory=True` на агентах.
- Нет явного сжатия контекста или управления токенами — полагается на контекстное окно LLM.
- Контрольные точки упоминаются в документации, но детали в OSS скудны.

### Уникальные Особенности

- **Синергия Flows + Crews**: Комбинирует автономные команды агентов с точными событийно-управляемыми рабочими процессами.
- **Конфигурация на основе YAML**: Агенты и задачи определяются декларативно, подходит для не-разработчиков.
- **Большое сообщество**: 100k+ сертифицированных разработчиков через `learn.crewai.com`.
- **Заявления о производительности**: В 5.76x быстрее, чем LangGraph в определённых задачах QA (самостоятельно заявлено).

### Потенциальные Пробелы Относительно Проектных Целей Entelecheia

- Нет изоляции или песочницы для выполнения кода.
- Нет формальной модели безопасности для выполнения инструментов.
- Память относительно проста — нет иерархического управления контекстом или архивирования.
- Однопроцессная среда выполнения Python ограничивает масштабируемость между машинами.
- Нет нативной интеграции браузера/оболочки для задач кодирования агентов.
- Оркестровка только на Python (нет мультиязычной среды выполнения).

## 2. LangGraph

**Репозиторий**: [langchain-ai/langgraph](https://github.com/langchain-ai/langgraph)
**Язык**: Python (также JS/TS через `langgraphjs`)
**Лицензия**: MIT

### Архитектура

- **Графовая машина состояний**: Узлы (агенты/функции) и рёбра (переходы) образуют направленный граф в стиле Pregel/Beam.
- **Агенты**: Узлы могут быть вызовами LLM, выполнениями инструментов или любыми функциями Python. Не строго типизированы как "агенты" — скорее как функции в графе.
- **Оркестровка**: `StateGraph` с типизированной схемой состояния. Узлы читают/записывают состояние. Условные рёбра для ветвления. Подграфы для композиции.
- **Коммуникация**: Объект состояния является единственным источником истины. Сообщения добавляются в списки состояния.

### Предоставление Инструментов

- Инструменты — это инструменты LangChain или произвольные вызываемые объекты, привязанные к узлам графа.
- Все инструменты, доступные в узле, предоставляются LLM в этом шаге.
- Нет встроенного ограничения инструментов; разработчики управляют тем, какие инструменты передаются на узел.

### Модель Безопасности

- **Нет изоляции**. Выполнение кода — ответственность разработчика.
- Человек-в-цикле через `interrupt()` — приостанавливает выполнение графа, позволяет проверку/изменение состояния.
- Долговременное выполнение: состояние сохраняется, может возобновляться после сбоев (контрольные точки).
- Нет примитивов изоляции для выполнения инструментов или доступа к файловой системе.

### Память/Контекст

- **Краткосрочная**: Рабочая память через состояние (списки сообщений).
- **Долгосрочная**: Постоянная память между сессиями через абстракцию `Store` (ключ-значение с эмбеддингами).
- Сжатие контекста не встроено — разработчики управляют размером состояния.
- Контрольные точки через `MemorySaver` или `SqliteSaver`.

### Уникальные Особенности

- **Долговременное выполнение**: Автоматически возобновляется с контрольной точки после сбоев/таймаутов — идеально для долго работающих агентов.
- **Человек-в-цикле через прерывания**: Мощный шаблон для рабочих процессов утверждения.
- **Интеграция с LangSmith**: Глубокая наблюдаемость, трассировка и оценка.
- **LangSmith Deployment**: Платформа промышленного развёртывания с визуальным прототипированием.
- **Deep Agents**: Новый подпроект для агентов, которые планируют, используют подагентов и файловые системы.
- **Экосистема LangChain**: Бесшовная интеграция с инструментами, моделями и компонентами LangChain.

### Потенциальные Пробелы Относительно Проектных Целей Entelecheia

- Тесно связан с экосистемой LangChain (хотя "может использоваться без LangChain").
- Нет модели безопасности — нет изоляции, нет системы разрешений.
- Низкоуровневый фреймворк — требует значительного шаблонного кода для взаимодействий агентов.
- Нет нативного протокола мульти-агентной коммуникации (A2A).
- Подход на основе графа состояний может стать громоздким для сложных взаимодействий агентов.
- Нет встроенной поддержки изоляции выполнения кода.

## 3. MetaGPT

**Репозиторий**: [`FoundationAgents`/MetaGPT](https://github.com/`FoundationAgents`/MetaGPT)
**Язык**: Python
**Лицензия**: MIT
**Исследование**: Опубликовано на ICLR 2024

### Архитектура

- **Мульти-агентная на основе SOP**: Моделирует компанию-разработчика ПО с предопределёнными ролями (PM, Архитектор, Инженер и т.д.).
- **Агенты (Роли)**: Каждая `Role` имеет профиль, цель, ограничения и набор `Action`. Роли используют циклы ReAct (думать → действовать) с тремя режимами: `REACT`, `BY_ORDER`, `PLAN_AND_ACT`.
- **Оркестровка**: Класс `Team` нанимает роли, инвестирует бюджет, запускает раунды. `Environment` управляет передачей сообщений между ролями через публикацию-подписку.
- **Коммуникация**: Публикация/подписка на основе сообщений через Environment. Роли подписываются на определённые теги сообщений.
- **Основная философия**: `Code = SOP(Team)` — материализованные Стандартные Операционные Процедуры.

### Предоставление Инструментов

- Действия — это предопределённые классы Python (`WriteCode`, `DesignAPI`, `DebugError` и т.д.) — ~40+ типов действий.
- Каждая роль получает определённые действия, назначенные при создании.
- Инструменты включают: поисковые системы (Serper, SerpAPI, DuckDuckGo, Google, Bing), браузеры (Playwright, Selenium), генерацию изображений (DALL-E), хранилища документов (Chroma, FAISS, Milvus, LanceDB, Qdrant).
- LLM видит только схемы действий для своих текущих назначенных действий, а не полный набор инструментов.

### Модель Безопасности

- **Нет изоляции**. Код генерируется и выполняется в той же среде.
- Dockerfile предоставлен, но для развёртывания, а не изоляции на задачу.
- Отслеживание бюджета: параметр `investment` ограничивает общую стоимость API LLM, вызывает исключение при превышении.
- Нет человека-в-цикле во время выполнения.
- Нет изоляции выполнения инструментов или модели разрешений.

### Память/Контекст

- **Краткосрочная**: Класс `Memory` в `RoleContext` — упорядоченный список сообщений на роль.
- **Долгосрочная**: Классы `LongTermMemory` и `BrainMemory` для постоянных знаний.
- **Рабочая память**: Отдельная рабочая память для операций планировщика.
- **Буфер сообщений**: Асинхронная очередь сообщений с фильтрацией по подписанным тегам.
- Сжатие контекста явно не обрабатывается — роли наблюдают отфильтрованное подмножество сообщений.

### Уникальные Особенности

- **Полная эмуляция SDLC**: Моделирует всю компанию-разработчика ПО с SOP — пользовательские истории, требования, проектные документы, код, тесты.
- **Множество хранилищ документов**: 5+ вариантов векторных БД.
- **Обширная поддержка провайдеров**: 12+ провайдеров LLM (OpenAI, Azure, Anthropic, Gemini, Ollama, Bedrock и др.).
- **Data Interpreter**: Специализированный агент для задач науки о данных.
- **Исследовательский выход**: Множество опубликованных статей (AFlow, SPO, SELA, FACT, Data Interpreter).
- **MGX**: Продукт программирования на естественном языке, построенный сверху.

### Потенциальные Пробелы Относительно Проектных Целей Entelecheia

- Жёсткая SOP — роли и действия предопределены; пользовательские роли требуют кодирования.
- Полное отсутствие изоляции безопасности.
- Одномашинная архитектура — нет распределённого развёртывания агентов.
- Нет управления агентами на основе браузера (только CLI/API).
- Модель памяти довольно базовая — очереди на роль.
- Нет поддержки протокола MCP/A2A между агентами.
- Отслеживание бюджета только по стоимости, не на основе ресурсов/безопасности.

## 4. ChatDev 2.0 (DevAll)

**Репозиторий**: [OpenBMB/ChatDev](https://github.com/OpenBMB/ChatDev)
**Язык**: Python (бэкенд), Vue 3 (фронтенд)
**Лицензия**: Apache 2.0
**Исследование**: Множество статей NeurIPS/arxiv

### Архитектура

- **Мульти-агентная платформа без кода**: Агенты и рабочие процессы определяются полностью в конфигурации YAML. Кодирование не требуется.
- **YAML-управляемый DAG рабочих процессов**: Узлы определяют агентов, рёбра определяют поток сообщений. Поддерживает подграфы. Визуальный холст перетаскивания в веб-интерфейсе.
- **Основные модули**: `runtime/` (выполнение агентов), `workflow/` (оркестровка DAG), `entity/` (конфигурация), `server/` (FastAPI + WebSocket), `frontend/` (веб-консоль Vue 3).
- **Агенты**: Определяются в конфигурациях узлов YAML с промптами, конфигурацией LLM, инструментами и настройками памяти.
- **Оркестровка**: Несколько типов исполнителей: последовательный, DAG, параллельный, цикл, динамическое ребро. Построитель топологии преобразует конфигурации YAML в исполняемые графы.

### Предоставление Инструментов

- **Система вызова функций**: `functions/function_calling/` содержит встроенные инструменты (`code_executor`, file, weather, web, video, `deep_research`, uv, user).
- **Регистрация пользовательских инструментов**: Функции Python в директории `functions/` автоматически обнаруживаются.
- **Поддержка MCP**: `mcp_example/mcp_server.py` демонстрирует интеграцию MCP.
- Инструменты назначаются на узел в конфигурации YAML.

### Модель Безопасности

- **Выполнение кода**: Специализированный `code_executor.py` с настраиваемыми параметрами выполнения.
- Доступно развёртывание Docker Compose.
- **Человек-в-цикле**: Рабочий процесс `demo_human.yaml`, узлы ввода пользователя, потоки подтверждения.
- **Потоковая передача WebSocket**: Мониторинг логов в реальном времени и проверка артефактов.
- Нет явной модели изоляции/песочницы для агентов.

### Память/Контекст

- **Множество бэкендов памяти**: Простая память, память `mem0` (постоянная, обучаемая), файловая память.
- **Конфигурация памяти в YAML**: `store`, `context_window_size`, тип памяти на узел.
- **Узлы сброса контекста**: Явные узлы `context_reset` в рабочем процессе.
- **Согласованность эмбеддингов памяти**: Тесты на согласованность эмбеддингов между сессиями.
- **Сканирование рабочего пространства**: `workspace_scanner.py` для внедрения контекста на основе файлов.

### Уникальные Особенности

- **Без кода**: Создание мульти-агентных систем без написания кода Python — YAML + веб-интерфейс.
- **Веб-консоль перетаскивания**: Визуальный дизайнер рабочих процессов, мониторинг запуска в реальном времени.
- **Богатые шаблоны рабочих процессов**: Визуализация данных, 3D-генерация (Blender), разработка игр, глубокое исследование, обучающее видео.
- **Поддержка подграфов**: Многоразовые подпроцессы (`react_agent.yaml`, `reflexion_loop.yaml`).
- **Интеграция с OpenClaw**: Может вызываться агентами кодирования OpenClaw для динамического создания команд агентов.
- **Python SDK**: Пакет PyPI `chatdev` для программного выполнения рабочих процессов.
- **Исследовательская линия**: MacNet, Puppeteer, IER, экспериментальное совместное обучение — всё построено на ChatDev.
- **Мульти-агентная электронная книга**: Курируемая коллекция мульти-агентных исследований.

### Потенциальные Пробелы Относительно Проектных Целей Entelecheia

- Ориентация на YAML ограничивает выразительность для сложной логики.
- Нет изоляции безопасности для выполнения кода.
- Веб-консоль является основным интерфейсом — менее подходит для безголового/встроенного использования.
- Django-подобный: предписывающая структура проекта, менее гибкая, чем нативные фреймворки Python.
- Нет поддержки кросс-языковых агентов.
- Сообщество ориентировано на исследования, а не на предприятия.

## 5. Google ADK (Agent Development Kit)

**Репозиторий**: [google/adk-python](https://github.com/google/adk-python)
**Язык**: Python (также издания Java, Go)
**Лицензия**: Apache 2.0

### Архитектура

- **Code-first**: Агенты, инструменты и оркестровка определяются в коде Python.
- **Основные абстракции**: Agent (чертёж), Tool (возможность), Runner (движок), Session (состояние разговора), Memory (межсессионный вызов), Artifact Service (файлы).
- **Типы агентов**: `LlmAgent` (управляемый LLM), `LoopAgent`, `SequentialAgent`, `ParallelAgent`, `RemoteA2aAgent`.
- **Мульти-агентная композиция**: Родительский агент имеет список `sub_agents`. Runner обрабатывает маршрутизацию между агентами.
- **Протокол A2A**: Нативная поддержка протокола Agent-to-Agent для удалённых агентов.
- **Интеграция с LangGraph**: `langgraph_agent.py` для встраивания графов LangGraph как агентов.

### Предоставление Инструментов

- **Богатая экосистема инструментов**: 50+ встроенных инструментов — Google Search, BigQuery, Bigtable, Spanner, PubSub, Vertex AI Search, инструменты MCP, инструменты OpenAPI, инструменты LangChain, инструменты CrewAI, использование компьютера, bash, выполнение кода, Google API.
- **Типы инструментов**: `FunctionTool`, `AgentTool` (обёртка агента как инструмента), `MCPTool`, `OpenAPITool`, `LangChainTool`, `CrewAiTool`, `SkillToolset`.
- **Подтверждение инструментов (HITL)**: Явный поток подтверждения перед выполнением инструмента с пользовательским вводом.
- **Шаблон Toolbox**: `toolbox_toolset.py` для объединения инструментов.

### Модель Безопасности

- **Изоляция выполнения кода**: Множество исполнителей — `container_code_executor.py`, `unsafe_local_code_executor.py`, `vertex_ai_code_executor.py`, `agent_engine_sandbox_code_executor.py`, `gke_code_executor.py`.
- **Система аутентификации**: Полный поток OAuth2, управление учётными данными, предобработчики аутентификации, `authenticated_function_tool.py`.
- **Человек-в-цикле**: Подтверждение инструментов, поддержка прерываний.
- **Навыки**: Упаковываемые, версионируемые возможности агентов с отдельным контекстом выполнения.
- **Идентичность агента**: Интеграция `agent_identity/` для сервисных аккаунтов.

### Память/Контекст

- **Сессия**: Полная история разговора на сессию, сохраняется через `SessionService` (в памяти, SQLite, PostgreSQL, Vertex AI).
- **Память**: Межсессионный вызов через `MemoryService` — в памяти, Vertex AI Memory Bank, Vertex AI RAG.
- **Сжатие контекста**: `compaction.py` и `llm_event_summarizer.py` для автоматического суммирования контекста.
- **Сервис артефактов**: Отдельная обработка нетекстовых данных (файлы, изображения).
- **Кэширование контекста**: `gemini_context_cache_manager.py` для кэширования контекста Gemini.

### Уникальные Особенности

## 6. OpenAI Swarm (экспериментальный, ныне устаревший)

**Репозиторий**: [openai/swarm](https://github.com/openai/swarm)
**Язык**: Python
**Лицензия**: MIT
**Статус**: Заменён на [OpenAI Agents SDK](https://github.com/openai/openai-agents-python)

### Архитектура

- **Минималистичные примитивы**: `Agent` (инструкции + функции) и handoffs (возврат другого Агента из функции).
- **Основной цикл** (~300 строк в `swarm/core.py`):

  1. Получить завершение от текущего Агента
  1. Выполнить вызовы инструментов, добавить результаты
  1. Переключить Агента, если функция возвращает Агента
  1. Обновить переменные контекста
  1. Повторять до отсутствия вызовов инструментов или достижения `max_turns`

- **Агенты**: Просто имя, модель, инструкции (строка или вызываемый объект), список функций. Нет иерархии агентов — плоское делегирование через handoffs.
- **Коммуникация**: Сообщения API Chat Completions. Без состояния между вызовами `client.run()`.

### Предоставление Инструментов

- Инструменты — это обычные функции Python. Авто-схема из аннотаций типов и docstrings.
- Все функции, назначенные Агенту, предоставляются каждому вызову LLM.
- Если функция возвращает `Agent`, выполнение передаётся (handoff).
- Параметр `context_variables` автоматически заполняется, если определён в сигнатуре функции.

### Модель Безопасности

- **Отсутствует**. Нет изоляции, нет изоляции, нет модели разрешений. Инструменты работают в процессе вызывающего.
- Образовательный/экспериментальный проект, явно не для производства.

### Память/Контекст

- **Без состояния**: Нет состояния между вызовами `client.run()`. Пользователь должен передавать `messages` и получать их обратно.
- **Переменные контекста**: Простой словарь, передаваемый через вызовы функций — может читаться/записываться функциями инструментов.
- Нет памяти, нет персистентности сессий, нет сжатия контекста.

### Уникальные Особенности

- **Экстремальная простота**: Весь фреймворк — ~4 исходных файла. Отлично для изучения оркестровки агентов.
- **Шаблон Handoff**: Элегантный — Агент это просто "инструкции + инструменты", и агенты делегируют, возвращая другого Агента из функции инструмента.
- **Поддержка потоковой передачи**: Встроенная потоковая передача с маркерами `{"delim":"start"}` / `{"delim":"end"}` для границ агентов.

### Потенциальные Пробелы Относительно Проектных Целей Entelecheia

- Устаревший (заменён на OpenAI Agents SDK).
- Полное отсутствие модели безопасности.
- Нет состояния/персистентности — полностью без состояния.
- Только OpenAI (Chat Completions API).
- Нет мульти-агентной коммуникации кроме handoffs.
- Экспериментальный/образовательный — не промышленный фреймворк.

## 7. Cline

**Репозиторий**: [cline/cline](https://github.com/cline/cline)
**Язык**: TypeScript (расширение VS Code) + Go (CLI)
**Лицензия**: Apache 2.0

### Архитектура

- **Расширение VS Code** + автономный CLI. Расширение является основным интерфейсом; CLI новее.
- **Основной цикл** (в `src/core/`):

  1. Разобрать задачу пользователя (текст + изображения)
  1. Проанализировать рабочее пространство (AST, поиск regex, чтение файлов)
  1. Выполнять инструменты в цикле: создание/редактирование файлов, команды терминала, действия браузера
  1. Отслеживать выводы (ошибки линтера, вывод терминала, скриншоты браузера)
  1. Автоисправление проблем, итерация до завершения задачи

- **Набор инструментов**: Операции с файлами (create, edit, diff), команды терминала (с интеграцией оболочки), браузер (headless, click/type/scroll), инструменты MCP.
- **Системный промпт**: Детальная инженерия промптов с инструкциями управления контекстом.

### Предоставление Инструментов

- Фиксированный набор встроенных инструментов: `read_file`, `write_to_file`, `replace_in_file`, `execute_command`, `browser_action`, `use_mcp_tool` и др.
- **Расширение MCP**: Может создавать/устанавливать новые серверы MCP по требованию ("добавь инструмент, который..."). Поддерживаются также серверы MCP сообщества.
- **@-упоминания**: `@url`, `@problems`, `@file`, `@folder` для внедрения контекста (снижение затрат API).
- Ограниченное количество инструментов на вызов — обычно 8-12 встроенных + инструменты MCP.

### Модель Безопасности

- **Человек-в-цикле для всех действий**: Каждое изменение файла и команда терминала должны быть одобрены пользователем в GUI.
- **Система контрольных точек**: Снимки рабочего пространства перед каждым шагом. Можно просмотреть/восстановить различия в любой момент.
- **Система разрешений**: `CommandPermissionController` со списками разрешений/запретов.
- **Cline Ignore**: Файл `.clineignore` для исключения чувствительных файлов.
- **Нет изоляции**: Команды терминала выполняются в реальной среде пользователя — мощь и риск.
- **Enterprise**: SSO, аудиторские следы, VPC/частная ссылка, self-hosted/on-prem.

### Память/Контекст

- **Управление контекстом**: Анализ AST проекта, поиск regex релевантных файлов, тщательный отбор того, что попадает в контекстное окно.
- **Сжатие контекста**: Когда контекст заполняется, генерируются сводки, и старый разговор сжимается.
- **Контрольные точки**: Полные снимки рабочего пространства для отката.
- **Состояние задачи**: `TaskState.ts` управляет памятью разговора и состоянием рабочего пространства.
- **Обнаружение циклов**: `loop-detection.ts` предотвращает бесконечные циклы вызовов инструментов.

### Уникальные Особенности

- **Полная интеграция IDE**: Живёт внутри VS Code, видит всё рабочее пространство.
- **Автоматизация браузера**: Возможность Claude Computer Use для веб-тестирования/отладки.
- **Цикл автоисправления**: Отслеживает ошибки линтера/компилятора и автоисправляет без вмешательства пользователя.
- **Создание инструментов MCP**: Может создавать новые серверы MCP на лету на основе запросов пользователя.
- **Proceed While Running**: Неблокирующее выполнение команд терминала.
- **Мультимодельность**: OpenRouter, Anthropic, OpenAI, Gemini, Bedrock, Azure, Vertex, Cerebras, Groq, Ollama, LM Studio.
- **Go CLI**: Автономная опция CLI для сред без VS Code.
- **Фреймворк оценки**: 3-слойное тестирование (contract, smoke, E2E/bench).

### Потенциальные Пробелы Относительно Проектных Целей Entelecheia

- Привязан к VS Code для полного опыта (CLI новее и менее зрелый).
- Однопользовательская архитектура — нет мульти-агентного сотрудничества.

## 8. Aider

**Репозиторий**: [Aider-AI/aider](https://github.com/Aider-AI/aider)
**Язык**: Python
**Лицензия**: Apache 2.0

### Архитектура

- **Парное программирование на основе терминала**: Инструмент CLI, который редактирует код в вашем репозитории.
- **Основной цикл** (`base_coder.py`):

  1. Построить карту репозитория (сводка кодовой базы на основе Tree-sitter AST)
  1. Отправить промпт + карту репозитория + файлы в LLM
  1. Разобрать ответ LLM на инструкции редактирования (унифицированные диффы, блоки поиска/замены, перезаписи целых файлов)
  1. Применить правки, линтить, запустить тесты, автоисправить сбои
  1. Git коммит с осмысленными сообщениями

- **Множество форматов редактирования**: `udiff`, `editblock`, `wholefile`, `search_replace`, `diff_fenced`, `editor_editblock`, `editor_whole`, `patch`, `architect`. Каждый — отдельный класс кодера со своим шаблоном промпта.
- **Режим Architect/Editor**: Двух-агентный шаблон — архитектор планирует, редактор реализует. Доказал высокую эффективность в бенчмарках.

### Предоставление Инструментов

- **Нет традиционного вызова инструментов**. Aider использует структурированные текстовые ответы (не вызов функций).
- LLM выводит инструкции редактирования в определённых форматах (блоки diff, поиск/замена), которые Aider разбирает и применяет.
- Команды оболочки: LLM может запросить выполнение команд оболочки (с подтверждением пользователя).
- Веб-скрапинг: Браузер на основе Playwright для чтения веб-страниц.
- Голосовой ввод: Речь-в-код через микрофон.
- Изображения: Можно добавлять скриншоты/изображения в чат для визуального контекста.

### Модель Безопасности

- **Нет изоляции**. Правки применяются непосредственно к вашим файлам. Git обеспечивает страховочную сеть.
- **Подтверждение пользователя**: Команды оболочки требуют явного одобрения пользователя.
- **Интеграция с Git**: Все изменения автоматически коммитятся — лёгкий откат.
- **Режим наблюдения**: Может отслеживать изменения файлов и автоматически переприменять при сбое тестов.
- **Линтинг/тестирование**: Автоматически запускает настроенные линтеры и тестовые наборы после изменений.
- Нет системы разрешений, нет изоляции, нет доступа на основе ролей.

### Память/Контекст

- **Карта репозитория**: Анализ Tree-sitter AST строит краткую карту всей кодовой базы (сигнатуры функций, определения классов, отношения импорта). Это помещает структуру кодовой базы в контекст без включения всего исходного кода.
- **История чата**: Полный разговор в контексте.
- **Выбор файлов**: LLM запрашивает конкретные файлы через синтаксис — только содержимое этих файлов добавляется в контекст.
- **Управление контекстным окном**: При приближении к лимиту контекста Aider отбрасывает старые ходы разговора и включает сводки.
- **Нет долгосрочной памяти**: Без состояния между сессиями. Каждый запуск `aider` начинается заново.

### Уникальные Особенности

- **Карта репозитория**: Понимание кодовой базы на основе AST, превосходящее RAG на основе эмбеддингов для задач кода.
- **Множество форматов редактирования**: Адаптируется к тому, что лучше работает для каждой модели (некоторые модели лучше с udiff, другие с поиском/заменой и т.д.).
- **Режим Architect/Editor**: Двухшаговый процесс с отдельными вызовами LLM для планирования и выполнения.
- **Полиглот**: 100+ языков программирования через Tree-sitter.
- **Таблицы лидеров LLM**: Поддерживает публичные бенчмарки для редактирования кода по моделям.
- **Голосовое кодирование**: Речь-в-код непосредственно в терминале.
- **Режим копирования-вставки**: Работает с веб-интерфейсами чата для моделей без доступа API.
- **Осведомлённость о Git**: Авто-коммиты, осмысленные сообщения коммитов, работает с существующими репозиториями git.
- **Самонаписание**: 88% собственного кода Aider было написано Aider.

### Потенциальные Пробелы Относительно Проектных Целей Entelecheia

- **Фокус на одном файле**: В первую очередь редактирует один файл за раз (хотя карта репозитория предоставляет контекст).
- **Нет мульти-агентной системы**: Только пара архитектор/редактор. Нет настраиваемых ролей агентов.
- **Нет экосистемы инструментов**: Не может использовать API, базы данных, веб-сервисы — только редактирование файлов и оболочку.
- **Нет изоляции**: Прямой доступ к файловой системе — мощно, но рискованно.
- **Нет постоянного состояния**: Каждая сессия независима. Нет обучения между сессиями.
- **Нет примитивов оркестровки**: Не фреймворк — автономный инструмент.
- **Хрупкость разбора текста**: Полагается на вывод LLM точных форматов редактирования — может сбоить при отклонениях модели.
- **Только терминал**: Не встраивается как библиотека в большинстве рабочих процессов (хотя API Python существует).

## Ключевые Выводы для Дорожной Карты Entelecheia

1. **Безопасность/Изоляция**: Почти все фреймворки не имеют изолированного выполнения. Google ADK является заметным исключением с изоляцией на основе контейнеров/Kubernetes. Это основная возможность для дифференциации.

1. **Мульти-агентная коммуникация**: Только Google ADK имеет формальный протокол взаимодействия между агентами (A2A). Большинство фреймворков используют ad-hoc передачу сообщений. Стандартизированный протокол (как A2A) является пробелом.

1. **Архитектуры памяти**: Большинство фреймворков имеют базовую краткосрочную память. Немногие имеют сложную иерархическую память (рабочую, краткосрочную, долгосрочную) с автоматическим управлением контекстом. Фильтрация MetaGPT и сжатие ADK — лучшие примеры.

1. **Управление предоставлением инструментов**: Все фреймворки предоставляют все инструменты LLM на вызов. Ни один фреймворк не подмножествует инструменты динамически на основе контекста/состояния/уровня безопасности. Это архитектурный пробел.

1. **Выполнение кода**: Только ADK имеет промышленное изолированное выполнение кода. ChatDev имеет базовый исполнитель кода. Cline/Aider полагаются на нативную среду. Это критический пробел в безопасности во всей экосистеме.

1. **Оценка**: ADK и Cline имеют формальные фреймворки оценки. Другие полагаются на ad-hoc тестирование или исследовательские бенчмарки. Встроенная оценка является дифференциатором.

1. **Устаревание OpenAI Swarm** в пользу OpenAI Agents SDK сигнализирует о рыночном тренде к промышленным фреймворкам вместо образовательных экспериментов.

1. **Долговременное выполнение LangGraph** уникально мощно для долго работающих агентов — большинство фреймворков предполагают краткосрочные задачи.

1. **Подход ChatDev 2.0 без кода** нацелен на принципиально другую персону пользователя (не-разработчиков), чем большинство фреймворков. Это ортогонально дизайну Entelecheia, ориентированному на разработчиков.

1. **Cline и Aider** — это приложения, а не фреймворки. Они демонстрируют мощь тесной интеграции инструментов (IDE, git, терминал, браузер), но не могут быть скомпонованы в более крупные агентные системы.

## Приложение: Напоминание о Текущем Состоянии Entelecheia

Это приложение является авторитетным корректирующим слоем для сравнений выше.

### Что активно сегодня

- 12 агентов Уровня 1 скомпилированы в рабочем пространстве.
- 1 крейт Уровня 2 для Веб-автоматизации активен.
- Дополнительные специализированные агенты архивированы как планы или частичные документы и не должны рассматриваться как полностью реализованные модули времени выполнения.

### Что относительно зрелое

- `packages/scepter`, `packages/shared` и `packages/tui`
- модель предоставления инструментов exec-only
- пути выполнения с поддержкой контейнеров
- зашифрованное хранение ключей провайдеров и связанная с RBAC сантехника аутентификации

### Что всё ещё частично

- WebUI по сравнению с TUI
- Покрытие команд CLI
- интеграции с десктопом/мобильными устройствами (мигрированы в [shittim-chest](https://github.com/celestia-island/shittim-chest))
- RAG и память, которые в настоящее время полагаются на документы в памяти, эмбеддинги на основе хэшей и обход графов вместо полностью интегрированного стека ONNX + pgvector
- полнота аудита и укрепление контейнеров
