# ADR-002: 組込みJavaScriptエンジンとしてのBoa

**日付**: 2026-02
**ステータス**: 承認済み

## 背景

IEPL（Interactive Execution Pipeline Layer）は、各COSMOSコンテナ内でLLM生成コードを実行するためのJavaScriptランタイムを必要とします。このランタイムは以下を満たす必要があります：

1. Rustアプリケーションに**埋め込み可能**であること — 軽量コンテナ内でinitプロセス（PID 1）として実行されます。
1. **安全でサンドボックス化**されていること — LLM生成コードは本質的に信頼できません。
1. ツールディスパッチメカニズムのための**ESモジュールインポート**をサポートすること（例：`import { file_read } from 'agent'`）。
1. **最小限の起動オーバーヘッド** — コンテナは一時的であり、迅速に準備が整う必要があります。
1. **クロスプラットフォーム**で、コンテナターゲット向けに簡単にコンパイルできること。

いくつかのJavaScript/TypeScriptランタイムが評価されました：

| ランタイム | 言語 | Rust埋め込み | サンドボックス制御 | 起動速度 | ESモジュールサポート |
| --- | --- | --- | --- | --- | --- |
| **Boa** | Pure Rust | ネイティブ（Rustクレート） | 完全（ホストがすべてを制御） | 高速（<10ms） | 部分的（IEPLには十分） |
| **Deno** | Rust + V8 | FFI経由で可能 | 限定的（V8アイソレート） | 低速（~100ms） | 完全 |
| **QuickJS** | C | FFI/バインディング経由 | 中程度 | 高速 | 部分的 |
| **V8** | C++ | FFI経由（v8クレート） | 限定的 | 低速 | 完全 |
| **wasmoon** | C (Lua) | FFI経由 | 良好 | 高速 | N/A (Lua) |
| **rquickjs** | C (QuickJS) | FFI経由 | 中程度 | 高速 | 部分的 |

## 決定

組込みJavaScriptランタイムとして**Boa Engine**（v0.21）を選択しました。

**主な理由：**

1. **Pure Rust — ゼロFFIオーバーヘッド。** Boaは完全にRustで書かれており、C/C++依存チェーンなしでCOSMOSバイナリにネイティブコンパイルされます。これにより、FFI関連のセキュリティ脆弱性、ビルドの複雑さ、クロスコンパイルの頭痛の種が一掃されます。initプロセスを制御するCOSMOSコンテナのコンテキストでは、これは重要な利点です。

1. **「呼び出して使用する」シンプルさ。** Boaはライブラリファーストのエンジンとして設計されています。数行のRustコードでインスタンス化、ホスト関数の設定、実行が可能です。管理する別プロセス、維持するIPCブリッジ、イベントループの複雑さはありません。COSMOSの`JsReplHandle`はBoaランタイム用の専用OSスレッドを作成し、標準のRustチャネルを介して通信します — クリーンで構成可能なアーキテクチャです。

1. **セキュリティが最優先であり、パフォーマンスではない。** COSMOSサンドボックスでは、JS実行時間の各ミリ秒はクリティカルパス上にありません — ボトルネックは常にLLM推論のラウンドトリップです。重要なのは、ランタイムが実行コードに対して**完全な制御**を提供することです。Boaのホスト関数登録APIにより、利用可能な関数（MCPツールディスパッチ関数のみ）を正確に定義でき、脱出ハッチはありません。ASTセキュリティバリデータ（`eval`、`require`、`process`などをブロック）はBoa AST上で動作し、ネイティブRustの保証による強制を提供します。

1. **IEPLに十分なESモジュールサポート。** IEPLパイプラインはシミュレートされたモジュールシステムを使用します — ESモジュールインポートは名前空間ビルダーレベルで解決され、実際のモジュールローダーではありません。Boaの機能はこのパターンに十分以上です。完全なNode.js互換のモジュール解決アルゴリズムは必要ありません。

## 結果

### ポジティブ

- **C/C++ビルド依存なし** — COSMOSは`cargo build`のみでクリーンにコンパイルされ、システムレベルのライブラリ要件はありません。
- **完全なサンドボックス制御** — 実行されるJSコードが利用可能なすべての関数は、ホストによって明示的に登録されます。デフォルトのI/O、ネットワーク、ファイルシステムアクセスはありません。
- **Rust型との緊密な統合** — カスタムホストオブジェクトのための`boa_gc::Trace`トレイト実装、ネイティブ`serde_json`相互運用、可能な場合はゼロコピー。
- **クラッシュ安全性** — BoaパニックはOSスレッド境界で捕捉され、不正なJSコードによるCOSMOSプロセスのクラッシュを防ぎます。
- **小さなバイナリフットプリント** — V8ベースのソリューションと比較して、BoaはCOSMOSバイナリサイズを大幅に小さく抑えます。

### ネガティブ

- **JavaScript準拠が不完全** — BoaはECMAScript 2024+を完全に実装していません。一部の高度な機能（例：`WeakRef`、`FinalizationRegistry`、完全な`Promise`統合、すべてのコンテキストでの`async/await`）には制限や未実装がある可能性があります。
- **パフォーマンスはV8/SpiderMonkeyに匹敵しない** — BoaのインタプリタはJITコンパイルエンジンよりも大幅に低速です。CPU集約的なワークロード（大規模データ処理、複雑なアルゴリズム）では、これは問題になります。しかし、IEPLのコンテキストでは、JSコードは主にMCPツールを呼び出すオーケストレーショングルーであり、計算ではありません。
- **エコシステムが小さい** — BoaはV8やQuickJSよりも貢献者が少なく、実戦テストも少ないです。バグの上流修正に時間がかかる可能性があります。
- **`eval`や動的コード生成なし** — 設計上（そしてASTバリデータによって強制）、動的コード評価はブロックされます。これにより特定のメタプログラミングパターンが制限されますが、セキュリティモデルには許容可能です。

### 受け入れたトレードオフ

**セキュリティと埋め込み可能性のためのパフォーマンス犠牲。** IEPL実行エンジンが複雑なアルゴリズムの実行や大規模データセットの処理を必要とする場合、Boaは誤った選択でしょう。しかしCOSMOSサンドボックスでは、JSコードは薄いオーケストレーションレイヤーです — その役割はMCPツールを正しい順序で呼び出し、エラーを処理し、結果を構成することです。実際の重労働はRustベースのエージェントツールによって行われます。BoaのV8と比較して10〜100倍遅い実行速度は、すべてのツール呼び出しがScepterへのネットワークラウンドトリップ（50〜500ms）を伴う場合、無関係です。
