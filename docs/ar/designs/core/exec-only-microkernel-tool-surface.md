# ADR-001: سطح أدوات النواة الدقيقة للتنفيذ فقط

**التاريخ**: 2026-02
**الحالة**: مقبول (معدّل 2026-06 — السطح مُقلّص من 5 إلى 3 بدائيات)

## السياق

في نظام تنسيق LLM متعدد الوكلاء، يجب أن يقرر النموذج أي الأدوات يستدعيها وكيف يركبها. النهج الساذج هو تعريض كل أداة MCP (118+ عبر 12 وكيلًا) مباشرة إلى LLM كتعريفات دوال منفصلة في التوجيه.

هذا يخلق عدة مشاكل:

1. **استهلاك نافذة السياق**: 118+ تعريف أداة يستهلك آلاف الـ tokens، تاركًا مساحة أقل للاستدعاء والمحادثة.
1. **مساحة سطح الأمان**: كل أداة معروضة على LLM هي متجه هجوم محتمل لحقن التوجيهات أو كسر الحماية.
1. **تجزئة فرض الصلاحيات**: إذا تم توزيع الأدوات مباشرة بمخرجات LLM، يجب على كل أداة التحقق من الصلاحيات بشكل مستقل — مما يقود إلى فرض غير متسق وفجوات.
1. **ارتباك النموذج**: تُظهر الأبحاث أن أداء LLM يتدهور عند عرضه لخيارات أدوات كثيرة جدًا (مشكلة "زيادة تحميل الأدوات").

## القرار

نعتمد تصميم **نواة دقيقة تنفيذية فقط**. يرى LLM **3 بدائيات تنفيذ** تمامًا كسطح أدواته:

| البدائية | الغرض |
| --- | --- |
| `exec` | تنفيذ كود TypeScript/JavaScript عبر خط أنابيب IEPL |
| `write_to_var` | كتابة قيمة نصية إلى متغير REPL مسمى |
| `write_to_var_json` | كتابة قيمة JSON إلى متغير REPL مسمى |

> **تعديل (2026-06)**: التصميم الأصلي عرض 5 بدائيات، بما في ذلك `ref_add` و`ref_remove` لإدارة متغيرات المرجع المسماة. تم **إزالة** هذين من السطح المرئي لـ LLM لأن آلية متغير المرجع غير مستخدمة من قبل أي إجراء مهارة SOP وأضافت عبء سياق بلا قيمة. الدالة `agent_allowed_tools()` في `packages/shared/domain_skills/src/tool_names.rs` تُرجع الآن فقط البدائيات الثلاث أعلاه. راجع سجل الالتزامات (`60d58f794`، `31cf9a00e`).

يتم استدعاء كل أدوات MCP الـ 118+ للوكلاء **بشكل غير مباشر** عبر استيراد وحدات ES داخل كود `exec`. يُولّد LLM كود TypeScript يستورد ويستدعي أدوات الوكلاء؛ يتم تحويل هذا الكود بواسطة SWC، والتحقق منه بواسطة مدقق أمان AST، وتنفيذه بواسطة محرك Boa JS داخل صندوق رمل COSMOS.

## النتائج

### إيجابية

- **عبء سياق أدنى**: 3 تعريفات أدوات مقابل 118+. يمكن لـ LLM التركيز على الاستدعاء بدلًا من ميكانيكا اختيار الأدوات.
- **أمان مركزي**: تمر كل استدعاءات الأدوات عبر `McpRouter` الذي يفرض القوائم البيضاء، التخويل المزدوج، مستويات الثقة، وتقييم المخاطر الديناميكي في نقطة اختناق واحدة.
- **تدفقات عمل قابلة للتركيب**: يمكن لـ LLM كتابة تركيبات أدوات معقدة بشكل عشوائي بـ TypeScript (حلقات، شرطية، معالجة أخطاء) بدلًا من أن يكون محدودًا لاستدعاءات أدوات فردية.
- **تنفيذ قابل للتدقيق**: يمر كل استدعاء `exec` عبر تحقق AST يرفض البنى الخطرة (`eval`، `require`، `process`، `Function`، `import()`، وصول `globalThis`).
- **ترحيل نماذج أسهل**: مزودو LLM الجدد يحتاجون فقط لدعم استدعاء الدوال بـ 3 أدوات، وليس 118.

### سلبية

- **عبء غير مباشر**: تمر استدعاءات الأدوات عبر توليد TypeScript ← تحويل SWC ← تحقق AST ← تنفيذ Boa ← توزيع MCP router، مضيفًا زمن انتقال لكل استدعاء.
- **اعتماد على جودة كود LLM**: فعالية النظام تعتمد على قدرة LLM على توليد كود TypeScript صحيح يستورد ويستدعي أدوات الوكلاء بشكل صحيح.
- **تعقيد التصحيح**: عندما تفشل استدعاء أداة، تمتد سلسلة الخطأ عبر توليد TypeScript، التحويل، التحقق، تنفيذ JS، وتوزيع MCP — مما يجعل التصحيح أصعب من استدعاء الأدوات المباشر.
- **ليست كل نماذج LLM متساوية**: قد تناضل النماذج الأقل قدرة مع توليد الكود لتدفقات عمل متعددة الأدوات معقدة مقارنة بالاستدعاء البسيط للدوال.

### مخاطر مخفّفة

- هجمات حقن التوجيهات التي تحاول استدعاء أدوات خطرة مباشرة
- سوء استخدام الأدوات العرضي بسبب ارتباك LLM بين خيارات كثيرة جدًا
- فرض صلاحيات غير متسق عبر الأدوات
