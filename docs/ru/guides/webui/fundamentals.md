# Основные концепции

> **Аудитория**: Разработчики, которым нужно концептуальное понимание дизайна shittim-chest.
> **Последнее обновление**: 2026-05-25

## Двухрепозиторная архитектура

shittim-chest и [entelecheia](https://github.com/celestia-island/entelecheia) образуют двухрепозиторную систему с намеренной границей:

- **entelecheia** — ядро оркестрации агентов (scepter, 13 агентов, среда выполнения Cosmos/IEPL). Владеет идентификацией, правами, конфигурациями агентов.
- **shittim-chest** — пользовательская оболочка. Владеет аутентификацией, сессиями, данными чата, конфигурацией провайдеров LLM, UI фронтенда и прокси-мостом к scepter.

Они общаются через JWT-аутентифицированный HTTP и WebSocket. Ни один не обращается к базе данных другого напрямую. Это разделение позволяет каждому репозиторию разрабатываться, развёртываться и масштабироваться независимо.

## Двойные режимы работы

shittim-chest поддерживает два режима работы:

### Автономный режим

Работает независимо с собственным слоем маршрутизации LLM. Поддерживает:

- Чат с потоковыми ответами (SSE + WebSocket)
- Генерацию изображений через настроенных провайдеров
- Аутентификацию пользователей (пароль + GitHub OAuth)
- Управление провайдерами (добавление/удаление провайдеров LLM)

Не требует entelecheia. Полезен для разработки и простых развёртываний.

### Режим прокси

Выступает как шлюз к системе агентов entelecheia. Добавляет:

- Пересылку запросов в scepter с передачей JWT
- Мост WebSocket для чата на основе агентов
- Вход вебхуков и пересылку триггеров
- Управление удалёнными устройствами через polemos
- Запросы прав RBAC и кеширование

Требует работающий экземпляр entelecheia. Два режима могут сосуществовать — автономный LLM для простого чата, прокси для оркестрации агентов.

## Модель аутентификации

Аутентификация использует токены JWT, выпускаемые `shittim_chest`:

1. **Хранение учётных данных**: Пароли (хеши argon2), сессии, токены обновления и API-ключи находятся в `shittim_chest_db`.
1. **GitHub OAuth**: Пользователи могут входить через GitHub; учётные записи создаются автоматически при первом входе.
1. **Хранение прав**: Группы пользователей, роли и матрицы прав находятся в `entelecheia_db`.
1. **Поток JWT**: При входе `shittim_chest` проверяет учётные данные локально, затем запрашивает права у scepter. Выпущенный JWT содержит `{ sub: user_id, groups: [...] }`.
1. **Общий секрет**: Секрет подписи JWT общий с scepter, так что оба сервиса могут независимо проверять токены.
1. **Ротация токенов**: Токены доступа истекают через 1 час; токены обновления — через 7 дней. Токены обновления ротируются при каждом использовании.

## Фронтенд (webui)

Webui — это унифицированный фронтенд в `packages/webui/`, с интерфейсом чата на `/` и панелью администратора на `/backend`, построенный на Vue 3 + Vite + Pinia (TSX через `@vitejs/plugin-vue-jsx`).

## Система провайдеров LLM

shittim-chest имеет независимый слой маршрутизации LLM:

- **Провайдеры**: Настраиваемые конечные точки API LLM (OpenAI-совместимые). Хранятся в `shittim_chest_db` с зашифрованными AES-256-GCM API-ключами.
- **Маршрутизатор**: Мульти-провайдерная маршрутизация с выбором на основе приоритета и автоматической отказоустойчивостью.
- **Категории**: Провайдеры могут быть помечены как `chat`, `image` или оба.
- **Управление**: Полный CRUD через REST API и панель администратора webui. Провайдеры можно тестировать на связность.
- **Потоковая передача**: Протоколы потоковой передачи SSE (простой, удобный для прокси) и WebSocket (двунаправленный).

## Система чата

- **Диалоги**: Сеансы чата на основе потоков с заголовками и метаданными
- **Сообщения**: Поддерживают текст, изображения и вызовы инструментов (function calling)
- **Потоковая передача**: Доставка ответов в реальном времени токен за токеном через SSE или WebSocket
- **Поиск**: Полнотекстовый поиск сообщений с запросами ILIKE
- **Экспорт**: Диалоги можно экспортировать в формате JSON или Markdown
- **Генерация изображений**: Генерация изображений на основе промптов через настроенных провайдеров с функцией «Вставить в чат»

## Управление удалёнными устройствами

shittim-chest предоставляет браузерный интерфейс для удалённых устройств, управляемых entelecheia/polemos:

- **Рабочий стол**: Просмотр удалённого рабочего стола на основе WebRTC с ретрансляцией кадров
- **Терминал**: Эмулятор терминала на основе xterm.js с ретрансляцией WebSocket
- **Файловый браузер**: Бэкенд файлового браузера SFTP (скелет)
- **Сигнализация**: Ретрансляция сигнализации WebRTC на основе WebSocket (SDP offer/answer, кандидаты ICE)

Вся связь с устройствами проходит через агент polemos в entelecheia — shittim-chest никогда не подключается к конечным точкам напрямую.

## Архитектура прокси

`shittim_chest` выступает как шлюз между пользователями и scepter:

- **Обратный HTTP-прокси**: `/api/proxy/*` пересылает аутентифицированные запросы в scepter с передачей JWT.
- **Мост WebSocket**: Потоковая передача чата использует двунаправленную пересылку WebSocket (`браузер ↔ shittim_chest ↔ scepter`).

Это позволяет `shittim_chest` применять ограничения скорости, логировать использование и управлять жизненным циклом соединения без необходимости scepter обрабатывать отдельные соединения браузера.

## Конвейер вебхуков

Внешние события достигают ядра агентов через конвейер вебхуков:

```text
GitHub/GitLab/Gitee → POST /api/webhook/{source} → проверка HMAC → Разобрать событие → Переслать в scepter через Unix-сокет → Диспетчеризация агента
```

Каждый провайдер имеет свой механизм проверки:

- **GitHub**: HMAC-SHA256 через `X-Hub-Signature-256`
- **GitLab**: Токен через `X-Gitlab-Token`
- **Gitee**: HMAC с запасным вариантом токена

Дополнительные функции: обнаружение дублирующихся доставок (LRU-кеш), журналирование доставок, белый список IP и общая пользовательская конечная точка вебхука.

## Модель RBAC

Права следуют групповой модели RBAC:

- **Группы**: Пользователи принадлежат одной или нескольким группам.
- **Роли**: Группы имеют назначенные роли.
- **Права**: Каждая роль определяет матрицу прав, охватывающую:
  - Квоты провайдеров (макс. токенов, макс. запросов)
  - Белые списки агентов (к каким агентам группа может получить доступ)
  - Административные возможности (управление пользователями, настройка провайдеров)

`shittim_chest` кеширует права в процессе с TTL (по умолчанию 5 минут). Инвалидация кеша происходит при истечении TTL, выходе из системы или явных изменениях прав, распространённых от scepter.

## Стратегия фронтенда

shittim-chest использует двухфазный подход к фронтенду:

**Фаза 1 (текущая)**: Фронтенд Vue 3 (`webui`, в `packages/webui/`), построенный на Vite + Pinia, с использованием TSX через `@vitejs/plugin-vue-jsx`. Он определяет контракт API и служит эталонной реализацией производственного качества.

**Фаза 2 (будущая)**: Фронтенд Rust → WASM, построенный на Tairitsu. Устаревший фронтенд выступает как живая спецификация и тестовый оракул — одинаковые взаимодействия пользователя должны давать одинаковые результаты.

## Мост типобезопасности

Типы TypeScript генерируются из кода Rust через внешний крейт протокола `arona`, обеспечивая согласованность фронтенда и бэкенда:

```text
Крейт Rust arona (git-зависимость)
  → #[derive(ts_rs::TS)]
  → генерация кода ts-rs → packages/webui/src/types/arona/ (TypeScript)
  → используется webui как @celestia-island/arona
```

Это устраняет ручную синхронизацию типов. Когда тип Rust в крейте `arona` изменяется, привязки TypeScript регенерируются и используются webui.
