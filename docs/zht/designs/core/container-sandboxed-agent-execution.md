# ADR-005：使用 COSMOS 的容器沙箱化代理執行

**日期**：2026-02
**狀態**：已接受

## 背景

在一個代理執行 LLM 生成程式碼的多代理系統中，代理之間的隔離對於以下事項至關重要：

1. **安全性**：不可信的 LLM 輸出不應能夠存取另一個代理的記憶、檔案或網路連線。
1. **狀態隔離**：每個代理的 REPL 狀態（JavaScript 變數、綁定、快照）必須獨立。
1. **資源控制**：行為不當的代理不應消耗無限的 CPU、記憶體或 PID。
1. **可重現性**：代理狀態應可快照和恢復，用於除錯和回滾。
1. **Fork/Merge 工作流程**：系統需要支援分支代理執行（fork）並將結果合併回來（merge），類似於 git 分支。

評估了多種隔離方法：

| 方法 | 隔離強度 | 資源控制 | 快照/Fork | 開銷 |
| --- | --- | --- | --- | --- |
| **每代理容器（Docker/OCI）** | 強（核心層級） | 完整（cgroups、seccomp、capabilities） | 原生（commit/snapshot） | 中等（~100ms 啟動，每容器 ~50MB） |
| **每代理程序** | 中等（UID/seccomp） | 部分（rlimit） | 手動（序列化狀態） | 低 |
| **每代理執行緒** | 弱（共享記憶體） | 最小 | 手動 | 最小 |
| **每代理 WASM 沙箱** | 強（線性記憶體） | 良好（gas 計量） | 手動 | 低 |
| **每代理 Boa 上下文** | 中等（JS 沙箱） | 有限 | 內建（命名空間序列化） | 最小 |

## 決策

我們選擇了**雙層容器架構**，以 **COSMOS** 作為每個代理容器內的初始化程序：

**外層（編排基礎設施）：**

- 透過 Bollard 的 Docker/Podman 用於基礎設施容器（PostgreSQL、Scepter 守護程序）。
- 完整的編排能力：網路、卷、健康檢查、compose。

**內層（代理沙箱）：**

- Youki/libcontainer（預設）或 Docker 用於每個代理的 COSMOS 容器。
- 每個代理獲得自己的容器，以 COSMOS 作為 PID 1。
- COSMOS 是**前端程序**，調解所有互動 — 它提供 JSON-RPC Unix Socket 伺服器、Boa JS REPL、MCP 路由器和回到 Scepter 的 HapLotes 橋接連線。

**為何 COSMOS 作為強制中介：**

所有與容器化代理的互動必須經過 COSMOS。直接容器操作（例如 `docker exec` 進入容器）繞過了安全模型、狀態管理和稽核追蹤。COSMOS 提供：

1. **工具派送調解**：`McpRouter` 在任何工具到達代理之前強制執行允許列表、雙重授權和信任等級。
1. **狀態持久化**：雙緩衝快照系統確保 REPL 狀態在崩潰後存活。
1. **橋接通訊**：HapLotes 橋接將 COSMOS 連接回 Scepter，進行代理間協調。
1. **安全強制**：Seccomp 設定檔、出口策略和能力限制在容器建立時套用，並由核心強制執行。

**為何 Youki/libcontainer 用於內部沙箱：**

- 無根且無守護程序 — 代理沙箱不需要 Docker 守護程序。
- OCI 相容 — 標準的 `config.json` 規範，與 OCI 工具相容。
- 快速的基於疊加層的 rootfs — 快照和 fork 操作僅複製已變更的檔案。
- 比 Docker 更低的開銷，適合短暫容器。

## 後果

### 正面

- **透過核心強制執行的強隔離**：cgroups（CPU/記憶體/PID 限制）、seccomp（系統呼叫過濾）、capabilities（`cap_drop`=ALL）、命名空間（PID/網路/掛載隔離）。
- **原生 Fork/Merge**：容器提交建立映像快照；可以從快照建立新容器。疊加層檔案系統僅追蹤已變更的檔案。
- **每個代理的資源限制**：預設 512MB 記憶體、1 CPU、100 PID，每個容器可配置。
- **稽核追蹤**：所有工具呼叫通過 COSMOS 的 MCP 路由器，為 OreXis 安全稽核記錄每次派送。
- **崩潰遏制**：Boa 恐慌或代理錯誤被限制在其容器內。其他代理和 Scepter 繼續運行。
- **Youki 用於輕量沙箱**：內部容器比完整的 Docker 容器啟動更快，消耗更少資源。

### 負面

- **COSMOS 作為 PID 1 的複雜性**：COSMOS 必須處理訊號轉發、殭屍回收和作為容器初始化程序的乾淨關機。這增加了普通應用程式沒有的責任。
- **容器啟動延遲**：每個代理容器需要約 100ms-1s 來啟動（取決於執行時期）。這比基於程序或基於執行緒的隔離慢。
- **資源開銷**：每個 COSMOS 容器消耗約 50-100MB 的記憶體，用於 Boa 執行時期、JS 堆積和 OS 開銷。對於 9 個容器化代理，這增加了約 0.5-1GB 的基準記憶體。
- **測試複雜性**：測試代理行為需要運行具有 COSMOS 的實際容器，這意味著測試需要 Docker 或 Youki 可用。雪花測試模式（建置 entelecheia 映像、運行 COSMOS 容器、透過 Unix Socket 連線）比單元測試更複雜。
- **需要維護兩個執行時期**：Docker/Bollard 和 Youki/libcontainer 的程式碼路徑都必須維護和測試。

### 已接受的取捨

**為安全性和隔離保證而犧牲資源開銷。** 每代理程序模型使用更少記憶體且啟動更快，但不會提供代理之間的核心層級隔離。在代理執行不可信 LLM 生成程式碼的系統中，容器隔離的安全保證值得資源成本。COSMOS 作為強制中介的設計確保即使攻擊者在容器內獲得程式碼執行權，也無法透過在 COSMOS 調解之外操作來繞過安全模型。
