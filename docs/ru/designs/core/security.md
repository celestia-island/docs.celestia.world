# Архитектура безопасности Entelecheia

> Комплексная модель эшелонированной защиты для платформы мульти-агентной оркестрации Entelecheia.

## Обзор

Entelecheia реализует **архитектуру эшелонированной безопасности**, охватывающую 14 независимо тестируемых слоёв безопасности — от изоляции контейнеров на аппаратном уровне до шлюзов разрешений инструментов, видимых LLM. В отличие от традиционных агентных фреймворков, которые раскрывают все инструменты напрямую LLM, дизайн **Exec-Only микроядра** Entelecheia означает, что LLM видит только 3 примитивных инструмента (`exec`, `write_to_var`, `write_to_var_json`), в то время как 148 инструментов MCP диспетчеризуются через типизированный конвейер IEPL с многоуровневой авторизацией.

## Слой 1: Exec-Only микроядро

**Крейты:** `scepter`, `mcp_types`
**Философия проектирования:** Минимизация поверхности атаки LLM

LLM работает в **песочнице exec-only**, где он может вызывать только три примитивные операции:

| Инструмент | Назначение | Параметры |
| --- | --- | --- |
| `exec` | Выполнить строку скрипта | Код JavaScript (транспилирован из TypeScript через IEPL) |
| `write_to_var` | Сохранить строковое значение | Имя переменной + значение |
| `write_to_var_json` | Сохранить значение JSON | Имя переменной + значение JSON |

Все 148 инструментов MCP (файловые операции, управление контейнерами, управление устройствами, веб-поиск и т.д.) **невидимы для LLM**. Они вызываются косвенно через конвейер IEPL, когда `exec` LLM вызывает импорты ES модулей (например, `import { file_read } from 'kalos'`).

**Модель угроз:** Даже если LLM скомпрометирован через инъекцию в промпт, он не может напрямую вызывать опасные инструменты, такие как `container_destroy` или `ssh_exec`. Конвейер IEPL обеспечивает проверку типов и верификацию разрешений перед выполнением любого инструмента.

**Реализация:** `packages/shared/mcp_types/src/` определяет типы IPC микроядра. Обработчик `exec` в `packages/cosmos/` транспилирует и выполняет скрипт через движок Boa, с вызовами инструментов, маршрутизируемыми через `McpRouter` в `skemma`.

## Слой 2: Шлюз разрешений двойной авторизации

**Крейт:** `security_policy` (5 772 строки)

Каждый инструмент MCP объявляет свои требования к доступу через перечисление **уровня разрешений**. Каждый навык (скрипт IEPL) объявляет уровень разрешений, необходимый ему для каждого инструмента. Оба должны согласоваться для выполнения вызова.

```rust
pub enum PermissionLevel {
    /// Операции только для чтения (file_read, list_dir и т.д.)
    Read,
    /// Операции записи в рабочем пространстве (file_write, exec_script)
    Write,
    /// Операции, затрагивающие внешние системы (ssh_exec, container_deploy)
    System,
    /// Операции с необратимыми последствиями (container_destroy, device_reboot)
    Destructive,
}
```

**Поток авторизации:**

1. Навык объявляет: "Мне нужен доступ `System` к `ssh_exec`"
1. Инструмент объявляет: "Я требую разрешение `System`"
1. Шлюз разрешений проверяет: `skill_level >= tool_requirement` И `навыку явно предоставлен этот инструмент`
1. Если любая проверка не проходит: вызов блокируется, логируется и сообщается sentinel OreXis

**Реализация:** `packages/shared/security_policy/src/` — 107 тестовых аннотаций, 4 tokio теста.

## Слой 3: Авторизация навыков по уровню доверия

**Крейт:** `domain_skills_permissions` (1 776 строк)

Навыки классифицируются по **уровням доверия**, которые определяют их объём разрешений по умолчанию:

| Уровень доверия | Описание | Разрешения по умолчанию |
| --- | --- | --- |
| `Builtin` | Поставляется с платформой | Полный доступ к инструментам |
| `Verified` | Проверен и подписан мейнтейнерами | Read + Write |
| `Community` | Предоставлен пользователями | Только чтение |
| `Untrusted` | Динамически загружен | Нет доступа к инструментам (только exec) |

Уровень доверия каждого навыка проверяется при загрузке и кэшируется. Попытки повысить уровень доверия логируются как события безопасности.

## Слой 4: Контейнерная изоляция (Внешнее кольцо)

**Крейт:** `container` (5 742 строки)

Каждое выполнение агента происходит внутри **контейнера Docker или Podman** с:

- Изоляцией сетевого пространства имён
- Корневой файловой системой только для чтения (кроме монтирования рабочего пространства)
- Профилем Seccomp, ограничивающим системные вызовы
- Ограничениями ресурсов (CPU, память, количество PID)
- Отсутствием доступа к сокету Docker хоста

**Реализация:** `packages/shared/container/src/` — 74 тестовых аннотации, 12 tokio тестов. Поддерживает как Docker (через Bollard API), так и Podman.

-----------------------------------------------------------------------------

## Слой 5: Песочница OCI (Внутреннее кольцо)

**Крейт:** `container_runtime` (3 645 строк)

Внутри контейнера Docker Entelecheia запускает **второй слой изоляции** с использованием Youki/libcontainer — бездемонной, rootless OCI-совместимой контейнерной среды выполнения. Это обеспечивает:

- Rootless выполнение (повышение привилегий невозможно)
- Изоляцию пространств имён, независимую от Docker
- Принудительное применение Cgroup v2
- Фильтр Seccomp (запрет по умолчанию)

**Зачем два слоя?** Docker обеспечивает крупнозернистую изоляцию (сеть, файловая система). Youki обеспечивает тонкую фильтрацию системных вызовов и учёт ресурсов. Если Docker скомпрометирован, песочница Youki всё ещё сдерживает агента.

## Слой 6: Контроль доступа RBAC

**Крейты:** `domain_auth` (380 строк), shittim-chest `rbac` (1 736 строк)

Ролевой контроль доступа, управляющий всеми операциями API:

- **Группы:** Пользователи принадлежат группам; группы имеют разрешения
- **Разрешения:** Детализированные разрешения (чтение/запись/администрирование для каждого типа ресурса)
- **Изоляция рабочих пространств:** Пользователи могут получать доступ только к рабочим пространствам, членами которых они являются
- **Операции между рабочими пространствами:** Требуют явных разрешений администратора

## Слой 7: JWT аутентификация

**Модуль:** shittim-chest `auth/jwt.rs` (264 строки)

- **Алгоритм:** HS256 (HMAC-SHA256)
- **Токены доступа:** Короткоживущие (настраиваемые, по умолчанию 15 мин)
- **Токены обновления:** Более долгоживущие с ротацией при использовании
- **Защита CSRF на основе nonce** для браузерных клиентов
- **Ограничение скорости** на конечных точках аутентификации (алгоритм GCRA)

-----------------------------------------------------------------------------

## Слой 8: Шифрование ключей API

**Крейт:** `aporia` (5 802 строки)

Все ключи API провайдеров LLM шифруются в состоянии покоя с использованием **AES-256-GCM** с:

- Уникальным nonce для каждой операции шифрования
- Ключом, производным от мастер-секрета (настраивается через окружение)
- Обнулением ключей в открытом виде из памяти после использования
- Поддержкой ротации ключей

-----------------------------------------------------------------------------

## Слой 9: Sentinel безопасности (OreXis)

**Крейт:** `orexis` (5 239 строк) — агент "иммунная система"

OreXis — это агент Layer-1, который:

- **Аудирует код** на наличие уязвимостей безопасности и соответствия лицензиям
- **Инспектирует вызовы инструментов** на соответствие зарегистрированным политикам безопасности
- **Блокирует/разблокирует** инструменты любого агента по шаблону
- **Мониторит** поведение агентов на предмет аномальных паттернов

Инструменты MCP (24): `standard_check`, `compliance_report`, `audit_alignment`, `audit_legality`, `agent_integrity`, `security_audit`, `tool_block`, `tool_unblock`, `policy_register`, `policy_list` и др.

-----------------------------------------------------------------------------

## Слой 10: Типобезопасный конвейер IEPL

**Крейты:** `iepl` (2 670 строк), `iepl_engine` (1 228 строк), `skemma` (7 960 строк)

Конвейер **Entelecheia Plugin Language** (IEPL) обеспечивает типобезопасность между кодом, сгенерированным LLM, и нативной диспетчеризацией инструментов:

1. LLM генерирует код TypeScript с использованием импортов ES модулей
1. **SWC** транспилирует TypeScript -> JavaScript (валидация синтаксиса)
1. **Движок Boa** выполняет JavaScript в изолированном контексте
1. Импорты ES модулей разрешаются в вызовы `__native_dispatch`
1. Каждая диспетчеризация маршрутизируется через `McpRouter` с полной проверкой типов

**Устраняемая угроза:** Инъекционные атаки через нетипизированные вызовы инструментов (распространены в агентных фреймворках на Python, где схемы инструментов проверяются только во время выполнения).

## Слой 11: Белый список реестров провайдеров

**Файл:** `configs/registries.toml` (337 строк)

Entelecheia поддерживает **хардкодный белый список** доверенных реестров пакетов в 15 экосистемах:

crates.io, PyPI, npm, Go modules, Docker Hub, Maven Central, NuGet, RubyGems, Hackage, Alpine APK, Debian APT, GitHub, GitLab, `HuggingFace`, PyTorch.

Любой импорт пакета из не входящего в белый список реестра **блокируется на уровне контейнера** перед выполнением.

## Слой 12: Защита от инъекций в промпты

**Механизм:** Граница песочницы IEPL

Вывод `exec` LLM выполняется в **изолированном контексте Boa JS** без доступа к:

- Файловой системе хоста
- Сетевым сокетам
- Переменным окружения
- Состоянию других агентов

Выводы инструментов, возвращаемые LLM, **санируются** — бинарные данные кодируются в base64, чрезмерный вывод обрезается, а потенциальные паттерны инъекций в промпты в результатах инструментов помечаются OreXis.

## Слой 13: Ограничение скорости

**Модуль:** shittim-chest `channel/rate_limit.rs` (118 строк)

Ограничение скорости для каждого пользователя и канала с использованием алгоритма **GCRA (Generic Cell Rate Algorithm)**:

- Настраиваемый размер всплеска и устойчивая скорость
- DashMap для каждого пользователя для O(1) поиска
- Автоматическая отсрочка при превышении лимита
- Отдельные лимиты для вызовов API, отправки сообщений и вызовов инструментов

-----------------------------------------------------------------------------

## Слой 14: Аудиторский след

**Крейты:** `orexis`, `timeline` (3 096 строк)

Каждый вызов инструмента, решение агента и событие безопасности:

1. Записывается во **временную шкалу** с полным контекстом (значок агента, имя навыка, параметры, результат)
1. Хэш-связывается с предыдущими событиями для обнаружения подделки
1. Сохраняется в PostgreSQL с настраиваемым сроком хранения
1. Доступен для запросов через CLI (`entelecheia-cli trace-chain <badge>`)

## Модель угроз

### Вне области

- Физический доступ к хостовым машинам
- Скомпрометированный демон Docker/Podman (считается доверенным)
- Эксплойты ядра (смягчаются, но не предотвращаются изоляцией пользовательского пространства)
- Атаки на цепочку поставок зависимостей Rust-крейтов (частично смягчаются `cargo-deny`)

### Принятые риски

- Уязвимости движка Boa JS (изолирован в контейнере)
- Недоступность провайдеров LLM (нет резервного пути выполнения)
- Повреждение данных PostgreSQL (смягчается резервным копированием, не предотвращается)
