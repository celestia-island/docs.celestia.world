# ADR-003: 통합 데이터 저장소를 위한 PostgreSQL + PgVector

**일자**: 2026-02
**상태**: 승인됨

## 배경

Entelecheia는 **두 가지 서로 다른 책임을 동시에** 수행하는 데이터 저장 계층을 필요로 합니다:

1. **전통적 관계형 데이터**: 사용자 계정, API 크리덴셜, 에이전트 세션, 태스크 메타데이터, 감사 로그, RBAC 정책, 컨테이너 상태, 타임라인 이벤트 — 관계형 무결성, 트랜잭션, SQL 쿼리의 혜택을 받는 모든 구조화된 데이터.

1. **벡터 유사도 검색**: RAG(검색 증강 생성), 기억 침전, 지식 그래프 탐색, 문서 검색을 위한 임베딩 — 고차원 최근접 이웃 검색이 필요한 데이터.

다음과 같은 여러 저장소 접근 방식이 평가되었습니다:

| 접근 방식 | 관계형 데이터 | 벡터 검색 | 이중 책임 | 성숙도 |
| --- | --- | --- | --- | --- |
| **PostgreSQL + PgVector** | 완전한 SQL, ACID 트랜잭션 | HNSW/IVFFlat 인덱스 | 단일 데이터베이스, 단일 쿼리 언어 | PostgreSQL: 35년 이상; PgVector: 안정적, 널리 배포됨 |
| **Qdrant** | 없음 (벡터 전용) | 우수 (목적 특화) | 별도 관계형 DB 필요 | 보통 |
| **Milvus** | 없음 (벡터 전용) | 우수 | 별도 관계형 DB 필요 | 보통 |
| **Weaviate** | 제한적 (내장 CRUD) | 양호 | 양쪽 모두 타협 | 보통 |
| **SQLite + 벡터 확장** | 완전한 SQL | 실험적 | 단일 파일, 제한된 동시성 | 낮음 (벡터 확장 미성숙) |
| **MongoDB + Atlas Vector Search** | 문서 저장소, SQL 없음 | 양호 | 타협된 쿼리 모델 | 높음 (단, 독점적 벡터 검색) |

## 결정

통합 저장소 백엔드로 **PostgreSQL with PgVector 확장**을 선택했습니다.

**주요 이유:**

1. **안정성과 검증된 신뢰성.** PostgreSQL은 35년 이상의 개발 역사, 성숙한 쿼리 최적화기, 견고한 ACID 트랜잭션, 방대한 도구 생태계를 갖춘 가장 실전 검증된 오픈소스 관계형 데이터베이스 엔진입니다. 인증 크리덴셜, RBAC 정책, 감사 로그를 처리하는 시스템에 있어 이 안정성은 선택 사항이 아닌 기본 요구 사항입니다. 이러한 워크로드에 더 새롭고 덜 검증된 데이터베이스를 선택하는 것은 불필요한 운영 위험을 초래할 것입니다.

1. **통합 이중 목적 저장소.** PgVector는 별도 데이터베이스 없이 PostgreSQL에 벡터 유사도 검색(HNSW 및 IVFFlat 인덱싱)을 확장합니다. 이는 다음을 의미합니다:

   - **관리할 데이터베이스가 하나**, 하나의 연결 풀, 하나의 백업 전략, 하나의 마이그레이션 시스템.
   - **관계형 데이터와 벡터 데이터 간 JOIN 쿼리** — 예: "사용자가 접근 권한을 가진 이 임베딩과 유사한 문서 찾기"를 단일 SQL 쿼리로 표현 가능.
   - 메타데이터 갱신과 임베딩 삽입 간의 **트랜잭션 일관성**.
   - **SQL을 범용 쿼리 언어로** — 개발자가 별도 벡터 쿼리 DSL을 배울 필요 없음.

1. **PgVector는 우리 규모에서 비교적 안정적입니다.** PgVector가 10억 규모 배포를 위한 목적 특화 벡터 데이터베이스만큼 최적화되지는 않았지만, Entelecheia 워크로드(에이전트 기억, 지식 문서, RAG 컨텍스트)를 충분히 처리합니다. 임베딩 차원(768-3072)과 데이터셋 크기(수천에서 낮은 수백만 벡터)는 PgVector의 편안한 영역 내에 있습니다.

1. **팀 친숙도와 생태계.** PostgreSQL은 세계에서 가장 널리 배포된 데이터베이스 엔진입니다. 팀은 SQL, PostgreSQL 관리, Rust ORM 생태계(SeaORM, SQLx)에 깊은 친숙도를 가지고 있습니다. 익숙하지 않은 벡터 데이터베이스를 선택하는 것은 미미한 이점을 위해 상당한 학습 투자가 필요할 것입니다.

1. **SQL 호환성에 대한 타협 없음.** 많은 새로운 벡터 데이터베이스는 SQL을 전혀 지원하지 않거나 제한된 방언만 지원합니다. 이는 애플리케이션이 관계형 데이터용과 벡터 검색용의 두 개별 쿼리 모델을 유지하도록 강제하여 코드 복잡성과 버그 발생 가능성을 증가시킬 것입니다.

## 결과

### 긍정적

- **단일 운영 표면**: 모니터링, 백업, 업그레이드, 디버깅할 데이터베이스가 하나입니다. SeaORM 마이그레이션이 스키마와 확장 설정을 모두 처리합니다.
- **트랜잭션 벡터 연산**: 임베딩 삽입과 메타데이터 갱신이 동일한 트랜잭션에서 발생하여 고아 또는 불일치 데이터를 방지합니다.
- **결합 쿼리를 위한 완전한 SQL 능력**: 권한 인식 벡터 검색, 시간 필터링된 유사도, 다중 테이블 조인이 네이티브 SQL 연산입니다.
- **SeaORM + PgVector 통합**: Rust 생태계는 성숙한 PostgreSQL 지원을 갖추고 있습니다. SeaORM 엔터티는 거리 연산자가 포함된 벡터 열을 포함할 수 있습니다.
- **프로덕션 검증 완료**: PostgreSQL의 복제, 특정 시점 복구, 연결 풀링(PgBouncer), 모니터링(`pg_stat_statements`)은 산업 표준입니다.

### 부정적

- **벡터 검색 성능 상한**: 매우 큰 임베딩 컬렉션(1억 개 이상의 벡터)의 경우, 목적 특화 벡터 데이터베이스(Qdrant, Milvus)가 PgVector를 크게 능가합니다. Entelecheia의 현재 규모는 이 한계에 접근하지 않지만, 이는 미래 고려 사항입니다.
- **PgVector는 핵심이 아닌 확장입니다**: PgVector는 PostgreSQL과 별도로 설치 및 유지 관리되어야 합니다. 컨테이너 이미지는 확장을 포함해야 합니다(우리는 `pgvector/pgvector:pg18`을 사용합니다). PostgreSQL을 업그레이드하려면 확장 재컴파일이 필요할 수 있습니다.
- **제한된 벡터 인덱스 유형**: PgVector는 HNSW와 IVFFlat을 지원합니다. 목적 특화 벡터 데이터베이스는 특정 분포에 더 효율적일 수 있는 더 전문화된 인덱스 유형(예: DiskANN, ScaNN)을 제공합니다.
- **리소스 경쟁**: 벡터 인덱싱(특히 HNSW 구축)은 동일한 PostgreSQL 인스턴스에서 OLTP 워크로드와 공유되는 CPU 및 메모리를 소비합니다. 규모에 따라 벡터 워크로드를 전용 복제본으로 분리하는 것이 필요해질 수 있습니다.

### 수용된 트레이드오프

**운영 단순성을 위한 성능 상한.** 이중 데이터베이스 아키텍처(관계형용 PostgreSQL + 벡터용 Qdrant/Milvus)는 규모에서 더 나은 벡터 검색 성능을 제공할 것입니다. 그러나 이는 운영 복잡성을 두 배로 늘리고, 두 시스템 간 데이터 동기화가 필요하며, 일관성 문제를 초래할 것입니다. Entelecheia의 현재 및 단기 규모(단일 사용자에서 소규모 팀 배포)에서는 통합 PostgreSQL 접근 방식이 올바른 트레이드오프입니다. 향후 벡터 검색이 병목이 된다면, 애플리케이션의 쿼리 모델을 변경하지 않고 PgVector가 있는 읽기 복제본이나 전용 벡터 캐시 계층을 점진적으로 도입할 수 있습니다.
