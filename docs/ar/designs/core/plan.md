# خطة التكامل الذاتي لممر صناعي تجريبي

> **الهدف**: يجب أن يُظهر النظام **تكامل واجهة ذاتي مستقل** مع ممر تجريبي صناعي مجهول
> تمامًا — اكتشاف الأجهزة، استنتاج نماذج البيانات، توليد تكوين المراقبة، وإغلاق حلقة
> الإنذار ← الاستجابة — دون هندسة يدوية لكل جهاز.
> **الموعد النهائي الحكومي الصارم**: ترتبط هذه القدرة بمعلم مشروع حكومي.

## العمل المتبقي

سلسلة الاكتشاف ← الاستنتاج ← المراقبة ← الإنذار ← **موافقة الكتابة** الكاملة قد
شُحنت (المراحل A.1–A.3، B، C، D.1، **D.2 ✓**). العمل الوحيد المتبقي هو **التحقق
الشامل من التشغيل الذاتي (المرحلة E)** — عملياتي وليس كودًا.

### D.2 — رحلة موافقة الكتابة (بشرية في الحلقة) ✓

```text
Agent decides write is needed
  → verify_write_safety → Denied
    → orexis.request_write_approval → WriteApprovalRequest broadcast
      → shittim-chest shows approval dialog (industrial.approveWrite)
        → [approved] → temporary whitelist entry → execute + read-back verify
        → [denied]   → agent receives denial, adjusts plan
```

## 1. عنونة الشارة الفرعية + التنفيذ المتوازي للمهارات

**الحكم**: يستحق التنفيذ. البنية التحتية مبنية بنسبة ~80%، ينقصها فقط الـ 20% الأخيرة.

**الحالة الحالية**:

- يدعم `BadgeRegistry` (`packages/scepter/src/state_machine/badge_registry.rs:92-120`) بالفعل `link_sessions()` بين الوالد والطفل.
- تحليل صيغة الشارة الفرعية `#001.005` موجود في `find_by_container_id_or_sub()` لكنه يتجاهل الرقم الفرعي بدلاً من حله إلى حاوية ابنة مميزة.
- حقلا `SnowflakeContainer.parent_id` و `branch_level` موجودان لكنهما للبيانات الوصفية فقط — لا يُستخدمان أبدًا للتوجيه.
- طابور أولوية عقدة الحافة (`edge_node_registry.rs:73-126`) جاهز للقفل الدقيق للموارد.
- سلسلة المهارات **تسلسلية** بشكل صارم — `pipeline.rs:68-226` تدور حول مهارة واحدة في كل مرة. المهارات المنسقة ذات `next_targets` المستقلة تعمل تسلسليًا بينما يمكنها العمل بالتوازي.

**ما هو مفقود**:

1. ✅ جعل `find_by_container_id_or_sub()` يحل `#001.005` ← الابن المتفرد

الأعمق للحاوية الوالد، مع الرجوع للوالد عندما
لا يوجد تفرع (متوافق مع الإصدارات السابقة).

1. ✅ إضافة بحث الابن/السليل إلى `SnowflakeManager`: `children_of`،

`children_of_badge`، `most_recent_child_of`، `deepest_descendant`
(`parent_id` ← فهرس عكسي).

1. ✅ التنفيذ المتوازي المعتمد على `FuturesUnordered` لـ `next_targets`:

`dispatch_parallel_targets` ينشر أهداف **الأوراق** المستقلة للمنسق

بالتزامن عبر `parallel_dispatch::fan_out` (مقيد بـ

`Semaphore`). يُعالج الحاجان أحاديّا النمط العالميان في مسار

## 2. خط أنابيب ترسيب الذاكرة

**الحكم**: مُضاعف للجودة وليس حرجًا. محجوز لخارطة الطريق طويلة الأمد.

**الحالة الحالية**:

- `PhiliaMemoryService` رسم بياني مسطح "خزّن ← ادمج ← استرجع" بدون أيضِيض (استقلاب).
- `memory_consolidate` تافه — ينشئ فقط عقدة حلقة، بدون تجريد/ت تلخيص.
- لا يوجد اضمحلال للذاكرة، أو شيخوخة، أو تقييم قديم، أو تدرج جودة عبر العقد.
- جميع العقد `MemoryNode` غير متمايزة — لا يوجد فصل حلقي/إجرائي/ذري.
- البحث المتجه في الذاكرة هو اجتياح قسري O(n) (لن يتوسع طويل الأمد).
- يمتلك `KnowledgeStore` (نظام منفصل) مراحل دورة حياة (مُنشأ←مُتجه←قابل للبحث←مُوحد←مُهمَل) والتحقق بالإجماع — أقرب نظير موجود للترسيب.

**لماذا ليس عاجلاً**:

- حقن سياق RAG (`RagContextBuffer` ← إعادة كتابة استعلام LLM ← `bundle_search`) يوفر سياقًا كافيًا لوكلاء استدعاء الأدوات الحاليين.
- فهرس pgvector HNSW يتعامل مع الاسترجاع بمقياس الإنتاج.
- يعمل النظام كـ "خزّن واسترجع" — الترسيب سيجعله "يستقلب"، لكن هذه جودة تزايدي وليست فجوة وظيفية.

**العمل المستقبلي** (بلا جدول زمني):

- التوحيد التلقائي: تلخيص دوري بقيادة LM للعقد ذات الصلة إلى "حلقات" ذات مستوى أعلى.
- تدرج الجودة: عدادات الوصول، الاضمحلال الزمني، تقييم الثقة.
- نموذج أولي ثلاثي القنوات (حلقي/إجرائي/ذري) باستراتيجيات استرجاع متمايزة.

## 3. التفاوض بين الوكلاء

**الحكم**: أولوية منخفضة. البدائيات موجودة ككتل بناء منخفضة المستوى؛ لا توجد حالة استخدام فورية.

**الحالة الحالية**:

- `deliver_message(message_type="Question")` موجود (`epieikeia/src/mcp/tools/deliver_message.rs:63`) — يمكنه دفع الأسئلة إلى صندوق بريد وكيل آخر.
- `inject_user_prompt` / `consume_injected_prompts` موجودان لكنهما **قائميان على الاستطلاع** — بلا تكامل مع خط الأنابيب. يجب على الوكلاء استدعاء `consume_injected_prompts` صراحةً لفحص البريد.
- يمتلك `Haplotes` أنواع توجيه المحادثة `AskAgent` / `ReplyAgent` / `Escalated` — لكنها جميعًا إقرارات استلام لا فعلية صفر منطق أعمال.
- متغيرات البيئة `NEGOTIATION_ROUND_TIMEOUT_SECS` / `NEGOTIATION_TOTAL_TIMEOUT_SECS` معرّفة في `RuntimeTuningConfig` لكنها **لا تُستهلك أبدًا** في أي مكان — كود ميت.

**لماذا أولوية منخفضة**:

- توجيه سلسلة المهارات المتسلسل الحالي + تمرير السياق كسلسلة يتعامل مع جميع حالات الاستخدام الحالية.
- تُعالج تعارضات الدمج بواسطة توجيه مهارة واحدة (`resolve_merge_conflict`)، وهو كافٍ.
- حلقة التفاوض (اعتراض سلسلة المهارات ← سؤال وكيل ← انتظار الاستجابة ← دمج) ستكون معقدة البناء والاختبار. لا حالة إنتاج تطلبها بعد.

**متى يُعاد الزيارة**: إذا احتاج الوكلاء يومًا للتفاوض الديناميكي على قرارات منتصف السلسلة (وليس فقط التوجيه والانتظار)، فالبدائيات مبنية بنسبة 40%. الفجوة هي حلقة تكامل خط الأنابيب.
