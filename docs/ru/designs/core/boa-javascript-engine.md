# ADR-002: Boa как Встроенный Движок JavaScript

**Дата**: 2026-02
**Статус**: Принято

## Контекст

IEPL (Interactive Execution Pipeline Layer) требуется среда выполнения JavaScript для исполнения кода, сгенерированного LLM, внутри каждого контейнера COSMOS. Эта среда выполнения должна:

1. Быть **встраиваемой** в приложение Rust — она будет работать как init-процесс (PID 1) внутри легковесных контейнеров.
1. Быть **безопасной и изолированной** — код, сгенерированный LLM, по своей природе не является доверенным.
1. Поддерживать **импорты ES модулей** для механизма диспетчеризации инструментов (например, `import { file_read } from 'agent'`).
1. Иметь **минимальные накладные расходы на запуск** — контейнеры эфемерны и должны быть готовы быстро.
1. Быть **кроссплатформенной** и легко компилируемой для целевой среды контейнера.

Было оценено несколько сред выполнения JavaScript/TypeScript:

| Среда выполнения | Язык | Встраиваема в Rust | Контроль Изоляции | Скорость Запуска | Поддержка ES Модулей |
| --- | --- | --- | --- | --- | --- |
| **Boa** | Чистый Rust | Нативная (крейт Rust) | Полный (хост контролирует всё) | Быстрая (<10мс) | Частичная (достаточно для IEPL) |
| **Deno** | Rust + V8 | Возможно через FFI | Ограниченный (изоляты V8) | Медленная (~100мс) | Полная |
| **QuickJS** | C | Через FFI/привязки | Умеренный | Быстрая | Частичная |
| **V8** | C++ | Через FFI (крейт v8) | Ограниченный | Медленная | Полная |
| **wasmoon** | C (Lua) | Через FFI | Хороший | Быстрая | Н/Д (Lua) |
| **rquickjs** | C (QuickJS) | Через FFI | Умеренный | Быстрая | Частичная |

## Решение

Мы выбрали **Boa Engine** (v0.21) в качестве встроенной среды выполнения JavaScript.

**Основные причины:**

1. **Чистый Rust — нулевые накладные расходы FFI.** Boa полностью написан на Rust, что означает, что он компилируется нативно в бинарный файл COSMOS без цепочки зависимостей C/C++. Это устраняет целый класс уязвимостей безопасности, связанных с FFI, сложность сборки и проблемы кросс-компиляции. В контексте контейнера COSMOS, где мы контролируем init-процесс, это критическое преимущество.

1. **Простота "вызвать и использовать".** Boa спроектирован как движок, ориентированный на библиотеки. Его можно создать, настроить с хост-функциями и выполнить в нескольких строках кода Rust. Нет отдельного процесса для управления, нет моста IPC для поддержки, нет сложности цикла событий. `JsReplHandle` в COSMOS создаёт выделенный поток ОС для среды выполнения Boa и взаимодействует через стандартные каналы Rust — чистая, компонуемая архитектура.

1. **Безопасность — высший приоритет, а не производительность.** В изолированной среде COSMOS каждая миллисекунда времени выполнения JS не находится на критическом пути — узким местом всегда является время обращения к LLM. Важно то, что среда выполнения даёт нам **полный контроль** над тем, что может делать выполняемый код. API регистрации хост-функций Boa позволяет нам точно определить, какие функции доступны (только функции диспетчеризации инструментов MCP), без путей обхода. Валидатор безопасности AST (который блокирует `eval`, `require`, `process` и т.д.) работает с AST Boa, давая нам нативную гарантию принудительного исполнения на Rust.

1. **Достаточная поддержка ES модулей для IEPL.** Конвейер IEPL использует симулированную систему модулей — импорты ES модулей разрешаются на уровне построителя пространства имён, а не реальным загрузчиком модулей. Возможностей Boa более чем достаточно для этого шаблона. Нам не нужен полноценный алгоритм разрешения модулей, совместимый с Node.js.

## Последствия

### Положительные

- **Отсутствие зависимости сборки C/C++** — COSMOS чисто компилируется просто с `cargo build`, без требований к системным библиотекам.
- **Полный контроль изоляции** — Каждая функция, доступная выполняемому JS-коду, явно регистрируется хостом. Нет доступа к вводу-выводу, сети или файловой системе по умолчанию.
- **Тесная интеграция с типами Rust** — реализация трейта `boa_gc::Trace` для пользовательских хост-объектов, нативное взаимодействие с `serde_json`, нулевое копирование где возможно.
- **Безопасность при сбоях** — Паники Boa перехватываются на границе потока ОС, предотвращая сбой процесса COSMOS из-за некорректного JS-кода.
- **Малый размер бинарного файла** — По сравнению с решениями на основе V8, Boa добавляет значительно меньше к размеру бинарного файла COSMOS.

### Отрицательные

- **Соответствие JavaScript неполное** — Boa не полностью реализует ECMAScript 2024+. Некоторые продвинутые функции (например, `WeakRef`, `FinalizationRegistry`, полная интеграция `Promise`, `async/await` во всех контекстах) могут иметь ограничения или отсутствующие реализации.
- **Производительность не конкурентоспособна с V8/`SpiderMonkey`** — Интерпретатор Boa значительно медленнее, чем JIT-компилируемые движки. Для задач с интенсивным использованием CPU (обработка больших данных, сложные алгоритмы) это имеет значение. Однако в контексте IEPL JS-код является в первую очередь связующим звеном оркестровки, вызывающим инструменты MCP, а не вычислениями.
- **Экосистема меньше** — У Boa меньше контрибьюторов и меньше боевого тестирования, чем у V8 или QuickJS. Исправление ошибок в upstream может занять больше времени.
- **Нет `eval` или динамической генерации кода** — По дизайну (и enforced валидатором AST) динамическое выполнение кода заблокировано. Это ограничивает определённые шаблоны метапрограммирования, но приемлемо для модели безопасности.

### Принятый Компромисс

**Жертва производительностью ради безопасности и встраиваемости.** Если бы движку выполнения IEPL требовалось выполнять сложные алгоритмы или обрабатывать большие наборы данных, Boa был бы неправильным выбором. Но в изолированной среде COSMOS JS-код является тонким слоем оркестровки — его задача вызывать инструменты MCP в правильном порядке, обрабатывать ошибки и компоновать результаты. Фактическая тяжёлая работа выполняется инструментами агентов на основе Rust. Скорость выполнения Boa в 10-100 раз медленнее по сравнению с V8 не имеет значения, когда каждый вызов инструмента включает сетевой обход к Scepter, занимающий 50-500мс.
