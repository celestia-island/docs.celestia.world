# コアコンセプト

> **対象**: shittim-chestの設計を概念的に理解したい開発者。
> **最終更新日**: 2026-05-25

## 2リポジトリアーキテクチャ

shittim-chestと[entelecheia](https://github.com/celestia-island/entelecheia)は、意図的な境界を持つ2リポジトリシステムを形成します:

- **entelecheia** — エージェントオーケストレーションコア（scepter、13エージェント、Cosmos/IEPLランタイム）。アイデンティティ、権限、エージェント設定を所有。
- **shittim-chest** — ユーザー向けシェル。認証、セッション、チャットデータ、LLMプロバイダー設定、フロントエンドUI、scepterへのプロキシブリッジを所有。

両者はJWT認証されたHTTPおよびWebSocketを通じて通信します。どちらも相手のデータベースに直接アクセスすることはありません。この分離により、各リポジトリは独立して開発、デプロイ、スケールできます。

## デュアル動作モード

shittim-chestは2つの動作モードをサポートします:

### スタンドアロンモード

独自のLLMルーティング層で独立して実行されます。以下をサポート:

- ストリーミング応答付きチャット（SSE + WebSocket）
- 設定済みプロバイダー経由の画像生成
- ユーザー認証（パスワード + GitHub OAuth）
- プロバイダー管理（LLMプロバイダーの追加/削除）

entelecheiaを必要としません。開発やシンプルなデプロイメントに有用です。

### プロキシモード

entelecheiaのエージェントシステムへのゲートウェイとして機能します。以下を追加:

- JWTパススルー付きscepterへのリクエスト転送
- エージェントベースチャット用WebSocketブリッジング
- Webhookイングレスとトリガー転送
- polemos経由のリモートデバイス管理
- RBAC権限クエリとキャッシング

動作中のentelecheiaインスタンスが必要です。2つのモードは共存可能です — シンプルなチャットにはスタンドアロンLLM、エージェントオーケストレーションにはプロキシ。

## 認証モデル

認証はshittim_chestが発行するJWTトークンを使用します:

1. **認証情報ストレージ**: パスワード（argon2ハッシュ）、セッション、リフレッシュトークン、APIキーは`shittim_chest_db`に保存。
1. **GitHub OAuth**: ユーザーはGitHubでサインイン可能。初回ログイン時にアカウントが自動作成。
1. **権限ストレージ**: ユーザーグループ、ロール、権限マトリックスは`entelecheia_db`に保存。
1. **JWTフロー**: ログイン時に、shittim_chestがローカルで認証情報を検証し、次にscepterから権限を取得。発行されるJWTは`{ sub: user_id, groups: [...] }`を含む。
1. **共有秘密鍵**: JWT署名秘密鍵はscepterと共有されるため、両方のサービスが独立してトークンを検証可能。
1. **トークンローテーション**: アクセストークンは1時間で期限切れ、リフレッシュトークンは7日間。リフレッシュトークンは使用ごとにローテーション。

## フロントエンド（webui）

webuiは`packages/webui/`にある統合フロントエンドで、チャットインターフェースは`/`に、管理パネルは`/backend`にあり、Vue 3 + Vite + Pinia（`@vitejs/plugin-vue-jsx`経由のTSX）で構築されています。

## LLMプロバイダーシステム

shittim-chestは独立したLLMルーティング層を持ちます:

- **プロバイダー**: 設定可能なLLM APIエンドポイント（OpenAI互換）。AES-256-GCMで暗号化されたAPIキーと共に`shittim_chest_db`に保存。
- **ルーター**: 優先度ベースの選択と自動フォールバックを持つマルチプロバイダールーティング。
- **カテゴリ**: プロバイダーは`chat`、`image`、またはその両方としてタグ付け可能。
- **管理**: REST APIおよびwebui管理パネルを通じた完全なCRUD。プロバイダーは接続テスト可能。
- **ストリーミング**: SSE（シンプル、プロキシフレンドリー）とWebSocket（双方向）の両方のストリーミングプロトコル。

## チャットシステム

- **会話**: タイトルとメタデータを持つスレッドベースのチャットセッション
- **メッセージ**: テキスト、画像、ツール呼び出し（関数呼び出し）をサポート
- **ストリーミング**: SSEまたはWebSocket経由のリアルタイムトークン単位の応答配信
- **検索**: ILIKEクエリによる全文メッセージ検索
- **エクスポート**: 会話はJSONまたはMarkdown形式でエクスポート可能
- **画像生成**: 設定済みプロバイダー経由のプロンプトベース画像生成、「チャットに挿入」機能付き

## リモートデバイス管理

shittim-chestはentelecheia/polemosによって管理されるリモートデバイス用のブラウザベースのインターフェースを提供します:

- **デスクトップ**: フレームリレー付きWebRTCベースのリモートデスクトップビューアー
- **ターミナル**: WebSocketリレー付きxterm.jsベースのターミナルエミュレーター
- **ファイルブラウザ**: SFTPファイルブラウザバックエンド（スケルトン）
- **シグナリング**: WebSocketベースのWebRTCシグナリングリレー（SDPオファー/アンサー、ICE候補）

すべてのデバイス通信はentelecheiaのpolemosエージェントを通じて流れます — shittim-chestがエンドポイントに直接接続することは決してありません。

## プロキシアーキテクチャ

shittim_chestはユーザーとscepter間のゲートウェイとして機能します:

- **HTTPリバースプロキシ**: `/api/proxy/*`は認証済みリクエストをJWTパススルー付きでscepterに転送。
- **WebSocketブリッジ**: チャットストリーミングは双方向WebSocket転送を使用（`browser ↔ shittim_chest ↔ scepter`）。

これにより、shittim_chestはscepterが個々のブラウザ接続を処理する必要なしに、レート制限を強制し、使用量をログに記録し、接続ライフサイクルを管理できます。

## Webhookパイプライン

外部イベントはwebhookパイプラインを通じてエージェントコアに到達します:

```text
GitHub/GitLab/Gitee → POST /api/webhook/{source} → HMAC検証 → イベント解析 → Unixソケット経由でscepterに転送 → エージェントディスパッチ
```

各プロバイダーは独自の検証メカニズムを持ちます:

- **GitHub**: `X-Hub-Signature-256`によるHMAC-SHA256
- **GitLab**: `X-Gitlab-Token`によるトークン
- **Gitee**: トークンフォールバック付きHMAC

追加機能: 重複配信検出（LRUキャッシュ）、配信ログ、IPホワイトリスト、汎用カスタムwebhookエンドポイント。

## RBACモデル

権限はグループベースのRBACモデルに従います:

- **グループ**: ユーザーは1つ以上のグループに所属。
- **ロール**: グループには割り当てられたロールがある。
- **権限**: 各ロールは以下をカバーする権限マトリックスを定義:
  - プロバイダークォータ（最大トークン数、最大リクエスト数）
  - エージェントホワイトリスト（グループがアクセス可能なエージェント）
  - 管理機能（ユーザー管理、プロバイダー設定）

shittim_chestはプロセス内で権限をTTL（デフォルト5分）でキャッシュします。キャッシュ無効化はTTL期限切れ、ログアウト、またはscepterから伝播された明示的な権限変更時に発生します。

## フロントエンド戦略

shittim-chestは2フェーズのフロントエンドアプローチを使用します:

**フェーズ1（現在）**: `@vitejs/plugin-vue-jsx`経由でTSXを使用し、Vite + Piniaで構築されたVue 3フロントエンド（`webui`、`packages/webui/`）。API契約を定義し、プロダクション品質のリファレンス実装として機能します。

**フェーズ2（将来）**: Tairitsuで構築されたRust → WASMフロントエンド。レガシーフロントエンドは生きた仕様およびテストオラクルとして機能します — 同一のユーザー操作は同一の結果を生成しなければなりません。

## 型安全性ブリッジ

TypeScript型はRustコードから外部`arona`プロトコルクレート経由で生成され、フロントエンド-バックエンドの一貫性を確保します:

```text
arona Rustクレート（git依存関係）
  → #[derive(ts_rs::TS)]
  → ts-rs codegen → packages/webui/src/types/arona/（TypeScript）
  → @celestia-island/aronaとしてwebuiで消費
```

これにより手動の型同期が不要になります。`arona`クレートのRust型が変更されると、TypeScriptバインディングが再生成され、webuiで消費されます。
