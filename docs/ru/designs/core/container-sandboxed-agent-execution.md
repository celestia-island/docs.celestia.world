# ADR-005: Изолированное в Контейнере Выполнение Агентов с COSMOS

**Дата**: 2026-02
**Статус**: Принято

## Контекст

В мульти-агентной системе, где агенты выполняют код, сгенерированный LLM, изоляция между агентами критична для:

1. **Безопасности**: Недоверенный вывод LLM не должен иметь доступ к памяти, файлам или сетевым соединениям другого агента.
1. **Изоляции состояния**: Состояние REPL каждого агента (переменные JavaScript, привязки, снимки) должно быть независимым.
1. **Контроля ресурсов**: Неправильно ведущий себя агент не должен потреблять неограниченный CPU, память или PID.
1. **Воспроизводимости**: Состояние агента должно быть доступно для снимков и восстановления для отладки и отката.
1. **Рабочих процессов fork/merge**: Система должна поддерживать ветвление выполнения агентов (fork) и слияние результатов обратно (merge), аналогично ветвлению git.

Было оценено несколько подходов к изоляции:

| Подход | Сила Изоляции | Контроль Ресурсов | Снимок/Fork | Накладные Расходы |
| --- | --- | --- | --- | --- |
| **Контейнер на агента (Docker/OCI)** | Сильная (уровень ядра) | Полный (cgroups, seccomp, capabilities) | Нативный (commit/snapshot) | Умеренные (~100мс запуск, ~50МБ на контейнер) |
| **Процесс на агента** | Умеренная (UID/seccomp) | Частичный (rlimit) | Ручной (сериализация состояния) | Низкие |
| **Поток на агента** | Слабая (общая память) | Минимальный | Ручной | Минимальные |
| **Песочница WASM на агента** | Сильная (линейная память) | Хороший (учёт газа) | Ручной | Низкие |
| **Контекст Boa на агента** | Умеренная (песочница JS) | Ограниченный | Встроенный (сериализация пространства имён) | Минимальные |

## Решение

Мы выбрали **двухуровневую контейнерную архитектуру** с **COSMOS** в качестве init-процесса внутри контейнера каждого агента:

**Внешний уровень (инфраструктура оркестровки):**

- Docker/Podman через Bollard для инфраструктурных контейнеров (PostgreSQL, демон Scepter).
- Полные возможности оркестровки: сеть, тома, проверки здоровья, compose.

**Внутренний уровень (песочницы агентов):**

- Youki/libcontainer (по умолчанию) или Docker для контейнеров COSMOS на агента.
- Каждый агент получает свой контейнер с COSMOS в качестве PID 1.
- COSMOS — это **процесс переднего плана**, который опосредует все взаимодействия — он предоставляет сервер JSON-RPC Unix-сокета, Boa JS REPL, маршрутизатор MCP и соединение моста HapLotes обратно к Scepter.

**Почему COSMOS как обязательный посредник:**

Все взаимодействия с контейнеризованным агентом должны проходить через COSMOS. Прямое манипулирование контейнером (например, `docker exec` в контейнер) обходит модель безопасности, управление состоянием и аудиторский след. COSMOS предоставляет:

1. **Посредничество диспетчеризации инструментов**: `McpRouter` применяет списки разрешений, двойную авторизацию и уровни доверия до того, как любой инструмент достигнет агента.
1. **Персистентность состояния**: Система двойной буферизации снимков гарантирует, что состояние REPL переживает сбои.
1. **Мостовая коммуникация**: Мост HapLotes соединяет COSMOS обратно с Scepter для меж-агентной координации.
1. **Принудительное применение безопасности**: Профили Seccomp, политики исходящего трафика и ограничения возможностей применяются при создании контейнера и enforced ядром.

**Почему Youki/libcontainer для внутренних песочниц:**

- Без root и без демона — не требуется демон Docker для песочниц агентов.
- Совместим с OCI — стандартная спецификация `config.json`, совместима с инструментами OCI.
- Быстрая корневая файловая система на основе overlay — операции снимков и fork копируют только изменённые файлы.
- Меньше накладных расходов, чем Docker для эфемерных контейнеров.

## Последствия

### Положительные

- **Сильная изоляция через принуждение ядра**: cgroups (ограничения CPU/памяти/PID), seccomp (фильтрация системных вызовов), capabilities (`cap_drop`=ALL), пространства имён (изоляция PID/сети/монтирования).
- **Нативный fork/merge**: Фиксация контейнера создаёт снимок образа; новые контейнеры могут быть созданы из снимка. Оверлейные файловые системы отслеживают только изменённые файлы.
- **Ограничения ресурсов на агента**: 512МБ памяти, 1 CPU, 100 PID по умолчанию, настраивается на контейнер.
- **Аудиторский след**: Все вызовы инструментов проходят через маршрутизатор MCP COSMOS, который регистрирует каждую диспетчеризацию для аудита безопасности OreXis.
- **Сдерживание сбоев**: Паника Boa или ошибка агента ограничена его контейнером. Другие агенты и Scepter продолжают работу.
- **Youki для легковесных песочниц**: Внутренние контейнеры запускаются быстрее и потребляют меньше ресурсов, чем полные контейнеры Docker.

### Отрицательные

- **Сложность COSMOS как PID 1**: COSMOS должен обрабатывать пересылку сигналов, сбор зомби-процессов и чистое завершение работы как init-процесс контейнера. Это добавляет ответственность, которой нет у обычного приложения.
- **Задержка запуска контейнера**: Каждый контейнер агента требует ~100мс-1с для запуска (в зависимости от среды выполнения). Это медленнее, чем изоляция на основе процессов или потоков.
- **Накладные расходы ресурсов**: Каждый контейнер COSMOS потребляет ~50-100МБ памяти для среды выполнения Boa, кучи JS и накладных расходов ОС. С 9 контейнеризованными агентами это добавляет ~0.5-1ГБ базовой памяти.
- **Сложность тестирования**: Тестирование поведения агента требует запуска реальных контейнеров с COSMOS, что означает, что тестам нужен доступный Docker или Youki. Шаблон тестирования snowflake (сборка образа entelecheia, запуск контейнера COSMOS, подключение через Unix-сокет) сложнее, чем модульное тестирование.
- **Две среды выполнения для поддержки**: Необходимо поддерживать и тестировать оба пути кода — Docker/Bollard и Youki/libcontainer.

### Принятый Компромисс

**Накладные расходы ресурсов ради гарантий безопасности и изоляции.** Модель процесс-на-агента использовала бы меньше памяти и запускалась бы быстрее, но не обеспечивала бы изоляцию на уровне ядра между агентами. В системе, где агенты выполняют недоверенный код, сгенерированный LLM, гарантия безопасности изоляции контейнера стоит затрат ресурсов. Дизайн COSMOS как обязательного посредника гарантирует, что даже если злоумышленник получит выполнение кода внутри контейнера, он не сможет обойти модель безопасности, работая вне посредничества COSMOS.
