
# ADR-001: Поверхность Инструментов Микроядра Только для Выполнения

**Дата**: 2026-02
**Статус**: Принято (изменено 2026-06 — поверхность уменьшена с 5 до 3 примитивов)

## Контекст

В мульти-агентной системе оркестровки LLM модель должна решать, какие инструменты вызывать и как их компоновать. Наивный подход — предоставить каждый инструмент MCP (118+ на 12 агентов) напрямую LLM как отдельные определения функций в промпте.

Это создаёт несколько проблем:

1. **Потребление контекстного окна**: 118+ определений инструментов потребляют тысячи токенов, оставляя меньше места для рассуждений и разговора.
1. **Поверхность атаки безопасности**: Каждый инструмент, предоставленный LLM, является потенциальным вектором атаки для инъекции промпта или джейлбрейка.
1. **Фрагментация принудительного применения разрешений**: Если инструменты диспетчеризуются напрямую выводом LLM, каждый инструмент должен независимо проверять разрешения — что приводит к непоследовательному принуждению и пробелам.
1. **Путаница модели**: Исследования показывают, что производительность LLM снижается при наличии слишком большого выбора инструментов (проблема "перегрузки инструментов").

## Решение

Мы принимаем дизайн **микроядра только для выполнения**. LLM видит ровно **3 примитива выполнения** в качестве своей поверхности инструментов:

| Примитив | Назначение |
| --- | --- |
| `exec` | Выполнить код TypeScript/JavaScript через конвейер IEPL |
| `write_to_var` | Записать строковое значение в именованную переменную REPL |
| `write_to_var_json` | Записать значение JSON в именованную переменную REPL |

> **Изменение (2026-06)**: Первоначальный дизайн предоставлял 5 примитивов, включая `ref_add` и `ref_remove` для управления именованными ссылочными переменными. Эти два были **удалены** из видимой LLM поверхности, потому что механизм ссылочных переменных не использовался ни одним SOP навыка и добавлял накладные расходы контекста без ценности. Функция `agent_allowed_tools()` в `packages/shared/domain_skills/src/tool_names.rs` теперь возвращает только три примитива выше. См. историю коммитов (`60d58f794`, `31cf9a00e`).

Все 118+ инструментов MCP агентов вызываются **косвенно** через импорты ES модулей внутри кода `exec`. LLM генерирует код TypeScript, который импортирует и вызывает инструменты агентов; этот код транспилируется SWC, проверяется проверщиком безопасности AST и выполняется движком Boa JS внутри песочницы COSMOS.

## Последствия

### Положительные

- **Минимальные накладные расходы контекста**: 3 определения инструментов против 118+. LLM может сосредоточиться на рассуждениях, а не на механике выбора инструментов.
- **Централизованная безопасность**: Все вызовы инструментов проходят через `McpRouter`, который принудительно применяет списки разрешений, двойную авторизацию, уровни доверия и динамическую оценку риска в единой точке контроля.
- **Компонуемые рабочие процессы**: LLM может писать сколь угодно сложные композиции инструментов на TypeScript (циклы, условия, обработка ошибок), а не ограничиваться одиночными вызовами инструментов.
- **Аудируемое выполнение**: Каждый вызов `exec` проходит проверку AST, которая отклоняет опасные конструкции (`eval`, `require`, `process`, `Function`, `import()`, доступ к `globalThis`).
- **Более простая миграция моделей**: Новым провайдерам LLM нужно поддерживать вызов функций только с 3 инструментами, а не со 118.

### Отрицательные

- **Накладные расходы косвенности**: Вызовы инструментов проходят через генерацию TypeScript → транспиляцию SWC → проверку AST → выполнение Boa → диспетчеризацию маршрутизатора MCP, добавляя задержку к каждому вызову.
- **Зависимость от качества кода LLM**: Эффективность системы зависит от способности LLM генерировать корректный код TypeScript, который правильно импортирует и вызывает инструменты агентов.
- **Сложность отладки**: Когда вызов инструмента завершается сбоем, цепочка ошибок охватывает генерацию TypeScript, транспиляцию, проверку, выполнение JS и диспетчеризацию MCP — что делает отладку сложнее, чем прямой вызов инструмента.
- **Не все LLM равны**: Модели с более низкими возможностями могут испытывать трудности с генерацией кода для сложных мульти-инструментальных рабочих процессов по сравнению с простым вызовом функций.

### Смягчённые Риски

- Атаки инъекции промпта, пытающиеся напрямую вызывать опасные инструменты
- Случайное неправильное использование инструментов из-за путаницы LLM среди слишком большого выбора
- Непоследовательное принудительное применение разрешений между инструментами
