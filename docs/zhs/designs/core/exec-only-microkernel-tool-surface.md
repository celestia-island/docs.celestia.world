# ADR-001：仅执行微内核工具表面

**日期**：2026-02
**状态**：已接受（2026-06 修订——表面从 5 个原语减少到 3 个）

## 背景

在多 agent LLM 编排系统中，模型必须决定调用哪些工具以及如何组合它们。天真的方法是将每个 MCP 工具（12 个 agent 的 118+ 个工具）作为单独的函数定义直接暴露在提示中给 LLM。

这造成了几个问题：

1. **上下文窗口消耗**：118+ 个工具定义消耗数千个 token，留下更少的空间用于推理和对话。
1. **安全表面积**：每个暴露给 LLM 的工具都是提示注入或越狱的潜在攻击向量。
1. **权限执行碎片化**：如果工具由 LLM 输出直接调度，每个工具必须独立验证权限——导致不一致的执行和缺口。
1. **模型混淆**：研究表明，当面对太多工具选择时，LLM 性能会下降（"工具过载"问题）。

## 决策

我们采用**仅执行微内核**设计。LLM 看到恰好 **3 个执行原语**作为其工具表面：

| 原语 | 用途 |
| --- | --- |
| `exec` | 通过 IEPL 管道执行 TypeScript/JavaScript 代码 |
| `write_to_var` | 将字符串值写入命名的 REPL 变量 |
| `write_to_var_json` | 将 JSON 值写入命名的 REPL 变量 |

> **修订（2026-06）**：原始设计暴露了 5 个原语，包括用于管理命名引用变量的 `ref_add` 和 `ref_remove`。这两个已从 LLM 可见表面**移除**，因为引用变量机制未被任何技能 SOP 使用，并且增加了上下文开销而无价值。`packages/shared/domain_skills/src/tool_names.rs` 中的 `agent_allowed_tools()` 函数现在仅返回上述三个原语。参见提交历史（`60d58f794`、`31cf9a00e`）。

所有 118+ 个 agent MCP 工具通过 `exec` 代码内的 ES 模块导入**间接**调用。LLM 生成导入并调用 agent 工具的 TypeScript 代码；此代码由 SWC 转译，由 AST 安全检查器验证，并在 COSMOS 沙箱内由 Boa JS 引擎执行。

## 后果

### 正面

- **最小上下文开销**：3 个工具定义 vs 118+ 个。LLM 可以专注于推理而非工具选择机制。
- **集中式安全**：所有工具调用通过 `McpRouter`，它在单一扼流点中强制执行允许列表、双重授权、信任级别和动态风险评估。
- **可组合的工作流**：LLM 可以用 TypeScript 编写任意复杂的工具组合（循环、条件、错误处理），而非仅限于单个工具调用。
- **可审计的执行**：每个 `exec` 调用都通过 AST 验证，拒绝危险结构（`eval`、`require`、`process`、`Function`、`import()`、`globalThis` 访问）。
- **更容易的模型迁移**：新的 LLM 提供商只需支持 3 个工具的函数调用，而非 118 个。

### 负面

- **间接开销**：工具调用经过 TypeScript 生成 → SWC 转译 → AST 验证 → Boa 执行 → MCP 路由器调度，为每次调用增加了延迟。
- **LLM 代码质量依赖**：系统的有效性取决于 LLM 生成正确 TypeScript 代码的能力，该代码正确导入并调用 agent 工具。
- **调试复杂性**：当工具调用失败时，错误链跨越 TypeScript 生成、转译、验证、JS 执行和 MCP 调度——使得调试比直接工具调用更困难。
- **并非所有 LLM 同等**：能力较低的模型可能难以生成复杂多工具工作流的代码，而简单函数调用则不然。

### 已缓解的风险

- 试图直接调用危险工具的提示注入攻击
- 由于 LLM 在太多选择中混淆而导致的意外工具误用
- 跨工具不一致的权限执行
