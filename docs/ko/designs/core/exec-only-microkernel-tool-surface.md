
# ADR-001: Exec-Only 마이크로커널 도구 표면

**일자**: 2026-02
**상태**: 승인됨 (2026-06 수정 — 표면이 5개에서 3개 프리미티브로 축소)

## 배경

다중 에이전트 LLM 오케스트레이션 시스템에서, 모델은 어떤 도구를 호출할지 그리고 어떻게 구성할지 결정해야 합니다. 순진한 접근 방식은 모든 MCP 도구(12개 에이전트에 걸쳐 118개 이상)를 프롬프트에 별도 함수 정의로 LLM에 직접 노출하는 것입니다.

이는 여러 문제를 야기합니다:

1. **컨텍스트 윈도우 소비**: 118개 이상의 도구 정의가 수천 개의 토큰을 소비하여 추론과 대화에 더 적은 여유를 남깁니다.
1. **보안 표면적**: LLM에 노출된 모든 도구는 프롬프트 주입이나 탈옥의 잠재적 공격 벡터입니다.
1. **권한 강제의 단편화**: 도구가 LLM 출력으로 직접 디스패치되면, 각 도구가 독립적으로 권한을 검증해야 하며 — 일관성 없는 강제와 공백으로 이어집니다.
1. **모델 혼란**: 연구에 따르면 너무 많은 도구 선택지가 제시될 때 LLM 성능이 저하됩니다("도구 과부하" 문제).

## 결정

**exec-only 마이크로커널** 설계를 채택합니다. LLM은 도구 표면으로 정확히 **3개의 실행 프리미티브**만 볼 수 있습니다:

| 프리미티브 | 목적 |
| --- | --- |
| `exec` | IEPL 파이프라인을 통해 TypeScript/JavaScript 코드 실행 |
| `write_to_var` | 명명된 REPL 변수에 문자열 값 쓰기 |
| `write_to_var_json` | 명명된 REPL 변수에 JSON 값 쓰기 |

> **수정사항 (2026-06)**: 원래 설계는 명명된 참조 변수 관리를 위한 `ref_add`와 `ref_remove`를 포함하여 5개의 프리미티브를 노출했습니다. 이 두 가지는 참조 변수 메커니즘이 어떤 스킬 SOP에서도 사용되지 않았고 가치 없이 컨텍스트 오버헤드를 추가했기 때문에 LLM 가시 표면에서 **제거**되었습니다. `packages/shared/domain_skills/src/tool_names.rs`의 `agent_allowed_tools()` 함수는 이제 위의 세 가지 프리미티브만 반환합니다. 커밋 이력 참조 (`60d58f794`, `31cf9a00e`).

118개 이상의 모든 에이전트 MCP 도구는 `exec` 코드 내에서 ES 모듈 임포트를 통해 **간접적으로** 호출됩니다. LLM은 에이전트 도구를 임포트하고 호출하는 TypeScript 코드를 생성합니다; 이 코드는 SWC로 트랜스파일되고, AST 보안 검사기에 의해 검증되며, COSMOS 샌드박스 내의 Boa JS 엔진에 의해 실행됩니다.

## 결과

### 긍정적

- **최소 컨텍스트 오버헤드**: 118개 이상이 아닌 3개 도구 정의. LLM이 도구 선택 역학보다 추론에 집중할 수 있음.
- **중앙 집중식 보안**: 모든 도구 호출이 `McpRouter`를 통과하며, 이는 허용 목록, 이중 인증, 신뢰 수준, 동적 위험 평가를 단일 병목 지점에서 강제함.
- **구성 가능한 워크플로우**: LLM이 단일 도구 호출에 제한되지 않고 TypeScript로 임의로 복잡한 도구 구성을 작성할 수 있음 (루프, 조건문, 오류 처리).
- **감사 가능한 실행**: 모든 `exec` 호출이 위험한 구문(`eval`, `require`, `process`, `Function`, `import()`, `globalThis` 접근)을 거부하는 AST 검증을 통과함.
- **쉬운 모델 마이그레이션**: 새 LLM 제공자는 118개가 아닌 3개 도구만 함수 호출을 지원하면 됨.

### 부정적

- **간접 호출 오버헤드**: 도구 호출이 TypeScript 생성 → SWC 트랜스파일 → AST 검증 → Boa 실행 → MCP 라우터 디스패치를 거치며, 각 호출에 지연 시간 추가.
- **LLM 코드 품질 의존성**: 시스템의 효과성이 에이전트 도구를 올바르게 임포트하고 호출하는 올바른 TypeScript 코드를 생성하는 LLM의 능력에 달려 있음.
- **디버깅 복잡성**: 도구 호출 실패 시 오류 체인이 TypeScript 생성, 트랜스파일, 검증, JS 실행, MCP 디스패치에 걸쳐 있어 직접 도구 호출보다 디버깅이 더 어려움.
- **모든 LLM이 동등하지 않음**: 저성능 모델은 단순 함수 호출에 비해 복잡한 다중 도구 워크플로우의 코드 생성에 어려움을 겪을 수 있음.

### 완화된 위험

- 위험한 도구를 직접 호출하려는 프롬프트 주입 공격
- 너무 많은 선택지 중 LLM 혼란으로 인한 우발적 도구 오용
- 도구 간 불일치한 권한 강제
