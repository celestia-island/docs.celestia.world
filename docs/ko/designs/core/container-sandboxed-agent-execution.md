# ADR-005: COSMOS를 통한 컨테이너 샌드박스 에이전트 실행

**일자**: 2026-02
**상태**: 승인됨

## 배경

에이전트가 LLM 생성 코드를 실행하는 다중 에이전트 시스템에서, 에이전트 간 격리는 다음을 위해 중요합니다:

1. **보안**: 신뢰할 수 없는 LLM 출력이 다른 에이전트의 메모리, 파일, 또는 네트워크 연결에 접근할 수 없어야 함.
1. **상태 격리**: 각 에이전트의 REPL 상태(JavaScript 변수, 바인딩, 스냅샷)는 독립적이어야 함.
1. **자원 제어**: 오작동하는 에이전트가 무제한 CPU, 메모리, 또는 PID를 소비해서는 안 됨.
1. **재현성**: 에이전트 상태는 디버깅과 롤백을 위해 스냅샷 가능하고 복원 가능해야 함.
1. **포크/병합 워크플로우**: 시스템은 git 브랜칭과 유사하게 에이전트 실행 분기(포크)와 결과 병합(머지)을 지원해야 함.

여러 격리 접근 방식이 평가되었습니다:

| 접근 방식 | 격리 강도 | 자원 제어 | 스냅샷/포크 | 오버헤드 |
| --- | --- | --- | --- | --- |
| **에이전트별 컨테이너 (Docker/OCI)** | 강함 (커널 수준) | 완전 (cgroups, seccomp, capabilities) | 네이티브 (commit/snapshot) | 보통 (~100ms 시작, ~50MB/컨테이너) |
| **에이전트별 프로세스** | 보통 (UID/seccomp) | 부분적 (rlimit) | 수동 (상태 직렬화) | 낮음 |
| **에이전트별 스레드** | 약함 (공유 메모리) | 최소 | 수동 | 최소 |
| **에이전트별 WASM 샌드박스** | 강함 (선형 메모리) | 양호 (가스 계량) | 수동 | 낮음 |
| **에이전트별 Boa 컨텍스트** | 보통 (JS 샌드박스) | 제한적 | 내장 (네임스페이스 직렬화) | 최소 |

## 결정

각 에이전트의 컨테이너 내부에서 init 프로세스로 **COSMOS**를 사용하는 **2계층 컨테이너 아키텍처**를 선택했습니다:

**외부 계층 (오케스트레이션 인프라):**

- 인프라 컨테이너(PostgreSQL, Scepter 데몬)를 위한 Bollard를 통한 Docker/Podman.
- 완전한 오케스트레이션 기능: 네트워킹, 볼륨, 상태 확인, compose.

**내부 계층 (에이전트 샌드박스):**

- 에이전트별 COSMOS 컨테이너를 위한 Youki/libcontainer (기본값) 또는 Docker.
- 각 에이전트는 COSMOS를 PID 1로 하여 자체 컨테이너를 가짐.
- COSMOS는 모든 상호작용을 중재하는 **프론트엔드 프로세스** — JSON-RPC Unix 소켓 서버, Boa JS REPL, MCP 라우터, 그리고 Scepter로의 HapLotes 브리지 연결을 제공.

**COSMOS를 필수 중개자로 하는 이유:**

컨테이너화된 에이전트와의 모든 상호작용은 COSMOS를 통과해야 합니다. 직접적인 컨테이너 조작(예: 컨테이너로의 `docker exec`)은 보안 모델, 상태 관리, 감사 추적을 우회합니다. COSMOS는 다음을 제공합니다:

1. **도구 디스패치 중재**: `McpRouter`가 어떤 도구가 에이전트에 도달하기 전에 허용 목록, 이중 인증, 신뢰 수준을 강제함.
1. **상태 영속성**: 이중 버퍼 스냅샷 시스템이 REPL 상태가 충돌에도 살아남도록 보장.
1. **브리지 통신**: HapLotes 브리지가 에이전트 간 조정을 위해 COSMOS를 Scepter로 연결.
1. **보안 강제**: Seccomp 프로필, 송신 정책, 기능 제한이 컨테이너 생성 시 적용되고 커널에 의해 강제됨.

**내부 샌드박스를 위해 Youki/libcontainer를 사용하는 이유:**

- 루트리스 및 데몬리스 — 에이전트 샌드박스에 Docker 데몬 불필요.
- OCI 호환 — 표준 `config.json` 명세, OCI 도구와 호환.
- 빠른 오버레이 기반 rootfs — 스냅샷 및 포크 작업이 변경된 파일만 복사.
- 임시 컨테이너에 Docker보다 낮은 오버헤드.

## 결과

### 긍정적

- **커널 강제를 통한 강력한 격리**: cgroups(CPU/메모리/PID 제한), seccomp(시스템 호출 필터링), capabilities(`cap_drop`=ALL), 네임스페이스(PID/네트워크/마운트 격리).
- **네이티브 포크/병합**: 컨테이너 commit이 이미지 스냅샷 생성; 스냅샷에서 새 컨테이너 생성 가능. 오버레이 파일시스템이 변경된 파일만 추적.
- **에이전트별 자원 제한**: 512MB 메모리, 1 CPU, 100 PID 기본값, 컨테이너별 구성 가능.
- **감사 추적**: 모든 도구 호출이 COSMOS의 MCP 라우터를 통과하며, OreXis 보안 감사를 위해 모든 디스패치를 기록.
- **충돌 격리**: Boa 패닉이나 에이전트 버그가 해당 컨테이너에 국한됨. 다른 에이전트와 Scepter는 계속 작동.
- **경량 샌드박스를 위한 Youki**: 내부 컨테이너가 완전 Docker 컨테이너보다 더 빠르게 시작하고 더 적은 자원 소비.

### 부정적

- **PID 1로서의 COSMOS 복잡성**: COSMOS는 컨테이너의 init 프로세스로서 시그널 전달, 좀비 수거, 깨끗한 종료를 처리해야 함. 이는 일반 애플리케이션이 갖지 않는 책임을 추가.
- **컨테이너 시작 지연**: 각 에이전트 컨테이너는 시작에 ~100ms-1s 소요 (런타임에 따라 다름). 이는 프로세스 기반 또는 스레드 기반 격리보다 느림.
- **자원 오버헤드**: 각 COSMOS 컨테이너는 Boa 런타임, JS 힙, OS 오버헤드를 위해 ~50-100MB 메모리 소비. 9개의 컨테이너화된 에이전트로, 이는 ~0.5-1GB의 기준 메모리 추가.
- **테스트 복잡성**: 에이전트 동작 테스트는 COSMOS를 포함한 실제 컨테이너 실행이 필요하므로, 테스트에 Docker나 Youki가 필요. 스노우플레이크 테스트 패턴(entelecheia 이미지 구축, COSMOS 컨테이너 실행, Unix 소켓 연결)은 단위 테스트보다 복잡.
- **유지해야 할 두 런타임**: Docker/Bollard와 Youki/libcontainer 코드 경로 모두 유지 및 테스트 필요.

### 수용된 절충

**보안 및 격리 보장을 위한 자원 오버헤드.** 프로세스별 에이전트 모델은 더 적은 메모리를 사용하고 더 빠르게 시작하지만, 에이전트 간 커널 수준 격리를 제공하지 않을 것입니다. 신뢰할 수 없는 LLM 생성 코드를 에이전트가 실행하는 시스템에서, 컨테이너 격리의 보안 보장은 자원 비용의 가치가 있습니다. COSMOS를 필수 중개자로 하는 설계는 공격자가 컨테이너 내에서 코드 실행을 획득하더라도 COSMOS의 중재 외부에서 작동하여 보안 모델을 우회할 수 없도록 보장합니다.
