
# Глубокое погружение в архитектуру

> **Аудитория**: Разработчики, которым нужно понять, как shittim-chest работает внутри.
> **Последнее обновление**: 2026-05-25

## Обзор проекта

shittim-chest — это **пользовательская оболочка** для [entelecheia](https://github.com/celestia-island/entelecheia), мультиагентной платформы на Rust. Граница намеренная:

- **entelecheia** владеет оркестрацией агентов (scepter, 13 агентов, среда выполнения Cosmos/IEPL), идентификацией и правами.
- **shittim-chest** владеет аутентификацией пользователей, управлением сессиями, данными чата, конфигурацией провайдеров LLM, представлением фронтенда и прокси-мостом к scepter.

Они общаются через JWT-аутентифицированный HTTP и WebSocket. shittim-chest никогда напрямую не обращается к базе данных entelecheia для операций с агентами.

## Стек бэкенда

### Маршрутизатор Axum

Основной бэкенд (`packages/core`) — это приложение Axum 0.8. Маршрутизатор монтирует следующие группы модулей:

```text
/                   → проверка работоспособности
/api/auth/*         → AuthService (вход, регистрация, GitHub OAuth, обновление, выход)
/api/chat/*         → ChatService (диалоги, сообщения, SSE/WS поток, поиск, экспорт)
/api/providers/*    → ProviderService (CRUD провайдеров LLM, шифрование API-ключей, тестирование)
/api/generation/*   → GenerationService (генерация изображений)
/api/devices/*      → DeviceService (листинг удалённых устройств, сессии, сигнализация)
/api/webhook/*      → WebhookService (GitHub, GitLab, Gitee, свой; проверка HMAC)
/api/proxy/*        → ProxyService (обратный HTTP-прокси + мост WebSocket к scepter)
/static/*           → Статический хостинг SPA (только продакшен)
```

### SeaORM + PostgreSQL

Доступ к базе данных использует SeaORM 1.x с PostgreSQL. `shittim_chest_db` хранит:

- Аутентификацию пользователей: хеши паролей (argon2), сессии, токены обновления, API-ключи, подключения OAuth
- Данные чата: диалоги, сообщения
- Конфигурации провайдеров LLM (API-ключи зашифрованы в покое с AES-256-GCM)
- Записи удалённых устройств и сессии устройств
- Конфигурации каналов для мультиплатформенного обмена сообщениями
- Журналы доставки вебхуков

5 миграций и 25 моделей сущностей находятся в `packages/core/src/{migration,entity}/`.

### Аутентификация JWT

`shittim_chest` выпускает JWT, содержащие `{ sub: user_id, groups: [...] }`. Секрет JWT общий с scepter, так что оба сервиса могут независимо проверять токены. Токены доступа истекают через 1 час; токены обновления — через 7 дней с ротацией при каждом использовании.

## Независимая возможность LLM

shittim-chest имеет собственный слой маршрутизации LLM, работающий независимо от entelecheia:

- **LlmRouter**: Мульти-провайдерный маршрутизатор с выбором на основе приоритета и отказоустойчивостью
- **Управление провайдерами**: Конечные точки CRUD для добавления/редактирования/удаления провайдеров LLM
- **Шифрование API-ключей**: API-ключи провайдеров зашифрованы в покое с AES-256-GCM
- **OpenAI-совместимость**: Работает с любым OpenAI-совместимым API (DeepSeek, OpenAI, локальные модели и др.)
- **Двойная потоковая передача**: SSE (Server-Sent Events) и WebSocket для ответов чата

Это означает, что shittim-chest может работать как самостоятельное приложение чата без entelecheia или использовать агентов entelecheia через прокси-слой.

## Поток аутентификации

### Последовательность входа

```text
Пользователь → shittim_chest: POST /api/auth/login { username, password }
shittim_chest → shittim_chest_db: SELECT user WHERE username = ? (проверить хеш argon2)
shittim_chest → scepter: GET /api/user/{id}/permissions
scepter → entelecheia_db: запросить группы + права
scepter → shittim_chest: { groups: [...], permissions: {...} }
shittim_chest → Пользователь: { access_token, refresh_token }
shittim_chest: Сохранить сессию + кешировать RBAC
```

### GitHub OAuth

```text
Пользователь → shittim_chest: GET /api/auth/github
shittim_chest → Пользователь: 302 редирект на GitHub OAuth
Пользователь → GitHub: авторизовать
GitHub → shittim_chest: GET /api/auth/github/callback?code=...
shittim_chest → GitHub: обменять код на токен доступа
shittim_chest → GitHub: GET /user (получить информацию о пользователе)
shittim_chest → shittim_chest_db: INSERT/UPDATE oauth_connections
shittim_chest → Пользователь: { access_token, refresh_token } (автосоздать пользователя, если новый)
```

## Архитектура чата

### Поток сообщений (Автономный LLM)

```text
Пользователь → POST /api/chat/conversations/:id/messages
shittim_chest: проверить JWT, загрузить диалог
shittim_chest → LlmRouter: направить запрос к лучшему провайдеру
LlmRouter → LLM Провайдер: POST chat/completions (потоковая передача)
LLM Провайдер → LlmRouter: поток SSE
LlmRouter → Пользователь: поток SSE/WS (токены по мере поступления)
shittim_chest: сохранить сообщение в shittim_chest_db
```

### SSE против WebSocket

- **SSE** (`/api/chat/stream`): Простая HTTP-потоковая передача, работает через прокси, авто-переподключение
- **WebSocket** (`/ws/chat/stream`): Двунаправленный, поддерживает отмену и взаимодействие в реальном времени

## Архитектура прокси

Конечная точка `/api/proxy/*` пересылает аутентифицированные запросы в scepter:

1. Браузер открывает `ws://shittim-chest:80/api/proxy/chat` с JWT
1. `shittim_chest` проверяет JWT, открывает соединение с scepter, пересылая JWT
1. Двунаправленная пересылка сообщений между браузером и scepter
1. `shittim_chest` применяет ограничения скорости, логирует использование, управляет жизненным циклом соединения

## Конвейер вебхуков

Вебхуки от внешних сервисов входят через `/api/webhook/*`:

```text
GitHub/GitLab/Gitee → POST /api/webhook/{source} → проверка HMAC → Разобрать событие → Переслать в scepter через Unix-сокет
```

Поддерживаемые источники: GitHub (HMAC-SHA256), GitLab (токен), Gitee (HMAC + запасной токен), плюс общая конечная точка `/api/webhook/custom/{name}`. Функции:

- Обнаружение дублирующихся доставок (LRU-кеш, 10,000 ID)
- Журнал доставок с API листинга
- Белый список IP для источников вебхуков

## Управление удалёнными устройствами

Удалённые устройства управляются через ретрансляцию сигнализации:

```text
Браузер (webui) → WS /api/devices/stream → shittim_chest (ретрансляция сигнала) → Unix-сокет → entelecheia/polemos
```

Функции:

- Листинг устройств и CRUD сессий через REST
- Сигнализация WebRTC (SDP offer/answer, кандидаты ICE)
- Ретрансляция терминала (WebSocket к xterm.js)
- Ретрансляция кадров рабочего стола
- Бэкенд файлового браузера SFTP

shittim-chest никогда не подключается к удалённым устройствам напрямую — все данные проходят через агент polemos в entelecheia.

## Владение данными

### shittim_chest_db

| Данные | Таблица | Обоснование |
| --- | --- | --- |
| Хеши паролей (argon2) | `auth_users` | Слой представления владеет потоком входа |
| Активные сессии, токены обновления | `sessions` | Управление сессиями — это задача фронтенда |
| Зашифрованные API-ключи | `api_keys` | Выдача API-ключей обращена к пользователю |
| Подключения OAuth | `oauth_connections` | Привязка сторонней аутентификации обращена к пользователю |
| Диалоги, сообщения | `conversations`, `messages` | Данные чата обращены к пользователю |
| Конфигурации провайдеров LLM | `llm_providers` | Управление провайдерами обращено к пользователю (ключи зашифрованы) |
| Записи удалённых устройств | `remote_devices`, `device_sessions` | Отслеживание устройств обращено к пользователю |
| Конфигурации каналов | `channel_configs` и др. | Мультиплатформенная конфигурация обращена к пользователю |

### entelecheia_db

| Данные | Обоснование |
| --- | --- |
| Идентификация пользователя, группы, назначения ролей | Ядро применяет права |
| GroupPermissions (квоты провайдеров, белые списки агентов) | Политика уровня агентов живёт с агентами |
| Конфигурации агентов, состояние Cosmos/IEPL | Данные оркестрации принадлежат ядру |

## Стратегия двойного фронтенда

### Фаза 1: Vue 3 (Текущая)

| Пакет | Технология | Порт | Назначение |
| --- | --- | --- | --- |
| `webui` | Vue 3 + Vite + Pinia (TSX) | `:3000 (общий)` | Унифицированный webui: чат, генерация изображений, устройства, администрирование (провайдеры, агенты, RBAC, вебхуки) |

### Фаза 2: Rust WASM (Будущая)

| Пакет | Технология | Назначение |
| --- | --- | --- |
| `webui` | Rust → WASM (Tairitsu) | Долгосрочный унифицированный webui (чат + администрирование) |

Устаревшие фронтенды служат живыми спецификациями. Во время перехода обе версии работают параллельно, и одинаковые взаимодействия пользователя должны давать одинаковые результаты.

## Режимы развёртывания с обратным прокси

shittim-chest поддерживает три режима обратного прокси, управляемые `SHITTIM_CHEST_PROXY_MODE` в `.env`.

### Режим 1: None (Прямой)

```bash
# .env
SHITTIM_CHEST_PROXY_MODE=none   # или не установлен
```

Сервер core привязывается напрямую к `SHITTIM_CHEST_HOST:SHITTIM_CHEST_PORT` (по умолчанию `0.0.0.0:80`). Без TLS, без контейнера обратного прокси. Подходит для:

- Локальной разработки
- За существующим обратным прокси (Cloudflare Tunnel, AWS ALB, метки Traefik)
- Сетей Docker, где другой сервис обрабатывает терминацию TLS

### Режим 2: Caddy Auto

```bash
# .env
SHITTIM_CHEST_PROXY_MODE=caddy
SHITTIM_CHEST_PROXY_DOMAIN=app.example.com
```

CLI создаёт контейнер `shittim-chest-caddy` (образ `caddy:2`), который:

1. Слушает порты 80/443 (настраивается через `SHITTIM_CHEST_PROXY_HTTP_PORT` / `SHITTIM_CHEST_PROXY_HTTPS_PORT`)
1. Автоматически получает TLS-сертификаты через Let's Encrypt (встроенный ACME Caddy)
1. Проксирует все запросы к бэкенду core в сети Docker

Caddyfile не нужен — CLI генерирует его автоматически. Домен должен иметь публичный DNS, указывающий на хост.

### Режим 3: Caddy Custom

```bash
# .env
SHITTIM_CHEST_PROXY_MODE=caddy
SHITTIM_CHEST_PROXY_CONFIG_PATH=/etc/caddy/Caddyfile
SHITTIM_CHEST_PROXY_EXTRA_VOLUMES=/etc/letsencrypt:/etc/letsencrypt
```

Тот же контейнер Caddy, но вы предоставляете свой Caddyfile (смонтированный с хоста). Используйте, когда вам нужно:

- Несколько виртуальных хостов
- Пользовательские пути TLS-сертификатов
- Дополнительное промежуточное ПО (базовая аутентификация, ограничение скорости и др.)
- Обслуживание статических файлов вместе с API

### Режим 4: Nginx Custom

```bash
# .env
SHITTIM_CHEST_PROXY_MODE=nginx
SHITTIM_CHEST_PROXY_CONFIG_PATH=/etc/nginx/conf.d/default.conf
```

Создаёт контейнер `nginx:bookworm` с вашим файлом конфигурации. Вы управляете TLS-сертификатами самостоятельно. Подходит для сред, где Nginx является стандартом.

### Жизненный цикл контейнера

Все контейнеры прокси управляются CLI через Docker API (`bollard`):

| Команда | Поведение |
| --- | --- |
| `just dev` / `chest up` | Создаёт/запускает контейнер прокси, если установлен `PROXY_MODE` |
| `just dev-stop` / `chest down` | Останавливает и удаляет контейнер прокси |
| Контейнер уже запущен | Повторно использует существующий контейнер (идемпотентно) |

Контейнер прокси присоединяется к той же сети Docker, что и бэкенд core, поэтому он достигает бэкенда через внутреннее имя хоста (`core` или `shittim-chest`).
