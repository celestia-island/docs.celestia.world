
# ADR-003: PostgreSQL + PgVector для унифицированного хранения данных

**Дата**: 2026-02
**Статус**: Принято

## Контекст

Entelecheia требуется слой хранения данных, который одновременно выполняет **две различные обязанности**:

1. **Традиционные реляционные данные**: Учётные записи пользователей, учётные данные API, сессии агентов, метаданные задач, журналы аудита, политики RBAC, состояние контейнеров, события временной шкалы — все структурированные данные, которым полезны реляционная целостность, транзакции и SQL-запросы.

1. **Векторный поиск по сходству**: Эмбеддинги для RAG (Retrieval-Augmented Generation), осаждение памяти, обход графа знаний и поиск документов — данные, требующие высокоразмерного поиска ближайших соседей.

Было оценено несколько подходов к хранению:

| Подход | Реляционные данные | Векторный поиск | Двойная ответственность | Зрелость |
| --- | --- | --- | --- | --- |
| **PostgreSQL + PgVector** | Полный SQL, ACID транзакции | Индексы HNSW/IVFFlat | Единая БД, единый язык запросов | PostgreSQL: 35+ лет; PgVector: стабилен, широко развёрнут |
| **Qdrant** | Нет (только векторы) | Отлично (специализированный) | Требуется отдельная реляционная БД | Умеренная |
| **Milvus** | Нет (только векторы) | Отлично | Требуется отдельная реляционная БД | Умеренная |
| **Weaviate** | Ограниченный (встроенный CRUD) | Хорошо | Компромисс по обоим | Умеренная |
| **SQLite + векторное расширение** | Полный SQL | Экспериментальный | Один файл, ограниченный параллелизм | Низкая (векторное расширение незрелое) |
| **MongoDB + Atlas Vector Search** | Документное хранилище, без SQL | Хорошо | Компромиссная модель запросов | Высокая (но проприетарный векторный поиск) |

## Решение

Мы выбрали **PostgreSQL с расширением PgVector** в качестве унифицированного бэкенда хранения.

**Основные причины:**

1. **Стабильность и проверенная надёжность.** PostgreSQL — наиболее проверенный в бою опенсорсный движок реляционных баз данных с 35+ годами разработки, зрелым оптимизатором запросов, надёжными ACID транзакциями и обширной экосистемой инструментов. Для системы, обрабатывающей учётные данные аутентификации, политики RBAC и журналы аудита, эта стабильность не опциональна — это базовое требование. Выбор более новой, менее проверенной базы данных для этих рабочих нагрузок внёс бы ненужный операционный риск.

1. **Унифицированное двухцелевое хранение.** PgVector расширяет PostgreSQL векторным поиском по сходству (индексация HNSW и IVFFlat) без необходимости в отдельной базе данных. Это означает:

   - **Одна база данных для управления**, один пул соединений, одна стратегия резервного копирования, одна система миграций.
   - **JOIN-запросы между реляционными и векторными данными** — например, «найти документы, похожие на этот эмбеддинг, к которым у пользователя есть доступ» может быть выражено одним SQL-запросом.
   - **Транзакционная согласованность** между обновлениями метаданных и вставками эмбеддингов.
   - **SQL как универсальный язык запросов** — разработчикам не нужно изучать отдельный DSL векторных запросов.

1. **PgVector относительно стабилен для нашего масштаба.** Хотя он не так оптимизирован, как специализированные векторные базы данных для развёртываний миллиардного масштаба, PgVector компетентно обрабатывает рабочую нагрузку Entelecheia (память агентов, документы знаний, контексты RAG). Размерности эмбеддингов (768-3072) и размеры наборов данных (от тысяч до нижних миллионов векторов) находятся в зоне комфорта PgVector.

1. **Знакомство команды и экосистема.** PostgreSQL — наиболее широко развёрнутый движок баз данных в мире. Команда имеет глубокое знакомство с SQL, администрированием PostgreSQL и экосистемой Rust ORM (SeaORM, SQLx). Выбор незнакомой векторной базы данных потребовал бы значительных затрат на обучение при маргинальной выгоде.

1. **Отсутствие компромиссов по совместимости с SQL.** Многие новые векторные базы данных либо вообще не поддерживают SQL, либо поддерживают ограниченный диалект. Это вынудило бы приложение поддерживать две отдельные модели запросов — одну для реляционных данных и одну для векторного поиска — увеличивая сложность кода и поверхность для ошибок.

## Последствия

### Положительные

- **Единая операционная поверхность**: Одна база данных для мониторинга, резервного копирования, обновления и отладки. Миграции SeaORM обрабатывают как схему, так и настройку расширений.
- **Транзакционные векторные операции**: Вставки эмбеддингов и обновления метаданных происходят в одной транзакции, предотвращая осиротевшие или несогласованные данные.
- **Полная мощь SQL для комбинированных запросов**: Векторный поиск с учётом разрешений, сходство с фильтрацией по времени и соединения нескольких таблиц являются нативными операциями SQL.
- **Интеграция SeaORM + PgVector**: Экосистема Rust имеет зрелую поддержку PostgreSQL. Сущности SeaORM могут включать векторные столбцы с операторами расстояния.
- **Производственно закалённый**: Репликация PostgreSQL, восстановление на момент времени, пулинг соединений (PgBouncer) и мониторинг (`pg_stat_statements`) являются индустриальным стандартом.

### Отрицательные

- **Потолок производительности векторного поиска**: Для очень больших коллекций эмбеддингов (100M+ векторов) специализированные векторные базы данных (Qdrant, Milvus) значительно превосходят PgVector. Текущий масштаб Entelecheia не приближается к этому пределу, но это вопрос на будущее.
- **PgVector — расширение, а не ядро**: PgVector должен устанавливаться и поддерживаться отдельно от PostgreSQL. Образы контейнеров должны включать расширение (мы используем `pgvector/pgvector:pg18`). Обновление PostgreSQL может потребовать перекомпиляции расширения.
- **Ограниченные типы векторных индексов**: PgVector поддерживает HNSW и IVFFlat. Специализированные векторные базы данных предлагают более специализированные типы индексов (например, DiskANN, ScaNN), которые могут быть более эффективны для определённых распределений.
- **Конкуренция за ресурсы**: Векторное индексирование (особенно построение HNSW) потребляет CPU и память, которые разделяются с рабочими нагрузками OLTP на том же экземпляре PostgreSQL. В масштабе может потребоваться выделение векторных рабочих нагрузок на отдельную реплику.

### Принятый компромисс

**Потолок производительности ради операционной простоты.** Двухбазовая архитектура (PostgreSQL для реляционных данных + Qdrant/Milvus для векторов) обеспечила бы лучшую производительность векторного поиска в масштабе. Однако это удвоило бы операционную сложность, потребовало бы синхронизации данных между двумя системами и внесло бы проблемы согласованности. Для текущего и ближайшего масштаба Entelecheia (развёртывания от одного пользователя до небольших команд) унифицированный подход с PostgreSQL является правильным компромиссом. Если векторный поиск станет узким местом в будущем, реплика для чтения с PgVector или выделенный слой векторного кэша могут быть введены инкрементально без изменения модели запросов приложения.
