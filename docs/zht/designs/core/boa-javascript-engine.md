# ADR-002：Boa 作為嵌入式 JavaScript 引擎

**日期**：2026-02
**狀態**：已接受

## 背景

IEPL（互動式執行管線層）需要一個 JavaScript 執行時期來在每個 COSMOS 容器內執行 LLM 生成的程式碼。此執行時期必須：

1. **可嵌入** Rust 應用程式 — 它將作為輕量級容器內的初始化程序（PID 1）運行。
1. **安全且沙箱化** — LLM 生成的程式碼本質上不可信。
1. 支援 **ES 模組匯入**，用於工具派送機制（例如 `import { file_read } from 'agent'`）。
1. 具有**最小的啟動開銷** — 容器是短暫的，必須快速就緒。
1. **跨平台**且易於為容器目標編譯。

評估了多個 JavaScript/TypeScript 執行時期：

| 執行時期 | 語言 | 可嵌入 Rust | 沙箱控制 | 啟動速度 | ES 模組支援 |
| --- | --- | --- | --- | --- | --- |
| **Boa** | 純 Rust | 原生（Rust crate） | 完整（主機控制一切） | 快速 (<10ms) | 部分（對 IEPL 足夠） |
| **Deno** | Rust + V8 | 可透過 FFI | 有限（V8 isolates） | 慢 (~100ms) | 完整 |
| **QuickJS** | C | 透過 FFI/綁定 | 中等 | 快速 | 部分 |
| **V8** | C++ | 透過 FFI（v8 crate） | 有限 | 慢 | 完整 |
| **wasmoon** | C (Lua) | 透過 FFI | 良好 | 快速 | N/A (Lua) |
| **rquickjs** | C (QuickJS) | 透過 FFI | 中等 | 快速 | 部分 |

## 決策

我們選擇 **Boa Engine**（v0.21）作為嵌入式 JavaScript 執行時期。

**主要理由：**

1. **純 Rust — 零 FFI 開銷。** Boa 完全以 Rust 編寫，這意味著它直接原生編譯到 COSMOS 二進位檔中，沒有 C/C++ 依賴鏈。這消除了整類 FFI 相關的安全漏洞、建置複雜性和交叉編譯難題。在我們控制初始化程序的 COSMOS 容器環境中，這是一個關鍵優勢。

1. **「呼叫即用」簡潔性。** Boa 被設計為程式庫優先的引擎。它可以被實例化、配置主機函數，並在幾行 Rust 程式碼中執行。無需管理單獨的程序、無需維護 IPC 橋接、無需事件循環複雜性。COSMOS 中的 `JsReplHandle` 為 Boa 執行時期創建專用的 OS 執行緒，並透過標準 Rust 通道通訊 — 一個乾淨、可組合的架構。

1. **安全性是最高優先級，而非效能。** 在 COSMOS 沙箱中，每毫秒的 JS 執行時間並非關鍵路徑 — 瓶頸始終是 LLM 推理往返。重要的是執行時期讓我們對被執行的程式碼能做的事情有**完全的控制**。Boa 的主機函數註冊 API 讓我們能精確定義哪些函數可用（僅 MCP 工具派送函數），沒有逃生口。AST 安全驗證器（阻止 `eval`、`require`、`process` 等）在 Boa AST 上運行，為我們提供了原生 Rust 級別的執行保證。

1. **對 IEPL 足夠的 ES 模組支援。** IEPL 管線使用模擬的模組系統 — ES 模組匯入在命名空間建構器層級解析，而非由真實的模組載入器處理。Boa 的能力對此模式綽綽有餘。我們不需要完整的 Node.js 相容模組解析演算法。

## 後果

### 正面

- **無 C/C++ 建置依賴** — COSMOS 僅需 `cargo build` 即可乾淨編譯，無需系統層級程式庫需求。
- **完整沙箱控制** — 被執行的 JS 程式碼可用的每個函數都由主機顯式註冊。無預設 I/O、網路或檔案系統存取。
- **與 Rust 型別的緊密整合** — 自訂主機物件的 `boa_gc::Trace` trait 實作、原生 `serde_json` 互通、盡可能零拷貝。
- **崩潰安全** — Boa 恐慌被 OS 執行緒邊界捕獲，防止 COSMOS 程序因格式錯誤的 JS 程式碼而崩潰。
- **較小的二進位足跡** — 與基於 V8 的方案相比，Boa 對 COSMOS 二進位檔大小的增加顯著較小。

### 負面

- **JavaScript 一致性不完整** — Boa 未完全實作 ECMAScript 2024+。某些進階功能（例如 `WeakRef`、`FinalizationRegistry`、完整的 `Promise` 整合、所有情境中的 `async/await`）可能存在限制或缺失的實作。
- **效能無法與 V8/`SpiderMonkey` 競爭** — Boa 的解譯器比 JIT 編譯引擎慢得多。對於 CPU 密集型工作負載（大型資料處理、複雜演算法），這很重要。然而，在 IEPL 環境中，JS 程式碼主要是呼叫 MCP 工具的編排膠水，而非計算。
- **生態系統較小** — Boa 的貢獻者和實戰測試比 V8 或 QuickJS 少。上游的錯誤修復可能需要更長時間。
- **無 `eval` 或動態程式碼生成** — 依照設計（並由 AST 驗證器強制執行），動態程式碼評估被封鎖。這限制了某些元程式設計模式，但對安全模型來說是可接受的。

### 已接受的取捨

**為安全性和可嵌入性而犧牲效能。** 如果 IEPL 執行引擎需要執行複雜演算法或處理大型資料集，Boa 將是錯誤的選擇。但在 COSMOS 沙箱中，JS 程式碼是一個薄薄的編排層 — 它的工作是按正確的順序呼叫 MCP 工具、處理錯誤並組合結果。實際的繁重工作由基於 Rust 的代理工具完成。與 V8 相比，Boa 10-100 倍的較慢執行速度是無關緊要的，因為每次工具呼叫都涉及與 Scepter 的網路往返，耗時 50-500ms。
