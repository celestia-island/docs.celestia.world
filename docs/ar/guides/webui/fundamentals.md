# المفاهيم الأساسية

> **الجمهور**: المطورون الذين يريدون فهمًا مفاهيميًا لتصميم shittim-chest.
> **آخر تحديث**: 2026-05-25

## بنية المستودعين

يشكل shittim-chest و [entelecheia](https://github.com/celestia-island/entelecheia) نظام مستودعين بحد متعمد:

- **entelecheia** — نواة تنسيق الوكلاء (scepter، 13 وكيلًا، وقت تشغيل Cosmos/IEPL). يمتلك الهوية والصلاحيات وإعدادات الوكلاء.
- **shittim-chest** — الصدفة المواجهة للمستخدم. يمتلك المصادقة والجلسات وبيانات المحادثة وإعداد مزود LLM وواجهة المستخدم الأمامية وجسر الوكيل إلى scepter.

يتواصلان عبر HTTP و WebSocket موثقين بـ JWT. لا يصل أي منهما إلى قاعدة بيانات الآخر مباشرة. يسمح هذا الفصل لكل مستودع بالتطوير والنشر والتوسع بشكل مستقل.

## أوضاع التشغيل المزدوجة

يدعم shittim-chest وضعين للتشغيل:

### الوضع المستقل

يعمل بشكل مستقل مع طبقة توجيه LLM الخاصة به. يدعم:

- المحادثة باستجابات بث (SSE + WebSocket)
- توليد الصور عبر مزودين مُعدّين
- مصادقة المستخدم (كلمة مرور + GitHub OAuth)
- إدارة المزودين (إضافة/إزالة مزودي LLM)

لا يتطلب entelecheia. مفيد للتطوير والنشر البسيط.

### وضع الوكيل

يعمل كبوابة لنظام وكلاء entelecheia. يضيف:

- إعادة توجيه الطلبات إلى scepter مع تمرير JWT
- جسر WebSocket للمحادثة المبنية على الوكلاء
- دخول Webhook وإعادة توجيه المشغلات
- إدارة الأجهزة البعيدة عبر polemos
- استعلاف صلاحيات RBAC والتخزين المؤقت

يتطلب مثيل entelecheia قيد التشغيل. يمكن أن تتعايش الأوضاعان — LLM مستقل للمحادثة البسيطة، ووكيل لتنسيق الوكلاء.

## نموذج المصادقة

تستخدم المصادقة رموز JWT تصدرها `shittim_chest`:

1. **تخزين بيانات الاعتماد**: كلمات المرور (تجزئات argon2) والجلسات ورموز التحديث ومفاتيح API تقيم في `shittim_chest_db`.
1. **GitHub OAuth**: يمكن للمستخدمين تسجيل الدخول بـ GitHub؛ تُنشأ الحسابات تلقائيًا عند أول تسجيل دخول.
1. **تخزين الصلاحيات**: مجموعات المستخدمين والأدوار ومصفوفات الصلاحيات تقيم في `entelecheia_db`.
1. **تدفق JWT**: عند تسجيل الدخول، يتحقق `shittim_chest` من بيانات الاعتماد محليًا، ثم يجلب الصلاحيات من scepter. يحتوي JWT الصادر على `{ sub: user_id, groups: [...] }`.
1. **سر مشترك**: يُشارك سر توقيع JWT مع scepter حتى تتمكن كلتا الخدمتين من التحقق من الرموز بشكل مستقل.
1. **تدوير الرمز**: تنتهي رموز الوصول خلال ساعة واحدة؛ ورموز التحديث خلال 7 أيام. تُدار رموز التحديث عند كل استخدام.

## الواجهة الأمامية (webui)

webui هي الواجهة الأمامية الموحدة في `packages/webui/`، مع واجهة المحادثة على `/` ولوحة الإدارة على `/backend`، مبنية بـ Vue 3 + Vite + Pinia (TSX عبر `@vitejs/plugin-vue-jsx`).

## نظام مزود LLM

يملك shittim-chest طبقة توجيه LLM مستقلة:

- **المزودون**: نقاط نهاية LLM API قابلة للإعداد (متوافقة مع OpenAI). مخزنة في `shittim_chest_db` بمفاتيح API مشفرة بـ AES-256-GCM.
- **الموجّه**: توجيه متعدد المزودين مع اختيار قائم على الأولوية وتجاوز تلقائي للأعطال.
- **الفئات**: يمكن وسم المزودين كـ `chat`، أو `image`، أو كليهما.
- **الإدارة**: CRUD كامل عبر REST API ولوحة إدارة webui. يمكن اختبار المزودين للاتصال.
- **البث**: كلًا من بروتوكولات بث SSE (بسيط، متوافق مع الوكيل) و WebSocket (ثنائي الاتجاه).

## نظام المحادثة

- **المحادثات**: جلسات محادثة قائمة على الخيوط بعناوين وبيانات وصفية
- **الرسائل**: يدعم النص والصور واستدعاءات الأدوات (استدعاء الدوال)
- **البث**: تسليم استجابة لحظي رمزًا برمز عبر SSE أو WebSocket
- **البحث**: بحث نصي كامل للرسائل باستعلامات ILIKE
- **التصدير**: يمكن تصدير المحادثات بصيغة JSON أو Markdown
- **توليد الصور**: توليد صور قائم على الموجز عبر مزودين مُعدّين، مع وظيفة "إدراج في المحادثة"

## إدارة الأجهزة البعيدة

يوفر shittim-chest واجهة قائمة على المتصفح للأجهزة البعيدة التي يديرها entelecheia/polemos:

- **سطح المكتب**: عارض سطح مكتب بعيد مبني على WebRTC مع ترحيل الإطارات
- **الطرفية**: محاكي طرفية مبني على xterm.js مع ترحيل WebSocket
- **متصفح الملفات**: خلفية متصفح ملفات SFTP (هيكلية)
- **الإشارات**: ترحيل إشارات WebRTC مبني على WebSocket (عرض/إجابة SDP، مرشحو ICE)

يتدفق كل اتصال بالأجهزة عبر وكيل polemos في entelecheia — لا يتصل shittim-chest بالأطراف النهائية مباشرة أبدًا.

## بنية الوكيل

يعمل `shittim_chest` كبوابة بين المستخدمين و scepter:

- **وكيل HTTP عكسي**: يعيد `/api/proxy/*` توجيه الطلبات الموثقة إلى scepter مع تمرير JWT.
- **جسر WebSocket**: يستخدم بث المحادثة إعادة توجيه WebSocket ثنائي الاتجاه (`المتصفح ↔ shittim_chest ↔ scepter`).

يتيح هذا لـ `shittim_chest` فرض حدود المعدل، وتسجيل الاستخدام، وإدارة دورة حياة الاتصال دون حاجة scepter للتعامل مع اتصالات المتصفح الفردية.

## خط أنابيب Webhook

تصل الأحداث الخارجية إلى نواة الوكلاء عبر خط أنابيب webhook:

```text
GitHub/GitLab/Gitee → POST /api/webhook/{source} → تحقق HMAC → تحليل الحدث → إعادة توجيه إلى scepter عبر مقبس Unix → إرسال الوكيل
```

لكل مزود آلية تحقق خاصة:

- **GitHub**: HMAC-SHA256 عبر `X-Hub-Signature-256`
- **GitLab**: رمز عبر `X-Gitlab-Token`
- **Gitee**: HMAC مع تراجع إلى الرمز

ميزات إضافية: كشف التسليم المكرر (ذاكرة LRU المؤقتة)، تسجيل التسليم، قائمة IP البيضاء، ونقطة نهاائية webhook مخصصة عامة.

## نموذج RBAC

تتبع الصلاحيات نموذج RBAC قائمًا على المجموعات:

- **المجموعات**: ينتمي المستخدمون إلى مجموعة واحدة أو أكثر.
- **الأدوار**: للمجموعات أدوار مخصصة.
- **الصلاحيات**: يعرّف كل دور مصفوفة صلاحيات تغطي:
  - حصص المزود (أقصى رموز، أقصى طلبات)
  - قوائم الوكلاء البيضاء (أي الوكلاء يمكن للمجموعة الوصول إليها)
  - القدرات الإدارية (إدارة المستخدمين، إعداد المزودين)

يخزّن `shittim_chest` الصلاحيات مؤقتًا داخل العملية بـ TTL (افتراضي 5 دقائق). يحدث إبطال ذاكرة التخزين المؤقت عند انتهاء TTL، أو تسجيل الخروج، أو تغييرات الصلاحيات الصريحة المنتشرة من scepter.

## استراتيجية الواجهة الأمامية

يستخدم shittim-chest نهج واجهة أمامية من مرحلتين:

**المرحلة 1 (الحالية)**: واجهة Vue 3 الأمامية (`webui`، في `packages/webui/`) مبنية بـ Vite + Pinia، باستخدام TSX عبر `@vitejs/plugin-vue-jsx`. تعرّف عقد API وتعمل كتنفيذ مرجعي بجودة الإنتاج.

**المرحلة 2 (مستقبلًا)**: واجهة Rust → WASM مبنية بـ Tairitsu. تعمل الواجهة الأمامية القديمة كمواصفة حربية وعدسة اختبار — يجب أن تنتج تفاعلات المستخدم المتطابقة نتائج متطابقة.

## جسر أمان الأنواع

تُولَّد أنواع TypeScript من كود Rust عبر كريت بروتوكول `arona` الخارجي، مما يضمن اتساق الواجهة الأمامية والخلفية:

```text
arona Rust crate (git dependency)
  → #[derive(ts_rs::TS)]
  → ts-rs codegen → packages/webui/src/types/arona/ (TypeScript)
  → consumed by webui as @celestia-island/arona
```

هذا يلقي المزامنة اليدوية للأنواع. عندما يتغير نوع Rust في كريت `arona`، تُعاد توليد ارتباطات TypeScript وتستهلكها webui.
