## Предварительные требования

### Системные требования

- **Операционная система**: Linux, macOS или Windows (требуется Docker CLI)
- **ОЗУ**: минимум 8 ГБ, рекомендуется 16 ГБ
- **Хранилище**: минимум 20 ГБ свободного места
- **CPU**: рекомендуется 4 ядра и более

> Примечание (проектное намерение)
> Основное требование для Windows — доступность Docker CLI, команды можно выполнять непосредственно в PowerShell или Windows Terminal.
> Однако контейнеры в конечном итоге всё равно требуют среду выполнения Linux:
> 1. Локальное решение обычно — Docker Desktop (как правило, зависит от бэкенда WSL2).
> 2. Альтернатива — установить только Docker CLI на хосте и пересылать через `docker context` на удалённый хост Linux Docker.

### Программные требования

#### Необходимое ПО

- **Docker или Podman** (среда выполнения контейнеров)

```bash
docker --version
docker compose version
```

Используйте официально рекомендуемый способ установки для вашей платформы:

- Linux: установите Docker Engine, Docker Desktop for Linux или Podman из дистрибутива
- macOS: установите Docker Desktop или Podman Desktop
- Windows: установите Docker Desktop или Podman Desktop

**Важное примечание**:

- Зависимости времени выполнения, такие как PostgreSQL, уже включены в контейнеризированную среду
- Однако для запуска рецептов `just` или вспомогательных скриптов репозитория на хосте всё ещё требуется Python 3.8+
- Не требуется устанавливать PostgreSQL отдельно на хосте
- В Windows команды можно выполнять непосредственно в PowerShell или Windows Terminal, но развёртывание всё равно требует доступную среду выполнения Docker/Podman Linux. Локальное развёртывание обычно означает использование Docker Desktop с бэкендом WSL2; также можно пересылать через Docker CLI/context хоста на удалённый хост Linux Docker.

- **Rust 1.85+** (требуется только для сборки при разработке)

```bash
rustup update stable
```

Используйте официальный способ установки rustup для вашей платформы:

- Linux/macOS: посетите <https://rustup.rs>
- Windows: скачайте и запустите `rustup-init.exe` с <https://rustup.rs>, затем выполните `rustup update stable`

#### Рекомендуемое ПО

- **just** (средство запуска команд)

```bash
  # с помощью cargo
  cargo install just

  # с помощью brew (macOS)
  brew install just
  ```

- **VS Code** с установленным расширением rust-analyzer

## Установка

### Шаг 1: Клонирование репозитория

```bash
git clone https://github.com/celestia-island/entelecheia.git
cd entelecheia
```

### Шаг 2: Настройка переменных окружения

```bash
# Отредактируйте конфигурацию после создания .env из .env.example
nano .env  # или используйте ваш любимый редактор
```

Используйте текущую оболочку или файловый менеджер, чтобы скопировать `.env.example` в `.env`.

POSIX shell:

```bash
cp .env.example .env
```

PowerShell:

```powershell
Copy-Item .env.example .env
```

#### Базовая конфигурация

```bash
# Конфигурация базы данных (автоматически настраивается внутри контейнера)
# DATABASE_URL=postgresql://entelecheia:password@localhost:5432/entelecheia
# DATABASE_MAX_CONNECTIONS=10

# Быстрая инициализация LLM, импорт ApoRia после запуска
# Один провайдер:
# LLM_API_KEY=your-api-key-here
# LLM_BASE_URL=https://api.openai.com/v1
# LLM_MODEL=gpt-4
# Несколько провайдеров (разделитель — точка с запятой):
# LLM_API_KEY=key1;key2
# LLM_BASE_URL=https://api.one/v1;https://api.two/v1
# LLM_PROTOCOL=openai;openai,api-key
# LLM_MODEL_DEEP=model-a1,model-a2;model-b1
# LLM_MODEL_NORMAL=model-a3;model-b2
# LLM_MODEL_BASIC=model-a4;model-b3

# Быстрый вход на уровне провайдера (рекомендуется)
OPENAI_API_KEY=your-api-key-here
# ANTHROPIC_API_KEY=
# DEEPSEEK_API_KEY=
# DASHSCOPE_API_KEY=
# BIGMODEL_API_KEY=
# ZAI_API_KEY=

# Конфигурация WebSocket
WS_BIND_ADDRESS=127.0.0.1:42470
WS_MAX_CONNECTIONS=100
```

#### Примечания по настройке переменных окружения LLM

> **Важное примечание**: текущая конфигурация провайдеров LLM централизованно управляется ApoRia. Переменные окружения служат только точкой входа для начальной загрузки и больше не являются долгосрочным источником конфигурации.

**Механизм работы**:

1. Когда TUI необходимо автоматически запустить сервер, он считывает общие переменные быстрой инициализации `LLM_*` или переменные уровня провайдера, такие как `OPENAI_API_KEY`. Конфигурация нескольких провайдеров использует параллельные массивы, разделённые точкой с запятой: `LLM_API_KEY`, `LLM_BASE_URL`, `LLM_PROTOCOL`, `LLM_MODEL_DEEP`, `LLM_MODEL_NORMAL`, `LLM_MODEL_BASIC`. Переменные окружения пакетов программирования (например, `BIGMODEL_API_KEY_CODING_PRO`) также поддерживают несколько ключей, разделённых точкой с запятой, с автоматической нумерацией `(#2)`, `(#3)`. Пользовательские провайдеры отображают доменное имя в скобках.
1. Перед запуском сервера TUI предварительно записывает начальную конфигурацию провайдеров в `res/prompts/agents/aporia/config.toml`
1. После предварительной записи актуальной является конфигурация ApoRia и страница Models в TUI
1. Существующие провайдеры с непустым API-ключом не будут перезаписаны переменными окружения

**Рекомендуемое использование**:

- Используйте переменные окружения для первоначальной настройки
- В дальнейшем поддерживайте конфигурацию через страницу Models или `res/prompts/agents/aporia/config.toml`

### Конфигурация провайдеров LLM

Entelecheia (玄枢) поддерживает несколько провайдеров LLM. Настройте предпочитаемого провайдера:

```bash
OPENAI_API_KEY=sk-...
```

#### Anthropic

```bash
ANTHROPIC_API_KEY=sk-ant-...
```

#### Локальный LLM (Ollama)

```bash
# Настройте локального провайдера через страницу Models или res/prompts/agents/aporia/config.toml
# endpoint = http://localhost:11434
# model = llama2
```

```bash
# Docker socket (обычно определяется автоматически)
DOCKER_HOST=unix:///var/run/docker.sock

# Настройки контейнера
CONTAINER_NETWORK=entelecheia-network
CONTAINER_REGISTRY=127.0.0.1:5000
```

### Параметры совместимости терминала

TUI зависит от управляющих последовательностей ANSI, событий мыши и рендеринга изображений (протоколы Sixel/Kitty). В ограниченных терминальных средах — таких как SSH-сессии, последовательные консоли, CI-раннеры или старые эмуляторы терминалов — можно использовать три параметра постепенной деградации:

#### `--no-image-render`

Отключает весь рендеринг изображений. Остальные функции — цвет, мышь, дифференциальное обновление — остаются полностью работоспособными.

```bash
just tui -- --no-image-render
```

Сценарий использования: терминал поддерживает цвет и мышь, но не поддерживает протоколы изображений Sixel/Kitty (наиболее распространённый случай).

#### `--no-ansi`

Отключает захват мыши и прослушивание специальных клавиш. Цвет и дифференциальное (частичное) обновление экрана сохраняются. Полезно, когда события мыши мешают выделению текста, копированию/вставке или прокрутке истории.

```bash
just tui -- --no-ansi
```

Сценарий использования: нужен цвет, но захват мыши создаёт проблемы (мультиплексоры терминалов, `screen`, базовая конфигурация `tmux` и т.д.).

#### `--no-ansi-pure`

Режим чистого монохрома — самая агрессивная деградация. Отключает все цвета ANSI (глобально принудительно `Color::Reset`), отключает захват мыши, выполняет полную перерисовку экрана на каждом кадре. Логотип заставки заменяется на версию из чистого ASCII-арта. Этот параметр подразумевает `--no-ansi`.

```bash
just tui -- --no-ansi-pure
```

Сценарий использования: работа через SSH с минимальной поддержкой терминала, последовательные консоли, `docker exec`, CI-среды или любой терминал, который некорректно обрабатывает цветовые коды ANSI.

#### Сравнение параметров

| Функция | По умолчанию | `--no-image-render` | `--no-ansi` | `--no-ansi-pure` |
| --- | --- | --- | --- | --- |
| Цвет | Полный | Полный | Полный | Отключён |
| Захват мыши | Да | Да | Нет | Нет |
| Рендеринг изображений | Да | Нет | Нет | Нет |
| Обновление экрана | Дифф. | Дифф. | Дифф. | Полная перерисовка |
| Логотип заставки | ANSI цветной | ANSI цветной | ANSI цветной | Чистый ASCII-арт |

## Управление базой данных

### Инициализация базы данных

```bash
# Создать базу данных
just db-create

# Запустить миграции
just db-migrate

# Инициализировать с тестовыми данными
just db-init
```

### Операции с базой данных

```bash
# Проверить состояние базы данных
just db-status

# Создать резервную копию
just db-backup

# Восстановить базу данных
just db-restore backups/backup_xxx.sql

# Сбросить базу данных (предупреждение: удаляет все данные)
just db-reset
```

### Управление миграциями

```bash
# Создать новую миграцию
cargo test -p scepter test_create_migration -- --nocapture --ignored

# Откатить последнюю миграцию
just db-migrate-down
```

## Развёртывание

### Развёртывание Docker

#### Сборка образа

```bash
docker build -t entelecheia:latest .
```

#### Запуск контейнера

```bash
docker run -d --name entelecheia \
  --env-file .env \
  -p 8424:8424 \
  -v /var/run/docker.sock:/var/run/docker.sock \
  entelecheia:latest
```

### Развёртывание Docker Compose

```bash
# Запустить все службы
docker compose up -d

# Просмотреть логи
docker compose logs -f

# Остановить службы
docker compose down
```

## Устранение неполадок

### Распространённые проблемы

#### Docker: Permission Denied

```bash
# Добавить пользователя в группу docker
sudo usermod -aG docker $USER

# Выйти и войти снова
```

#### Порт уже занят

```bash
# Проверить, какой процесс занимает порт
lsof -i :8424

# Завершить процесс
kill -9 <PID>
```

#### Сбой сборки

```bash
# Очистить артефакты сборки
cargo clean

# Обновить зависимости
cargo update

# Пересобрать
just build
```

#### Контейнер не запускается

```bash
# Проверить логи Docker
docker compose logs

# Пересобрать контейнеры
docker compose down
docker compose build --no-cache
docker compose up -d
```

## Запуск Webhook-бота

Webhook-бот находится в `plugins/github-webhook/`. Для каждой платформы есть отдельный каталог.

### Предварительные условия

- Python 3.10+ (текущий бот)
- Node.js 18+ (будущая миграция на TypeScript)
- Токены ботов для каждой платформы (см. [Руководство по настройке Webhook](webhook-setup.md))

### Запуск отдельного бота

```bash
# GitHub
cd plugins/github-webhook/github
pip install -r requirements.txt
python bot.py

# Gitee
cd plugins/github-webhook/gitee
pip install -r requirements.txt
python bot.py

# Discord
cd plugins/github-webhook/discord
pip install -r requirements.txt
python bot.py
```

### Запуск всех ботов

```bash
just webhooks-up
```

### Переменные окружения

Скопируйте пример файла окружения и настройте:

```bash
cp plugins/github-webhook/.env.example plugins/github-webhook/.env
```

Подробности конфигурации для каждой платформы см. в [Руководстве по настройке Webhook](webhook-setup.md).
