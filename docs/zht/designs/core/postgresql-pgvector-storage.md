# ADR-003：以 PostgreSQL + PgVector 實作統一資料儲存

**日期**：2026-02
**狀態**：已接受

## 背景

Entelecheia 需要一個資料儲存層，**同時承擔兩種截然不同的職責**：

1. **傳統關聯式資料**：使用者帳號、API 憑證、Agent 對話、任務元資料、審計日誌、RBAC 策略、容器狀態、時間線事件——所有受益於關聯完整性、交易與 SQL 查詢的結構化資料。

1. **向量相似度搜尋**：用於 RAG（檢索增強生成）、記憶沉澱、知識圖譜遍歷與文件檢索的嵌入向量——需要高維度最近鄰搜尋的資料。

曾評估過多種儲存方案：

| 方案 | 關聯式資料 | 向量搜尋 | 雙重職責 | 成熟度 |
| --- | --- | --- | --- | --- |
| **PostgreSQL + PgVector** | 完整 SQL、ACID 交易 | HNSW/IVFFlat 索引 | 單一資料庫、單一查詢語言 | PostgreSQL：35+ 年；PgVector：穩定、廣泛部署 |
| **Qdrant** | 無（僅向量） | 優秀（專用） | 需要獨立的關聯式資料庫 | 中等 |
| **Milvus** | 無（僅向量） | 優秀 | 需要獨立的關聯式資料庫 | 中等 |
| **Weaviate** | 有限（內建 CRUD） | 良好 | 兩方面均有妥協 | 中等 |
| **SQLite + 向量擴充** | 完整 SQL | 實驗性 | 單一檔案，有限並發 | 低（向量擴充不成熟） |
| **MongoDB + Atlas Vector Search** | 文件儲存，無 SQL | 良好 | 查詢模型有妥協 | 高（但向量搜尋為專有） |

## 決策

我們選擇了**帶有 PgVector 擴充的 PostgreSQL** 作為統一的儲存後端。

**主要理由：**

1. **穩定性與經實證的可靠性。** PostgreSQL 是經過最多次實戰考驗的開源關聯式資料庫引擎，擁有 35+ 年的開發歷史、成熟的查詢最佳化器、堅如磐石的 ACID 交易，以及龐大的工具生態系統。對於一個處理驗證憑證、RBAC 策略和審計日誌的系統而言，這種穩定性不是可選項——它是基準要求。為這些工作負載選擇一個較新、未經充分驗證的資料庫，將引入不必要的運營風險。

1. **統一的雙用途儲存。** PgVector 擴充了 PostgreSQL 的向量相似度搜尋功能（HNSW 和 IVFFlat 索引），而無需單獨的資料庫。這意味著：

   - **只需管理一個資料庫**，一個連線池，一個備份策略，一個遷移系統。
   - **跨關聯式資料與向量資料的 JOIN 查詢**——例如，「找到與此嵌入向量相似且使用者有權限存取的文件」可以表示為單一 SQL 查詢。
   - **元資料更新與嵌入向量插入之間的事務一致性**。
   - **SQL 作為通用的查詢語言**——開發者無需學習獨立的向量查詢 DSL。

1. **PgVector 對我們的規模而言相對穩定。** 雖然不像專用向量資料庫那樣針對十億級部署進行最佳化，但 PgVector 能夠勝任 Entelecheia 的工作負載（Agent 記憶、知識文件、RAG 上下文）。嵌入維度（768-3072）和資料集大小（數千到低百萬個向量）完全在 PgVector 的舒適範圍內。

1. **團隊熟悉度與生態系統。** PostgreSQL 是全球部署最廣泛的資料庫引擎。團隊對 SQL、PostgreSQL 管理和 Rust ORM 生態系統（SeaORM、SQLx）有深厚的熟悉度。選擇一個不熟悉的向量資料庫需要投入大量學習成本，而邊際收益有限。

1. **SQL 相容性不打折。** 許多較新的向量資料庫要嘛完全不支援 SQL，要嘛只支援有限的方言。這將迫使應用程式維護兩個獨立的查詢模型——一個用於關聯式資料，一個用於向量搜尋——增加程式碼複雜度和錯誤產生的表面。

## 後果

### 正面

- **單一運營表面**：只需監控、備份、升級與除錯一個資料庫。SeaORM 遷移同時處理結構和擴充設定。
- **事務性向量操作**：嵌入向量插入和元資料更新在同一交易中進行，防止孤立或不一致的資料。
- **組合查詢的完整 SQL 能力**：權限感知的向量搜尋、時間篩選的相似度，以及多表 JOIN 均為原生 SQL 操作。
- **SeaORM + PgVector 整合**：Rust 生態系統具有成熟的 PostgreSQL 支援。SeaORM 實體可包含帶有距離運算子的向量欄位。
- **生產級堅固性**：PostgreSQL 的複製、時間點恢復、連線池（PgBouncer）和監控（`pg_stat_statements`）均為業界標準。

### 負面

- **向量搜尋效能上限**：對於極大的嵌入向量集合（1 億+ 向量），專用向量資料庫（Qdrant、Milvus）顯著優於 PgVector。Entelecheia 當前的規模尚未接近此限制，但這是未來的考量。
- **PgVector 是擴充，而非核心**：PgVector 必須與 PostgreSQL 分開安裝和維護。容器映像必須包含該擴充（我們使用 `pgvector/pgvector:pg18`）。升級 PostgreSQL 可能需要重新編譯擴充。
- **有限的向量索引類型**：PgVector 支援 HNSW 和 IVFFlat。專用向量資料庫提供更多專用索引類型（如 DiskANN、ScaNN），可能對特定分佈更有效率。
- **資源競爭**：向量索引建構（尤其是 HNSW）消耗 CPU 和記憶體，這些資源與同一 PostgreSQL 實例上的 OLTP 工作負載共享。在大規模下，將向量工作負載分離到專用副本可能變得必要。

### 接受的權衡

**效能上限換取運營簡單性。** 雙資料庫架構（PostgreSQL 用於關聯式 + Qdrant/Milvus 用於向量）在大規模下提供更好的向量搜尋效能。然而，它會使運營複雜度翻倍，要求兩個系統之間的資料同步，並引入一致性挑戰。對於 Entelecheia 當前和近期的規模（單使用者到小型團隊部署），統一的 PostgreSQL 方案是正確的權衡。如果未來向量搜尋成為瓶頸，可以逐步引入帶有 PgVector 的唯讀副本或專用向量快取層，而不改變應用程式的查詢模型。
