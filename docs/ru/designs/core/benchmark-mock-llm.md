# Проектирование сервера Benchmark и Mock LLM

> Ключевой тезис: **Сколько задач способна выполнить любая модель + система инструментов Entelecheia?**
>
> Классификация «слабый/сильный» не задаётся заранее. Для какого провайдера передан ключ через переменную окружения — для модели этого провайдера создаётся конфигурация кодирующего пакета (coding plan) и проводится тестирование. Если нет ни одного ключа → немедленный выход с ошибкой.

## 0. Ключевое наблюдение: повторное использование существующего реестра провайдеров

В Entelecheia **уже есть** полноценная система обнаружения провайдеров, управляемая переменными окружения:

- репозиторий `provider-registry`: 926 TOML-файлов, покрывающих всех основных провайдеров — OpenAI / Anthropic / DeepSeek / GLM / Qwen / Kimi / MiniMax и др.
- `derive_config_from_env()`: перебирает все TOML точек входа (entrypoint); любой провайдер, для которого установлена переменная `env_var`, активируется автоматически
- `ModelTier` (Deep / Normal / Basic) — три уровня + цепочка fallback
- `_shared_llm_provider::ProviderRegistry`: глобальная регистрация, 5 протоколов (OpenAI Chat / Responses / Anthropic v1/v2 / Gemini)

**Создавать новый реестр провайдеров не нужно** — benchmark runner напрямую переиспользует `_shared_config` + `_shared_llm_provider`.

## 1. Концепция кодирующего пакета (Coding Plan)

Для каждой доступной модели автоматически формируется benchmark profile:

| Поле | Источник | Описание |
|------|------|------|
| provider_id | TOML точки входа | напр. `deepseek` / `zhipu_glm` |
| model_id | значения по умолчанию точки входа | напр. `deepseek-coder` / `glm-4-plus` |
| tier | ModelTier::Deep | для задач кодирования используется уровень Deep |
| base_url | api.base_url точки входа | реальный удалённый API |
| api_key | переменная окружения (api.env_var точки входа) | считывается во время выполнения |
| protocol | api.protocol точки входа | значение перечисления GenProtocol |
| context_window | карточка модели | считывается из `models/<provider>/<model>.toml` |
| max_output_tokens | карточка модели | то же, что выше |
| supports_function_calling | карточка модели | определяет способ вызова инструментов |

Profile формируется во время выполнения из переменных окружения динамически; заранее заготовленный конфигурационный файл не используется.

## 3. Mock LLM сервер

### 3.1 Протокол Record/Replay

Mock-сервер совместим с OpenAI Chat Completions API (`/v1/chat/completions`) и поддерживает два режима работы:

**Режим записи (при первом запуске на реальной модели)**:
```text
Client → Mock Server → Real API → Mock Server (запись ответа) → Client
```

**Режим воспроизведения (для CI/офлайна)**:
```text
Client → Mock Server (сопоставление запроса → возврат записанного ответа) → Client
```

### 3.2 Стратегия сопоставления запросов

Запрос сопоставляется по хешу следующих полей:
- `model` (имя модели)
- хеш содержимого `messages` (после нормализации)
- структурный хеш `tools` (при наличии)
- параметры `temperature`, `max_tokens` и др.

**Строгий режим (Strict, включён по умолчанию в CI)**: любой запрос без совпадения немедленно вызывает ошибку, без fallback на реальный API. Гарантирует полноту фикстур.

**Мягкий режим (Lenient, для разработки)**: при отсутствии совпадения выполняется fallback на реальный API с записью ответа.

### 3.3 Хранилище фикстур

```text
tests/fixtures/llm/
├── swe-bench-verified/
│   ├── gpt-4o/
│   │   ├── <request_hash>.json      # записанный ответ
│   │   └── index.toml                # индекс сводки запросов
│   ├── claude-sonnet/
│   └── llama-8b/
└── aider-polyglot/
    └── ...
```

### 3.4 Выбор реализации

| Вариант | Плюсы | Минусы |
|------|------|------|
| **AIMock** (CopilotKit) | зрелый, поддерживает streaming/tool-calls/MCP, Docker-образ | внешняя зависимость |
| **Собственный простой сервер** | полный контроль, ноль внешних зависимостей | нужно самому обрабатывать streaming и краевые случаи |
| **VCR.py / wiremock** | зрелая языковая экосистема | не специализирован для LLM, требуется адаптация |

**Рекомендация**: сначала использовать собственный простой сервер (один маршрут на axum/actix — сопоставление + возврат JSON), а при необходимости в streaming — мигрировать на AIMock.

## 4. Адаптер SWE-bench

### 4.1 Процедура выполнения задач

```text
for instance in dataset:
    1. Загрузить Docker-образ SWE-bench (три слоя: base + env + instance)
    2. Запустить контейнер, смонтировать репозиторий кода
    3. Внедрить текст issue как описание задачи
    4. Запустить Entelecheia Agent Runtime (подключение к bash/файловой системе контейнера)
    5. Agent выполняется до завершения или таймаута (step cap: 50, wall-clock: 15 мин)
    6. Выполнить git diff внутри контейнера → извлечь patch
    7. Вывести JSONL: {instance_id, model_name_or_path, model_patch}
    8. Уничтожить контейнер
```

### 4.2 Инъекция Agent Runtime

При работе Entelecheia внутри контейнера SWE-bench требуется:
- инструменты файловых операций → отображение на файловую систему контейнера
- инструменты выполнения команд → отображение на bash контейнера
- инструменты поиска → `rg`/`grep` (предустановлены в контейнере)
- **отключить** агентов, не относящихся к данной задаче (PoleMos/промышленные протоколы и т. п.), чтобы сократить шум в контексте

### 4.3 Оценка

Непосредственно используется нативный harness SWE-bench:
```bash
python -m swebench.harness.run_evaluation \
    --dataset_name princeton-nlp/SWE-bench_Verified \
    --predictions_path entelecheia_predictions.jsonl \
    --max_workers 8 --run_id entelecheia-eval
```

Результат: для каждого экземпляра — resolved/unresolved; агрегируется resolution rate.

## 5. Матрица экспериментов

### 5.1 Ключевое сравнение

Фиксированный набор данных (SWE-bench Verified), варьируются два измерения:

|  | Baseline (mini-SWE-agent) | Entelecheia |
|--|---------------------------|-------------|
| **GPT-4o** | A₁ | B₁ |
| **Claude Sonnet** | A₂ | B₂ |
| **Llama 3.1 8B** | A₃ | B₃ |
| **Qwen 2.5 7B** | A₄ | B₄ |

- Bᵢ/Aᵢ = коэффициент усиления (amplification factor) для модели i
- Сравнение по строкам: выше ли AF у слабых моделей (Llama/Qwen), чем у сильных (GPT-4o)?

### 5.2 Абляционные эксперименты

| Конфигурация | Цель |
|------|------|
| Полная Entelecheia (12 агентов + все навыки) | мощный базовый уровень |
| Только HubRis + KaLos + SkeMma (планирование + файлы + исполнение) | измерить прирост от многоагентной оркестрации |
| Только KaLos + bash (один агент + файловые инструменты) | близко к baseline, измерить прирост цепочки навыков |
| Без soul-идентичности (без идентификационных/метафорических промптов) | измерить эффект persona-промптов |

## 7. Точки интеграции с существующей архитектурой Entelecheia

| Компонент | Способ интеграции |
|------|---------|
| `ApoRia::llm_chat` | переключение base_url на mock или реальный API |
| Оркестрация `SkoPeo` | новый режим выполнения `benchmark`, пропускающий интерактивные подтверждения |
| Планирование `HubRis` | принимает описание benchmark-задачи как входные данные |
| Контейнер `NeiKos` | управление жизненным циклом Docker-контейнеров SWE-bench |
| Файлы `KaLos` | отображение на файловую систему контейнера |
| Безопасность `OreXis` | в режиме benchmark политики безопасности ослабляются (разрешено произвольное выполнение кода) |

## 8. Замечания

- **Контроль затрат**: SWE-bench Verified полностью — 500 задач × 4 модели × 2 конфигурации = 4000 запусков. При среднем 50 шагов/задачу и ~2K токенов на шаг — порядка 400M токенов. Необходимо задать лимит `--max_cost`.
- **Ресурсы контейнеров**: каждый экземпляр SWE-bench требует отдельный Docker-контейнер; рекомендуется ≥120 ГБ диска, ≥32 ГБ RAM.
- **Детерминизм**: режим Mock гарантирует воспроизводимость в CI; в режиме Real фиксируется `temperature=0` + записывается хеш запроса для обнаружения дрейфа.
- **Обнаружение загрязнения**: в SWE-bench существует проблема утечки данных в память (arXiv:2506.12286); рекомендуется оставлять часть самостоятельно синтезированных задач как holdout.
