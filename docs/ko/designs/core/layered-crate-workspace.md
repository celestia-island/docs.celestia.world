
# ADR-004: 60개 이상 크레이트 계층형 워크스페이스 아키텍처

**일자**: 2026-03
**상태**: 승인됨

## 배경

Entelecheia는 단일 `packages/shared` 크레이트(38K 라인, 187개 `.rs` 파일)로 시작했으며, 이 크레이트는 모든 공유 인프라(타입, MCP 프로토콜, LLM 프로바이더, 컨테이너 관리, 데이터베이스, 보안, 설정 등)를 포함했습니다. 프로젝트가 12개 에이전트 + 1개 도메인 에이전트 + 3개 바이너리 패키지로 성장하면서 다음과 같은 문제들이 발생했습니다:

1. **컴파일 시간**: `shared`의 어떤 변경이든 하나의 구조체만 수정되더라도 187개 파일 전체를 재컴파일해야 했습니다.
1. **의존성 오염**: MCP 타입만 필요한 에이전트 크레이트가 데이터베이스 드라이버, 컨테이너 런타임, LLM 프로바이더까지 추이적 의존성으로 강제로 끌어오게 되었습니다.
1. **불분명한 소유권**: 하나의 크레이트에 187개 파일이 몰려 있어, 어떤 모듈이 어떤 기능을 "소유"하는지 불분명하여 리팩터링이 위험했습니다.
1. **기능 플래그 폭발**: Cargo 기능을 통한 조건부 컴파일로 불필요한 의존성을 피하려 했으나, 이로 인해 테스트 구성의 조합 폭발이 발생했습니다.

## 결정

단일 `packages/shared`를 **37개의 집중된 하위 크레이트**로 분해하고, 엄격한 의존성 방향을 따라 **6개의 의존성 계층**(L0~L5)으로 구성합니다:

```text
L0 (리프) → L1 → L2 → L3 → L4 → L5 → 소비자 (scepter, agents, tui)
```

**계층 정의:**

| 계층 | 크레이트 | 규칙 |
| --- | --- | --- |
| **L0** | core, logging, macros | 다른 entelecheia 크레이트에 대한 의존성 없음 |
| **L1** | domain_enums, mcp_types, text, concurrent | L0에만 의존 |
| **L2** | config, agent_registry, state_types | L0-L1에 의존 |
| **L3** | domain_agent, container, agent_lifecycle, agent_runtime, thread_types, toolchain, infra_utils | L0-L2에 의존 |
| **L4** | state_sync, domain_skills, hooks, domain_auth, container_runtime, skills_permissions, timeline, iepl | L0-L3에 의존 |
| **L5** | llm_provider, prompt, custom_agent, storage, infra_jsonrpc, infra_services, e2e_events, adapter, plugin_host, rag, embedding, security_policy | L0-L4에 의존 |

모든 내부 의존성 선언은 버전 일관성을 위해 `workspace = true`를 사용합니다. 얇은 어그리게이터 크레이트는 존재하지 않으며, 소비자는 개별 하위 크레이트에서 직접 임포트합니다.

## 결과

### 긍정적

- **증분 컴파일**: `shared-core`(L0) 변경은 여전히 전파되지만, `shared-security-policy`(L5) 변경은 해당 크레이트와 직접 소비자만 재컴파일합니다. 빌드 시간이 크게 개선되었습니다.
- **명확한 소유권 경계**: 각 크레이트는 집중된 책임을 가집니다. 코드 리뷰 범위는 크레이트 경계에 의해 자연스럽게 제한됩니다.
- **의존성 격리**: 에이전트 크레이트는 필요한 공유 크레이트만 임포트합니다. SkeMma는 데이터베이스 드라이버를 끌어오지 않습니다. EleOs는 컨테이너 런타임을 끌어오지 않습니다.
- **순환 의존성 방지**: 계층형 아키텍처는 구조적으로 순환 의존성 생성을 불가능하게 만듭니다 — L3 크레이트는 L5 크레이트에 의존할 수 없습니다.
- **격리 테스트 가능**: 각 크레이트의 테스트는 전체 워크스페이스의 의존성 트리 없이 독립적으로 실행됩니다.

### 부정적

- **워크스페이스 관리 오버헤드**: 단일 워크스페이스에 60개 이상의 크레이트가 있다는 것은 더 많은 `Cargo.toml` 파일을 유지 관리하고, 버전 업 시 더 많은 `[dependencies]` 섹션을 갱신하며, 더 신중한 의존성 선언이 필요함을 의미합니다.
- **크로스 크레이트 리팩터링 난이도 증가**: 타입을 L2에서 L3로 이동하려면 모든 L2 소비자를 갱신하고, L3+ 크레이트가 실수로 이전 위치를 통해 이동된 타입에 의존하지 않는지 확인해야 합니다.
- **크레이트 네이밍 장황함**: 내부 크레이트 이름은 `_shared_*` 접두사 규칙(예: `_shared_domain_skills_permissions`)을 사용하는데, 이는 장황하지만 워크스페이스 명확성을 위해 필요합니다.
- **과도한 분해 가능성**: 일부 크레이트(예: ~200라인의 `shared-text`)는 자체 크레이트 오버헤드를 정당화하지 못할 수 있습니다. 분해는 엄격한 필요성보다는 "성장 가능성이 있으면 분리한다"는 철학을 따랐습니다.

### 수용된 트레이드오프

**컴파일 시간 및 아키텍처 명확성을 위한 관리 복잡성.** `shared`를 37개 크레이트로 분해하는 것은 Rust 워크스페이스 설계 중에서도 과감한 편에 속합니다. 중간 지점(10~15개 크레이트)이 관리 측면에서 더 단순했을 것입니다. 그러나 프로젝트의 광범위한 표면(26개 LLM 프로바이더, 2개 컨테이너 런타임, 12개 에이전트, 전체 보안 파이프라인, 데이터베이스, IEPL)을 고려할 때, 세분화된 분해는 각 조각이 독립적으로 진화할 수 있도록 보장합니다. `workspace = true` 패턴은 버전 관리 오버헤드를 완화합니다.
