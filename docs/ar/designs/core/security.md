# بنية أمان Entelecheia

> نموذج دفاع متعمق شامل لمنصة تنسيق الوكلاء المتعددين Entelecheia.

## نظرة عامة

تطبق Entelecheia **بنية أمان دفاعية متعمقة** تمتد عبر 14 طبقة أمان قابلة للاختبار بشكل مستقل — من عزل الحاويات على مستوى الأجهزة إلى بوابات صلاحيات الأدوات المواجهة لـ LLM. على عكس أطر الوكلاء التقليدية التي تعرض جميع الأدوات مباشرة إلى LLM، يعني تصميم **النواة الدقيقة للتنفيذ فقط** الخاص بـ Entelecheia أن LLM يرى فقط 3 أدوات بدائية (`exec`، `write_to_var`، `write_to_var_json`)، بينما يتم توزيع 148 أداة MCP عبر خط أنابيب IEPL محدد النوع مع تخويل متعدد الطبقات.

## الطبقة 1: النواة الدقيقة للتنفيذ فقط

**الصناديق:** `scepter`, `mcp_types`
**فلسفة التصميم:** تقليل سطح هجوم LLM

يعمل LLM في **صندوق رمل تنفيذي فقط** حيث يمكنه استدعاء ثلاث عمليات بدائية فقط:

| الأداة | الغرض | المعاملات |
| --- | --- | --- |
| `exec` | تنفيذ سلسلة سكربت | كود JavaScript (محوّل من TypeScript عبر IEPL) |
| `write_to_var` | تخزين قيمة نصية | اسم المتغير + القيمة |
| `write_to_var_json` | تخزين قيمة JSON | اسم المتغير + قيمة JSON |

كل أدوات MCP الـ 148 (عمليات الملفات، إدارة الحاويات، التحكم في الأجهزة، بحث الويب، إلخ) **غير مرئية لـ LLM**. يتم استدعاؤها بشكل غير مباشر عبر خط أنابيب IEPL عندما تستدعي `exec` الخاصة بـ LLM استيراد وحدات ES (مثلًا، `import { file_read } from 'kalos'`).

**نموذج التهديد:** حتى لو تم اختراق LLM عبر حقن التوجيهات، لا يمكنه استدعاء أدوات خطرة مباشرة مثل `container_destroy` أو `ssh_exec`. يفرض خط أنابيب IEPL فحص النوع والتحقق من الصلاحيات قبل تنفيذ أي أداة.

**التنفيذ:** يعرف `packages/shared/mcp_types/src/` أنواع IPC للنواة الدقيقة. يعمل معالج `exec` في `packages/cosmos/` على تحويل وتنفيذ السكربت عبر محرك Boa، مع توجيه استدعاءات الأدوات عبر `McpRouter` الخاص بـ `skemma`.

## الطبقة 2: بوابة صلاحيات التخويل المزدوج

**الصندوق:** `security_policy` (5,772 سطرًا)

تصرح كل أداة MCP بمتطلبات الوصول الخاصة بها عبر معدّد **مستوى الصلاحية**. تصرح كل مهارة (سكربت IEPL) بمستوى الصلاحية الذي تحتاجه لكل أداة. يجب أن يتفق كلاهما لكي يُسمح بالاستدعاء بالمتابعة.

```rust
pub enum PermissionLevel {
    /// Read-only operations (file_read, list_dir, etc.)
    Read,
    /// Write operations within the workspace (file_write, exec_script)
    Write,
    /// Operations affecting external systems (ssh_exec, container_deploy)
    System,
    /// Operations with irreversible consequences (container_destroy, device_reboot)
    Destructive,
}
```

**تدفق التخويل:**

1. تصرح المهارة: "أحتاج وصول `System` إلى `ssh_exec`"
1. تصرح الأداة: "أطلب صلاحية `System`"
1. تفحص بوابة الصلاحية: `skill_level >= tool_requirement` و`المهارة ممنوحة صراحةً هذه الأداة`
1. إذا فشل أي فحص: يُحظر الاستدعاء، يُسجل، ويُبلّغ إلى حارس OreXis

**التنفيذ:** `packages/shared/security_policy/src/` — 107 تعليق اختبار، 4 اختبارات tokio.

## الطبقة 3: تخويل المهارة بمستوى الثقة

**الصندوق:** `domain_skills_permissions` (1,776 سطرًا)

تُصنّف المهارات إلى **مستويات ثقة** تحدد نطاق صلاحياتها الافتراضي:

| مستوى الثقة | الوصف | الصلاحيات الافتراضية |
| --- | --- | --- |
| `Builtin` | يأتي مع المنصة | وصول كامل للأدوات |
| `Verified` | مراجع وموقّع من المشرفين | قراءة + كتابة |
| `Community` | مُقدّم من المستخدمين | قراءة فقط |
| `Untrusted` | محمّل ديناميكيًا | لا وصول للأدوات (exec فقط) |

يتم التحقق من مستوى ثقة كل مهارة وقت التحميل وتخزينه مؤقتًا. تسجل محاولات تصعيد مستوى الثقة كأحداث أمنية.

## الطبقة 4: عزل الحاويات (الحلقة الخارجية)

**الصندوق:** `container` (5,742 سطرًا)

يحدث كل تنفيذ وكيل داخل **حاوية Docker أو Podman** مع:

- عزل نطاق الشبكة
- نظام ملفات جذر للقراءة فقط (باستثناء تركيب مساحة العمل)
- ملف seccomp يقيد استدعاءات النظام
- حدود الموارد (CPU، ذاكرة، عدد PID)
- لا وصول إلى مقبس Docker الخاص بالمضيف

**التنفيذ:** `packages/shared/container/src/` — 74 تعليق اختبار، 12 اختبار tokio. يدعم كلًا من Docker (عبر Bollard API) و Podman.

---

## الطبقة 5: صندوق رمل OCI (الحلقة الداخلية)

**الصندوق:** `container_runtime` (3,645 سطرًا)

داخل حاوية Docker، تشغل Entelecheia **طبقة عزل ثانية** باستخدام Youki/libcontainer — بيئة تشغيل حاويات متوافقة مع OCI بدون خادم وبدون جذر. هذا يوفر:

- تنفيذ بدون جذر (لا تصعيد امتيازات ممكن)
- عزل النطاقات مستقل عن Docker
- فرض cgroup v2
- فلتر seccomp (رفض افتراضي)

**لماذا طبقتان؟** يوفر Docker عزلًا خشن الحبيبات (الشبكة، نظام الملفات). يوفر Youki تصفية دقيقة لاستدعاءات النظام وحساب الموارد. إذا تم اختراق Docker، لا يزال صندوق رمل Youki يحتوي على الوكيل.

## الطبقة 6: تحكم الوصول RBAC

**الصناديق:** `domain_auth` (380 سطرًا)، shittim-chest `rbac` (1,736 سطرًا)

تحكم الوصول القائم على الأدوار يحكم جميع عمليات API:

- **المجموعات:** ينتمي المستخدمون إلى مجموعات؛ المجموعات لديها منح
- **المنح:** صلاحيات دقيقة (قراءة/كتابة/إدارة لكل نوع مورد)
- **عزل مساحة العمل:** يمكن للمستخدمين الوصول فقط إلى مساحات العمل التي هم أعضاء فيها
- **عمليات مساحة العمل المتقاطعة:** تتطلب منح صريحة من المسؤول

---

## الطبقة 7: مصادقة JWT

**الوحدة:** shittim-chest `auth/jwt.rs` (264 سطرًا)

- **الخوارزمية:** HS256 (HMAC-SHA256)
- **رموز الوصول:** قصيرة العمر (قابلة للتهيئة، افتراضي 15 دقيقة)
- **رموز التحديث:** أطول عمرًا مع تدوير عند الاستخدام
- **حماية CSRF قائمة على Nonce** لعملاء المتصفح
- **تحديد المعدل** على نقاط نهاية المصادقة (خوارزمية GCRA)

---

## الطبقة 8: تشفير مفاتيح API

**الصندوق:** `aporia` (5,802 سطرًا)

تُشفّر كل مفاتيح API لمزودي LLM أثناء التخزين باستخدام **AES-256-GCM** مع:

- nonce فريد لكل عملية تشفير
- مفتاح مشتق من سر رئيسي (مُهيّأ عبر البيئة)
- تصفير المفاتيح النصية من الذاكرة بعد الاستخدام
- دعم تدوير المفاتيح

---

## الطبقة 9: حارس الأمان (OreXis)

**الصندوق:** `orexis` (5,239 سطرًا) — وكيل "الجهاز المناعي"

OreXis هو وكيل Layer-1 يقوم بـ:

- **تدقيق الكود** بحثًا عن الثغرات الأمنية والامتثال للتراخيص
- **فحص استدعاءات الأدوات** مقابل سياسات الأمان المسجلة
- **حظر/إلغاء حظر** أدوات أي وكيل بالنمط
- **مراقبة** سلوك الوكيل بحثًا عن أنماط شاذة

أدوات MCP (24): `standard_check`، `compliance_report`، `audit_alignment`، `audit_legality`، `agent_integrity`، `security_audit`، `tool_block`، `tool_unblock`، `policy_register`، `policy_list`، إلخ.

## الطبقة 10: خط أنابيب آمن النوع IEPL

**الصناديق:** `iepl` (2,670 سطرًا)، `iepl_engine` (1,228 سطرًا)، `skemma` (7,960 سطرًا)

يضمن خط أنابيب **لغة إضافات Entelecheia** (IEPL) أمان النوع بين الكود المُولّد من LLM وتوزيع الأدوات الأصلية:

1. يُولّد LLM كود TypeScript باستخدام استيراد وحدات ES
1. **SWC** يحوّل TypeScript ← JavaScript (تحقق بناء الجملة)
1. **محرك Boa** ينفذ JavaScript في سياق معزول
1. يتم حل استيراد وحدات ES إلى استدعاءات `__native_dispatch`
1. يتم توجيه كل توزيع عبر `McpRouter` مع فحص كامل للنوع

**التهديد المخفّف:** هجمات الحقن عبر استدعاءات أدوات غير محددة النوع (شائعة في أطر الوكلاء القائمة على Python حيث يتم التحقق من مخططات الأدوات فقط وقت التشغيل).

## الطبقة 11: القائمة البيضاء لسجل المزودين

**الملف:** `configs/registries.toml` (337 سطرًا)

تحتفظ Entelecheia **بقائمة بيضاء مشفّرة** لسجلات الحزم الموثوقة عبر 15 منظومة:

crates.io، PyPI، npm، وحدات Go، Docker Hub، Maven Central، NuGet، RubyGems، Hackage، Alpine APK، Debian APT، GitHub، GitLab، `HuggingFace`، PyTorch.

يتم **حظر** أي استيراد حزمة من سجل غير مدرج في القائمة البيضاء **على مستوى الحاوية** قبل التنفيذ.

---

## الطبقة 12: الدفاع ضد حقن التوجيهات

**الآلية:** حدود صندوق رمل IEPL

يتم تنفيذ مخرجات `exec` الخاصة بـ LLM في **سياق Boa JS معزول** بدون وصول إلى:

- نظام ملفات المضيف
- مقابس الشبكة
- متغيرات البيئة
- حالة الوكلاء الآخرين

تكون مخرجات الأدوات المُعادة إلى LLM **معقّمة** — يتم ترميز البيانات الثنائية بـ base64، يتم اقتطاع المخرجات المفرطة، ويتم وضع علامة على أنماط حقن التوجيهات المحتملة في نتائج الأدوات من قبل OreXis.

---

## الطبقة 13: تحديد المعدل

**الوحدة:** shittim-chest `channel/rate_limit.rs` (118 سطرًا)

تحديد المعدل لكل مستخدم وكل قناة باستخدام **GCRA (خوارزمية معدل الخلية العامة)**:

- حجم الدفعة والمعدل المستدام قابلان للتهيئة
- DashMap لكل مستخدم للبحث O(1)
- تراجع تلقائي عند تجاوز الحد
- حدود منفصلة لاستدعاءات API وإرسال الرسائل واستدعاءات الأدوات

---

## الطبقة 14: مسار التدقيق

**الصناديق:** `orexis`, `timeline` (3,096 سطرًا)

كل استدعاء أداة، قرار وكيل، وحدث أمني يتم:

1. تسجيله في **الجدول الزمني** بسياق كامل (شارة الوكيل، اسم المهارة، المعاملات، النتيجة)
1. ربطه بالتجزئة بالأحداث السابقة لكشف العبث
1. حفظه في PostgreSQL مع احتفاظ قابل للتهيئة
1. قابل للاستعلام عبر CLI (`entelecheia-cli trace-chain <badge>`)

## نموذج التهديد

### خارج النطاق

- الوصول الفيزيائي إلى آلات المضيف
- اختراق خادم Docker/Podman (يُفترض موثوقًا)
- استغلالات النواة (مخفّفة لكن غير ممنوعة بعزل مساحة المستخدم)
- هجمات سلسلة التوريد على تبعيات صندوق Rust (مخفّفة جزئيًا بـ `cargo-deny`)

### المخاطر المقبولة

- ثغرات محرك Boa JS (معزولة داخل الحاوية)
- انقطاعات مزود LLM (لا مسار تنفيذ احتياطي)
- تلف بيانات PostgreSQL (مخفّفة بالنسخ الاحتياطية، غير ممنوعة)
