# ADR-002: 임베디드 JavaScript 엔진으로서의 Boa

**일자**: 2026-02
**상태**: 승인됨

## 배경

IEPL(Interactive Execution Pipeline Layer)은 각 COSMOS 컨테이너 내에서 LLM 생성 코드를 실행하기 위해 JavaScript 런타임이 필요합니다. 이 런타임은 다음을 충족해야 합니다:

1. Rust 애플리케이션에 **임베디드 가능**해야 함 — 경량 컨테이너 내에서 init 프로세스(PID 1)로 실행됩니다.
1. **안전하고 샌드박스**되어야 함 — LLM 생성 코드는 본질적으로 신뢰할 수 없습니다.
1. 도구 디스패치 메커니즘을 위한 **ES 모듈 임포트** 지원 (예: `import { file_read } from 'agent'`).
1. **최소한의 시작 오버헤드** — 컨테이너는 일시적이므로 빠르게 준비되어야 합니다.
1. 컨테이너 대상으로 **크로스 플랫폼**이며 쉽게 컴파일 가능해야 합니다.

여러 JavaScript/TypeScript 런타임이 평가되었습니다:

| 런타임 | 언어 | Rust 임베딩 | 샌드박스 제어 | 시작 속도 | ES 모듈 지원 |
| --- | --- | --- | --- | --- | --- |
| **Boa** | 순수 Rust | 네이티브 (Rust crate) | 완전 (호스트가 모든 것을 제어) | 빠름 (<10ms) | 부분적 (IEPL에 충분) |
| **Deno** | Rust + V8 | FFI로 가능 | 제한적 (V8 isolates) | 느림 (~100ms) | 완전 |
| **QuickJS** | C | FFI/바인딩으로 | 보통 | 빠름 | 부분적 |
| **V8** | C++ | FFI로 (v8 crate) | 제한적 | 느림 | 완전 |
| **wasmoon** | C (Lua) | FFI로 | 양호 | 빠름 | 해당 없음 (Lua) |
| **rquickjs** | C (QuickJS) | FFI로 | 보통 | 빠름 | 부분적 |

## 결정

**Boa Engine**(v0.21)을 임베디드 JavaScript 런타임으로 선택했습니다.

**주요 이유:**

1. **순수 Rust — FFI 오버헤드 제로.** Boa는 전체가 Rust로 작성되어, COSMOS 바이너리에 C/C++ 의존성 체인 없이 네이티브로 컴파일됩니다. 이는 FFI 관련 보안 취약점, 빌드 복잡성, 크로스 컴파일 문제 전체 클래스를 제거합니다. COSMOS 컨테이너 환경에서 init 프로세스를 제어하는 우리에게 이는 결정적인 장점입니다.

1. **"호출 및 사용"의 단순성.** Boa는 라이브러리 우선 엔진으로 설계되었습니다. 몇 줄의 Rust 코드로 인스턴스화, 호스트 함수 구성, 실행이 가능합니다. 별도의 프로세스 관리, IPC 브리지 유지, 이벤트 루프 복잡성이 없습니다. COSMOS의 `JsReplHandle`은 Boa 런타임을 위한 전용 OS 스레드를 생성하고 표준 Rust 채널을 통해 통신합니다 — 깔끔하고 구성 가능한 아키텍처입니다.

1. **보안이 최우선이며, 성능이 아닙니다.** COSMOS 샌드박스에서 JS 실행 시간의 모든 밀리초는 중요 경로상에 있지 않습니다 — 병목 현상은 항상 LLM 추론 왕복입니다. 중요한 것은 런타임이 실행 코드가 할 수 있는 것에 대한 **완전한 제어**를 제공한다는 것입니다. Boa의 호스트 함수 등록 API는 어떤 함수가 사용 가능한지(MCP 도구 디스패치 함수만) 정밀하게 정의할 수 있으며, 탈출구가 없습니다. AST 보안 검증기(`eval`, `require`, `process` 등 차단)는 Boa AST를 기반으로 작동하여, Rust 네이티브 수준의 강제 보장을 제공합니다.

1. **IEPL에 충분한 ES 모듈 지원.** IEPL 파이프라인은 시뮬레이션된 모듈 시스템을 사용합니다 — ES 모듈 임포트는 실제 모듈 로더가 아닌 네임스페이스 빌더 수준에서 해석됩니다. Boa의 기능은 이 패턴에 충분합니다. 완전한 Node.js 호환 모듈 해석 알고리즘은 필요하지 않습니다.

## 결과

### 긍정적

- **C/C++ 빌드 의존성 없음** — COSMOS는 `cargo build`만으로 깔끔하게 컴파일되며, 시스템 수준 라이브러리가 필요 없습니다.
- **완전한 샌드박싱 제어** — 실행되는 JS 코드에 사용 가능한 모든 함수는 호스트에 의해 명시적으로 등록됩니다. 기본 I/O, 네트워크, 파일 시스템 접근 없음.
- **Rust 타입과의 긴밀한 통합** — 사용자 정의 호스트 객체에 대한 `boa_gc::Trace` 트레이트 구현, 네이티브 `serde_json` 상호 운용, 가능한 경우 제로 카피.
- **충돌 안전성** — Boa 패닉은 OS 스레드 경계에서 포착되어, 잘못된 JS 코드로 인한 COSMOS 프로세스 충돌을 방지합니다.
- **작은 바이너리 크기** — V8 기반 솔루션 대비 Boa가 COSMOS 바이너리 크기에 추가하는 양이 현저히 적습니다.

### 부정적

- **JavaScript 적합성이 불완전함** — Boa는 ECMAScript 2024+를 완전히 구현하지 않습니다. 일부 고급 기능(예: `WeakRef`, `FinalizationRegistry`, 완전한 `Promise` 통합, 모든 컨텍스트에서의 `async/await`)은 제한이 있거나 구현이 누락될 수 있습니다.
- **V8/`SpiderMonkey` 대비 성능 경쟁력 없음** — Boa 인터프리터는 JIT 컴파일 엔진에 비해 현저히 느립니다. CPU 집약적 워크로드(대규모 데이터 처리, 복잡한 알고리즘)에는 문제가 됩니다. 그러나 IEPL 환경에서 JS 코드는 주로 MCP 도구를 호출하는 오케스트레이션 접착제이며, 계산이 아닙니다.
- **생태계가 작음** — Boa는 V8이나 QuickJS에 비해 기여자와 실전 검증이 적습니다. 업스트림 버그 수정에 더 긴 시간이 소요될 수 있습니다.
- **`eval` 또는 동적 코드 생성 없음** — 설계상(그리고 AST 검증기에 의해 강제됨), 동적 코드 평가가 차단됩니다. 이는 특정 메타프로그래밍 패턴을 제한하지만, 보안 모델에 수용 가능합니다.

### 수용된 절충

**보안 및 임베딩 용이성을 위한 성능 희생.** IEPL 실행 엔진이 복잡한 알고리즘이나 대용량 데이터셋을 처리해야 한다면, Boa는 잘못된 선택이었을 것입니다. 그러나 COSMOS 샌드박스에서 JS 코드는 얇은 오케스트레이션 계층입니다 — 그 역할은 MCP 도구를 올바른 순서로 호출하고, 오류를 처리하며, 결과를 구성하는 것입니다. 실제 무거운 작업은 Rust 기반 에이전트 도구가 수행합니다. Boa가 V8 대비 10-100배 느린 실행 속도는 모든 도구 호출이 50-500ms가 소요되는 Scepter 간 네트워크 왕복을 수반할 때 무관합니다.
