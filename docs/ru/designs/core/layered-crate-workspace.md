
# ADR-004: Слоистая Архитектура Рабочего Пространства из 60+ Крейтов

**Дата**: 2026-03
**Статус**: Принято

## Контекст

Entelecheia начинался с монолитного крейта `packages/shared` (38K строк, 187 `.rs` файлов), который содержал всю общую инфраструктуру: типы, протокол MCP, провайдеров LLM, управление контейнерами, базу данных, безопасность, конфигурацию и многое другое. По мере роста проекта до 12 агентов + 1 агент домена + 3 бинарных пакета возникло несколько проблем:

1. **Время компиляции**: Любое изменение в `shared` требовало перекомпиляции всех 187 файлов, даже если была изменена только одна структура.
1. **Загрязнение зависимостей**: Крейты агентов, которым нужны были только типы MCP, были вынуждены транзитивно зависеть от драйверов баз данных, сред выполнения контейнеров и провайдеров LLM.
1. **Неясное владение**: С 187 файлами в одном крейте было неясно, какой модуль "владеет" какой функциональностью, что делало рефакторинг рискованным.
1. **Взрыв флагов функций**: Условная компиляция через функции Cargo использовалась для избегания подтягивания ненужных зависимостей, но это приводило к комбинаторному взрыву в тестовых конфигурациях.

## Решение

Разложить монолитный `packages/shared` на **37 сфокусированных подкрейтов**, организованных в **6 слоёв зависимостей** (от L0 до L5), следуя строгому направлению зависимостей:

```text
L0 (лист) → L1 → L2 → L3 → L4 → L5 → потребители (scepter, агенты, tui)
```

**Определения слоёв:**

| Слой | Крейты | Правило |
| --- | --- | --- |
| **L0** | core, logging, macros | Нулевые внутренние зависимости от других крейтов entelecheia |
| **L1** | domain_enums, mcp_types, text, concurrent | Зависят только от L0 |
| **L2** | config, agent_registry, state_types | Зависят от L0-L1 |
| **L3** | domain_agent, container, agent_lifecycle, agent_runtime, thread_types, toolchain, infra_utils | Зависят от L0-L2 |
| **L4** | state_sync, domain_skills, hooks, domain_auth, container_runtime, skills_permissions, timeline, iepl | Зависят от L0-L3 |
| **L5** | llm_provider, prompt, custom_agent, storage, infra_jsonrpc, infra_services, e2e_events, adapter, plugin_host, rag, embedding, security_policy | Зависят от L0-L4 |

Все объявления внутренних зависимостей используют `workspace = true` для согласованности версий. Агрегирующего крейта не существует — потребители импортируют напрямую из отдельных подкрейтов.

## Последствия

### Положительные

- **Инкрементальная компиляция**: Изменение в `shared-core` (L0) всё ещё распространяется, но изменение в `shared-security-policy` (L5) перекомпилирует только этот крейт и его прямых потребителей. Время сборки значительно улучшилось.
- **Чёткие границы владения**: Каждый крейт имеет сфокусированную ответственность. Объём проверки кода естественно ограничен границами крейта.
- **Изоляция зависимостей**: Крейты агентов импортируют только те общие крейты, которые им нужны. SkeMma не подтягивает драйверы баз данных. EleOs не подтягивает среды выполнения контейнеров.
- **Предотвращение циклических зависимостей**: Слоистая архитектура делает структурно невозможным создание циклических зависимостей — крейты L3 не могут зависеть от крейтов L5.
- **Тестируемость в изоляции**: Тесты каждого крейта выполняются независимо, без требования полного дерева зависимостей рабочего пространства.

### Отрицательные

- **Накладные расходы управления рабочим пространством**: 60+ крейтов в одном рабочем пространстве означает больше файлов `Cargo.toml` для поддержки, больше секций `[dependencies]` для обновления при повышении версий и более тщательное объявление зависимостей.
- **Межкрейтовый рефакторинг сложнее**: Перемещение типа из L2 в L3 требует обновления всех потребителей L2 и проверки, что ни один крейт L3+ случайно не зависит от перемещённого типа через старое местоположение.
- **Многословность именования крейтов**: Внутренние имена крейтов используют префиксную конвенцию `_shared_*` (например, `_shared_domain_skills_permissions`), что многословно, но необходимо для ясности рабочего пространства.
- **Потенциальная избыточная декомпозиция**: Некоторые крейты (например, `shared-text` с ~200 строками) могут не оправдывать собственные накладные расходы крейта. Декомпозиция следовала философии "разделяй, если может вырасти", а не строгой необходимости.

### Принятый Компромисс

**Сложность управления ради времени компиляции и архитектурной ясности.** Декомпозиция `shared` на 37 крейтов находится на агрессивном конце дизайна рабочих пространств Rust. Золотая середина (10-15 крейтов) была бы проще в управлении. Однако, учитывая широкую поверхность проекта (26 провайдеров LLM, 2 среды выполнения контейнеров, 12 агентов, полный конвейер безопасности, база данных, IEPL), мелкозернистая декомпозиция гарантирует, что каждая часть может развиваться независимо. Шаблон `workspace = true` смягчает накладные расходы управления версиями.
