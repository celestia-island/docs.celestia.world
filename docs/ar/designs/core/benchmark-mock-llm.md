# تصميم خادم Benchmark & Mock LLM

> الموضوع الأساسي: **أي نموذج + نظام أدوات Entelecheia، كم مهمة يمكنه إنجازها؟**
>
> لا افتراض مسبق لتصنيف "ضعيف/قوي". أي مزود تم تمرير مفتاحه في متغير البيئة، يتم إنشاء تكوين حزمة برمجية لذلك النموذج واختباره. لا يوجد أي مفتاح → خطأ وخروج مباشر.

## 0. اكتشاف رئيسي: إعادة استخدام سجل المزودين الموجود

يمتلك Entelecheia **بالفعل** نظام اكتشاف مزودين مدفوعًا بمتغيرات البيئة كاملاً:

- مستودع `provider-registry`: 926 ملف TOML، يغطي جميع المزودين الرئيسيين OpenAI / Anthropic / DeepSeek / GLM / Qwen / Kimi / MiniMax وغيرها
- `derive_config_from_env()`: يجتاز جميع ملفات TOML الخاصة بنقاط الدخول، أي مزود تم ضبط `env_var` الخاص به يُنشَّط تلقائيًا
- `ModelTier` ثلاثي المستويات (Deep / Normal / Basic) + سلسلة احتياطية
- `_shared_llm_provider::ProviderRegistry`: تسجيل شامل، 5 بروتوكولات (OpenAI Chat / Responses / Anthropic v1/v2 / Gemini)

**لا حاجة لإنشاء سجل مزودين جديد** — يعيد مشغل benchmark استخدام `_shared_config` + `_shared_llm_provider` مباشرة.

## 1. مفهوم حزمة البرمجة (Coding Plan)

يُولَّد تلقائيًا ملف تعريف benchmark لكل نموذج متاح:

| الحقل | المصدر | الشرح |
|------|------|------|
| provider_id | TOML نقطة الدخول | مثل `deepseek` / `zhipu_glm` |
| model_id | افتراضيات نقطة الدخول | مثل `deepseek-coder` / `glm-4-plus` |
| tier | ModelTier::Deep | مهام البرمجة تستخدم Deep tier |
| base_url | entrypoint api.base_url | واجهة برمجة بعيدة حقيقية |
| api_key | متغير البيئة (entrypoint api.env_var) | يُقرأ وقت التشغيل |
| protocol | entrypoint api.protocol | قيمة تعداد GenProtocol |
| context_window | بطاقة النموذج | من `models/<provider>/<model>.toml` |
| max_output_tokens | بطاقة النموذج | نفسه |
| supports_function_calling | بطاقة النموذج | يحدد طريقة استدعاء الأدوات |

يُبنى الملف الشخصي ديناميكيًا من متغيرات البيئة وقت التشغيل، دون تخزين ملفات تكوين مسبقًا.

## 3. خادم Mock LLM

### 3.1 بروتوكول التسجيل/إعادة التشغيل

خادم Mock متوافق مع OpenAI Chat Completions API (`/v1/chat/completions`)، يدعم وضعي عمل:

**وضع التسجيل (عند تشغيل نموذج حقيقي لأول مرة)**:
```
Client → Mock Server → Real API → Mock Server (录制响应) → Client
```

**وضع إعادة التشغيل (في CI/دون اتصال)**:
```
Client → Mock Server (匹配请求 → 返回录制的响应) → Client
```

### 3.2 استراتيجية مطابقة الطلبات

تُطابَق الطلبات عبر تجزئة الحقول التالية:
- `model` (اسم النموذج)
- تجزئة محتوى `messages` (بعد التطبيع)
- تجزئة بنية `tools` (إن وجدت)
- معاملات مثل `temperature`، `max_tokens`

**وضع Strict** (افتراضي في CI): أي طلب غير مطابق يسبب خطأ فوريًا، بدون احتياطي إلى API الحقيقي. يضمن اكتمال الـ fixtures.

**وضع Lenient** (للتطوير): عند عدم التطابق، احتياطي إلى API الحقيقي وتسجيل.

### 3.3 تخزين الـ Fixtures

```
tests/fixtures/llm/
├── swe-bench-verified/
│   ├── gpt-4o/
│   │   ├── <request_hash>.json      # 录制的响应
│   │   └── index.toml                # 请求摘要索引
│   ├── claude-sonnet/
│   └── llama-8b/
└── aider-polyglot/
    └── ...
```

### 3.4 اختيار التنفيذ

| الخيار | المزايا | العيوب |
|------|------|------|
| **AIMock** (CopilotKit) | ناضج، يدعم streaming/tool-calls/MCP، صورة Docker | اعتماد خارجي |
| **خادم بسيط ذاتي البناء** | تحكم كامل، صفر اعتماد خارجي | تحتاج لمعالجة streaming/edge case بنفسك |
| **VCR.py / wiremock** | نظام بيئي ناضج للغة | غير مخصص لـ LLM، يحتاج تكييف |

**التوصية**: ابدأ بخادم بسيط ذاتي البناء (موجّه axum/actix، مطابقة + إعادة JSON)، ثم انتقل إلى AIMock لاحقًا عند الحاجة إلى streaming.

## 4. محوّل SWE-bench

### 4.1 تدفق تنفيذ المهمة

```
for instance in dataset:
    1. 拉取 SWE-bench Docker 镜像 (base + env + instance 三层)
    2. 启动容器，挂载代码仓库
    3. 注入 issue 文本作为任务描述
    4. 启动 Entelecheia Agent Runtime（连接到容器内的 bash/文件系统）
    5. Agent 执行直到完成或超时（step cap: 50, wall-clock: 15min）
    6. 容器内执行 git diff → 提取 patch
    7. 输出 JSONL: {instance_id, model_name_or_path, model_patch}
    8. 销毁容器
```

### 4.2 حقن زمن تشغيل الوكيل

عند تشغيل Entelecheia داخل حاوية SWE-bench، يحتاج إلى:
- أدوات معالجة الملفات ← تُعيَّن إلى نظام ملفات الحاوية
- أدوات تنفيذ الأوامر ← تُعيَّن إلى bash داخل الحاوية
- أدوات البحث ← `rg`/`grep` (مثبتة مسبقًا في الحاوية)
- **تعطيل** الوكلاء غير المتعلقين بهذه المهمة (PoleMos/البروتوكولات الصناعية، إلخ)، لتقليل ضوضاء السياق

### 4.3 التقييم

استخدام harness الأصلي لـ SWE-bench مباشرة:
```bash
python -m swebench.harness.run_evaluation \
    --dataset_name princeton-nlp/SWE-bench_Verified \
    --predictions_path entelecheia_predictions.jsonl \
    --max_workers 8 --run_id entelecheia-eval
```

المخرجات: لكل مثيل resolved/unresolved، ومعدل الحل المُجمَّع.

## 5. مصفوفة التجارب

### 5.1 المقارنة الأساسية

تثبيت مجموعة البيانات (SWE-bench Verified)، وتغيير بُعدين:

|  | Baseline (mini-SWE-agent) | Entelecheia |
|--|---------------------------|-------------|
| **GPT-4o** | A₁ | B₁ |
| **Claude Sonnet** | A₂ | B₂ |
| **Llama 3.1 8B** | A₃ | B₃ |
| **Qwen 2.5 7B** | A₄ | B₄ |

- Bᵢ/Aᵢ = معامل التضخيم للنموذج i
- المقارنة عبر الصفوف: هل معامل التضخيم للنماذج الضعيفة (Llama/Qwen) أعلى من النماذج القوية (GPT-4o)؟

### 5.2 تجارب الاستئصال

| التكوين | الغرض |
|------|------|
| Entelecheia كامل (12 وكيل + جميع المهارات) | خط أساس كامل |
| HubRis + KaLos + SkeMma فقط (تخطيط + ملفات + تنفيذ) | قياس الزيادة من تنسيق الوكلاء المتعدد |
| KaLos + bash فقط (وكيل واحد + أدوات الملفات) | قريب من الـ baseline، قياس زيادة سلسلة المهارات |
| بدون هوية الروح (إزالة توجيهات الهوية/الاستعارة) | قياس تأثير توجيهات الشخصية |

## 7. نقاط التكامل مع الهندسة المعمارية الحالية لـ Entelecheia

| المكوّن | طريقة التكامل |
|------|---------|
| `ApoRia::llm_chat` | تبديل base_url للإشارة إلى mock أو API حقيقي |
| تنسيق `SkoPeo` | إضافة وضع تنفيذ `benchmark`، تخطي التأكيد التفاعلي |

| ملفات `KaLos` | تعيين إلى نظام ملفات الحاوية |
| أمان `OreXis` | تخفيف سياسات الأمان في وضع benchmark (السماح بتنفيذ كود تعسفي) |

## 8. ملاحظات

- **التحكم في التكلفة**: SWE-bench Verified الكامل 500 مسألة × 4 نماذج × 2 تكوين = 4000 تشغيل. بتقدير متوسط 50 خطوة/مسألة، كل خطوة ~2K رموز، حوالي 400M رمز. يجب ضبط حد `--max_cost`.
- **موارد الحاويات**: كل مثيل SWE-bench يحتاج حاوية Docker مستقلة، يُنصح بـ ≥120GB قرص، ≥32GB ذاكرة.
- **الحتمية**: وضع Mock يضمن قابلية التكرار في CI؛ الوضع الحقيقي يثبّت `temperature=0` + يسجل تجزئة الطلبات للكشف عن الانحراف.
- **كشف التلوث**: SWE-bench لديه مشكلة تسرب ذاكرة (arXiv:2506.12286)، يُنصح بالاحتفاظ ببعض المهام المُركّبة ذاتيًا كـ holdout.
