# ADR-001：僅執行微核心工具表面

**日期**：2026-02
**狀態**：已接受（2026-06 修訂 — 表面從 5 個原語減少為 3 個）

## 背景

在多代理 LLM 編排系統中，模型必須決定呼叫哪些工具以及如何組合它們。天真的方法是將每個 MCP 工具（跨 12 個代理的 118+ 個）直接暴露給 LLM，作為提示中的單獨函數定義。

這會產生幾個問題：

1. **上下文視窗消耗**：118+ 個工具定義消耗數千個 Token，為推理和對話留下更少的空間。
1. **安全表面積**：每個暴露給 LLM 的工具都是提示注入或越獄的潛在攻擊向量。
1. **權限強制碎片化**：如果工具直接由 LLM 輸出發送，每個工具必須獨立驗證權限 — 導致不一致的強制和漏洞。
1. **模型混淆**：研究顯示當呈現過多工具選擇時，LLM 效能會下降（「工具過載」問題）。

## 決策

我們採用**僅執行微核心**設計。LLM 僅看到精確的 **3 個執行原語**作為其工具表面：

| 原語 | 目的 |
| --- | --- |
| `exec` | 透過 IEPL 管線執行 TypeScript/JavaScript 程式碼 |
| `write_to_var` | 將字串值寫入命名的 REPL 變數 |
| `write_to_var_json` | 將 JSON 值寫入命名的 REPL 變數 |

> **修訂（2026-06）**：原始設計暴露了 5 個原語，包括用於管理命名參考變數的 `ref_add` 和 `ref_remove`。這兩個已從 LLM 可見表面**移除**，因為參考變數機制未被任何技能 SOP 使用，且增加了上下文開銷而無價值。`packages/shared/domain_skills/src/tool_names.rs` 中的 `agent_allowed_tools()` 函數現在僅傳回上述三個原語。參見提交歷史（`60d58f794`、`31cf9a00e`）。

所有 118+ 個代理 MCP 工具都透過 `exec` 程式碼內的 ES 模組匯入**間接**呼叫。LLM 生成匯入並呼叫代理工具的 TypeScript 程式碼；此程式碼由 SWC 轉譯，由 AST 安全檢查器驗證，並由 COSMOS 沙箱內的 Boa JS 引擎執行。

## 後果

### 正面

- **最小上下文開銷**：3 個工具定義 vs 118+。LLM 可以專注於推理而非工具選擇機制。
- **集中式安全**：所有工具呼叫通過 `McpRouter`，它在單一阻塞點中強制執行允許列表、雙重授權、信任等級和動態風險評估。
- **可組合的工作流程**：LLM 可以在 TypeScript 中編寫任意複雜的工具組合（循環、條件、錯誤處理），而非僅限於單一工具呼叫。
- **可稽核的執行**：每個 `exec` 呼叫都經過拒絕危險結構（`eval`、`require`、`process`、`Function`、`import()`、`globalThis` 存取）的 AST 驗證。
- **更容易的模型遷移**：新的 LLM 提供者只需要支援 3 個工具的函數呼叫，而非 118 個。

### 負面

- **間接開銷**：工具呼叫經過 TypeScript 生成 → SWC 轉譯 → AST 驗證 → Boa 執行 → MCP 路由器派送，為每次呼叫增加延遲。
- **LLM 程式碼品質依賴**：系統的有效性取決於 LLM 生成正確匯入並呼叫代理工具的正確 TypeScript 程式碼的能力。
- **除錯複雜性**：當工具呼叫失敗時，錯誤鏈跨 TypeScript 生成、轉譯、驗證、JS 執行和 MCP 派送 — 使除錯比直接工具呼叫更困難。
- **並非所有 LLM 都相等**：較低能力的模型可能在複雜多工具工作流程的程式碼生成上遇到困難，相比於簡單的函數呼叫。

### 已緩解的風險

- 嘗試直接呼叫危險工具的提示注入攻擊
- 由於 LLM 在過多選擇中混淆而導致的意外工具濫用
- 跨工具的不一致權限強制
