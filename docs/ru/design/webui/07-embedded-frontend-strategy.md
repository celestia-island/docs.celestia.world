# Стратегия встроенного фронтенда

## Обзор

shittim-chest поддерживает два режима хостинга фронтенда: в режиме Dev `dev.py` отслеживает исходники фронтенда и запускает `pnpm build` при изменениях, а бэкенд обслуживает как статические файлы, так и API на `:3000`; в режиме Release статические файлы фронтенда встраиваются в бинарник Rust во время компиляции и обслуживаются на `:80`. Режимы переключаются через функцию Cargo `embedded-frontend` с условной компиляцией на уровне кода с использованием `#[cfg(feature = "embedded-frontend")]`.

## Сравнение архитектур

```mermaid
flowchart TB
    subgraph Dev[Режим Dev: dev.py + Бэкенд]
        D1[dev.py отслеживает src фронтенда] --> D2[pnpm build → dist/]
        D2 --> D3[shittim_chest :3000 обслуживает static + API]
    end
    subgraph Release[Режим Release: Встроенный]
        R1[Браузер] --> R2[shittim_chest :80]
        R2 --> R3[API + LLM]
        R2 --> R4[/static/*\nВстроенный SPA]
    end
```

| Измерение | Dev (без функции) | Release (embedded-frontend) |
| --- | --- | --- |
| Исходники фронтенда | Собираются Vite, обслуживаются бэкендом | Встраивание `include_dir!` во время компиляции |
| Горячая перезагрузка | Авто-пересборка через dev.py | Не поддерживается (статический) |
| Маршрутизация API-запросов | Прямое соединение браузера (тот же источник) | Прямое соединение браузера |
| Размер бинарника | Только бэкенд | + директория dist/ фронтенда |
| Требует Node | Да (только сборка) | Нет |
| Метод запуска | `dev.py` (отслеживает + пересобирает) | `just up` одноразовый запуск |

## Детали реализации

### Условная компиляция

```rust
# [cfg(feature = "embedded-frontend")]
static ARONA_DIR: Dir<'_> = include_dir!("$CARGO_MANIFEST_DIR/../../dist/arona");

async fn serve_arona() -> impl IntoResponse {
    #[cfg(feature = "embedded-frontend")]
    {
        // Чтение из встроенной Dir во время компиляции
    }
    #[cfg(not(feature = "embedded-frontend"))]
    {
        // Чтение из файловой системы ./dist/arona/index.html
    }
}
```

Условная компиляция работает на **уровне тела функции**, а не на уровне модуля, сохраняя публичный API идентичным в обоих режимах.

### Запасной вариант SPA

Приложение является одностраничным приложением. Все маршруты, не соответствующие статическим ресурсам, возвращают `index.html`:

```text
GET /               → index.html
GET /chat/123       → index.html (маршрутизатор фронтенда обрабатывает)
GET /backend        → index.html
GET /backend/providers → index.html (маршрутизатор фронтенда обрабатывает)
```

### Определение MIME-типа

Обслуживание статических файлов возвращает правильный Content-Type на основе расширения файла:

| Расширение | Content-Type |
| --- | --- |
| `.js` | `application/javascript` |
| `.css` | `text/css` |
| `.html` | `text/html` |
| `.json` | `application/json` |
| `.png` | `image/png` |
| `.svg` | `image/svg+xml` |
| `.woff/.woff2` | `font/woff2` |
| Другое | `application/octet-stream` |

## Сборка фронтенда в Dockerfile

```text
Этап 1 (фронтенд):
  node:22-slim → pnpm install → pnpm build:all → /app/dist/arona/

Этап 2 (сборщик):
  rust:1.85-slim → COPY /app/dist/ → cargo build --features embedded-frontend

Этап 3 (среда выполнения):
  debian:bookworm-slim → COPY бинарник → ENTRYPOINT ["./shittim_chest"]
```

Сборка фронтенда и компиляция Rust завершаются в одном Dockerfile. Финальный образ среды выполнения содержит только скомпилированный бинарник.

## Дизайнерские решения

1. **Режим Dev использует dev.py для авто-пересборки**: `dev.py` отслеживает исходники фронтенда и пересобирает при изменениях, а бэкенд обслуживает всё на одном порту.
1. **Режим Release не требует обратного прокси**: Бинарник встраивает SPA, обеспечивая одно-процессное развёртывание и снижая эксплуатационную сложность.
1. **Фронтенд не загружается динамически во время выполнения**: Избегает зависимостей файловой системы и несоответствия версий. Образ Release содержит только один бинарный файл.
1. **Единый SPA**: Фронтенд обслуживается на `/` с панелью администратора на `/backend`.
