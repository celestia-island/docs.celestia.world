
# ADR-001: Exec-Onlyマイクロカーネルツールサーフェス

**日付**: 2026-02
**ステータス**: 承認済み（2026-06修正 — サーフェスを5から3のプリミティブに削減）

## 背景

マルチエージェントLLMオーケストレーションシステムでは、モデルはどのツールを呼び出し、それらをどのように構成するかを決定する必要があります。ナイーブなアプローチは、すべてのMCPツール（12エージェントにわたる118以上）をプロンプト内の個別の関数定義としてLLMに直接公開することです。

これによりいくつかの問題が発生します：

1. **コンテキストウィンドウの消費**: 118以上のツール定義が数千トークンを消費し、推論と会話の余地が少なくなります。
1. **セキュリティサーフェスエリア**: LLMに公開されるすべてのツールは、プロンプトインジェクションやジェイルブレイクの潜在的な攻撃ベクトルです。
1. **権限強制の断片化**: ツールがLLM出力によって直接ディスパッチされる場合、各ツールは独立して権限を検証する必要があり、一貫性のない強制とギャップにつながります。
1. **モデルの混乱**: 研究によると、LLMのパフォーマンスはあまりにも多くのツール選択肢が提示されると低下します（「ツール過負荷」問題）。

## 決定

**exec-onlyマイクロカーネル**設計を採用します。LLMはツールサーフェスとして正確に**3つの実行プリミティブ**を見ます：

| プリミティブ | 目的 |
| --- | --- |
| `exec` | IEPLパイプラインを通じてTypeScript/JavaScriptコードを実行 |
| `write_to_var` | 文字列値を名前付きREPL変数に書き込み |
| `write_to_var_json` | JSON値を名前付きREPL変数に書き込み |

> **修正（2026-06）**: 元の設計では、名前付き参照変数を管理するための`ref_add`と`ref_remove`を含む5つのプリミティブを公開していました。この2つは、参照変数メカニズムがいずれのスキルSOPでも使用されておらず、価値のないコンテキストオーバーヘッドを追加していたため、LLM可視サーフェスから**削除**されました。`packages/shared/domain_skills/src/tool_names.rs`の`agent_allowed_tools()`関数は現在、上記の3つのプリミティブのみを返します。コミット履歴（`60d58f794`、`31cf9a00e`）を参照してください。

118以上のすべてのエージェントMCPツールは、`exec`コード内のESモジュールインポートを通じて**間接的に**呼び出されます。LLMはエージェントツールをインポートして呼び出すTypeScriptコードを生成します；このコードはSWCによってトランスパイルされ、ASTセキュリティチェッカーによって検証され、COSMOSサンドボックス内のBoa JSエンジンによって実行されます。

## 結果

### ポジティブ

- **最小限のコンテキストオーバーヘッド**: 118以上ではなく3つのツール定義。LLMはツール選択の仕組みではなく推論に集中できます。
- **集中化されたセキュリティ**: すべてのツール呼び出しが`McpRouter`を通過し、単一のチョークポイントで許可リスト、二重認証、信頼レベル、動的リスク評価を強制します。
- **構成可能なワークフロー**: LLMは単一のツール呼び出しに制限されるのではなく、TypeScriptで任意に複雑なツール構成（ループ、条件分岐、エラーハンドリング）を記述できます。
- **監査可能な実行**: すべての`exec`呼び出しは危険な構文（`eval`、`require`、`process`、`Function`、`import()`、`globalThis`アクセス）を拒否するAST検証を通過します。
- **モデル移行の容易さ**: 新しいLLMプロバイダは118ではなく3つのツールでの関数呼び出しをサポートするだけで済みます。

### ネガティブ

- **間接化のオーバーヘッド**: ツール呼び出しはTypeScript生成 → SWCトランスパイル → AST検証 → Boa実行 → MCPルーターディスパッチを経由し、各呼び出しにレイテンシが追加されます。
- **LLMコード品質への依存**: システムの有効性は、エージェントツールを適切にインポートして呼び出す正しいTypeScriptコードを生成するLLMの能力に依存します。
- **デバッグの複雑さ**: ツール呼び出しが失敗した場合、エラーチェーンはTypeScript生成、トランスパイル、検証、JS実行、MCPディスパッチに及び、直接的なツール呼び出しよりもデバッグが困難です。
- **すべてのLLMが同等ではない**: 低能力モデルは、シンプルな関数呼び出しと比較して複雑なマルチツールワークフローのコード生成に苦労する可能性があります。

### 軽減されたリスク

- 危険なツールを直接呼び出そうとするプロンプトインジェクション攻撃
- あまりにも多くの選択肢によるLLMの混乱による偶発的なツール誤用
- ツール間での一貫性のない権限強制
