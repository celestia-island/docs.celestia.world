## 1. CrewAI

**المستودع**: [crewAIInc/crewAI](https://github.com/crewAIInc/crewAI)
**اللغة**: Python
**الترخيص**: MIT
**الحجم**: ~23k+ نجمة. مستقل عن LangChain.

### البنية

- **الوكلاء**: معرّفون عبر YAML (الدور، الهدف، القصة الخلفية) أو صنف `Agent` بلغة Python. كل وكيل يلفّ LLM مع وصول للأدوات.
- **التنسيق**: وضعان:
  - **Crews**: فريق من الوكلاء بعمليات تسلسلية أو هرمية. التسلسلي يشغل المهام بالترتيب؛ الهرمي يعيّن وكيل "مدير" للتفويض.
  - **Flows**: DAG مدفوع بالأحداث مع مزخرفات `@start`، `@listen`، `@router`. حالة محددة النوع Pydantic. يدعم مجموعات شروط `and_`/`or_`.
- **الاتصال**: تمرير الرسائل عبر بيئة تشغيل Crew/Flow. ينتج الوكلاء مخرجات منظّمة (`output_pydantic`، `output_json`).
- **أنواع العمليات**: `Process.sequential` و`Process.hierarchical`.

### تعريض الأدوات

- أدوات مدمجة عبر حزمة `crewai[tools]` (SerperDev، إلخ). أدوات مخصصة كدوال Python.
- دعم MCP (Model Context Protocol) موثق.
- الأدوات معيّنة لكل وكيل وقت التعريف.
- لا حد صريح على الأدوات لكل استدعاء LLM — كل الأدوات المعيّنة معروضة في كل دور.

### نموذج الأمان

- **لا عزل**. الوكلاء تعمل في نفس عملية Python مثل التنسيق.
- "AMP Suite" المؤسسية تقدم مستوى تحكم مع قابلية الملاحظة وتحكمات الوصول (احتكارية).
- إنسان في الحلقة عبر `human_input=True` على المهام.
- لا ذكر لعزل تنفيذ الكود.

### الذاكرة/السياق

- قصيرة المدى: ذاكرة الوكيل عبر سجل المحادثة.
- طويلة المدى: مخازن ذاكرة اختيارية مُفعّلة عبر `memory=True` على الوكلاء.
- لا دمج سياق صريح أو إدارة token — يعتمد على نافذة سياق LLM.
- نقاط التفتيش مذكورة في الوثائق لكن التفاصيل ضئيلة في OSS.

### ميزات فريدة

- **تناغم Flows + Crews**: دمج فرق وكلاء مستقلة مع تدفقات عمل مدفوعة بالأحداث دقيقة.
- **تهيئة YAML أولاً**: الوكلاء والمهام معرّفون إقراريًا، مناسبون لغير المطورين.
- **مجتمع كبير**: أكثر من 100 ألف مطور معتمد عبر `learn.crewai.com`.
- **ادعاءات أداء**: أسرع 5.76× من LangGraph في مهام QA معينة (ذاتي البلاغ).

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- لا عزل أو تحديد لتنفيذ الكود.
- لا نموذج أمان رسمي لتنفيذ الأدوات.
- الذاكرة بسيطة نسبيًا — لا إدارة سياق هرمية أو أرشفة.
- بيئة تشغيل Python أحادية العملية تحد التوسع عبر الأجهزة.
- لا تكامل متصفح/قشرة أصلي لمهام برمجة الوكلاء.
- التنسيق Python فقط (لا بيئة تشغيل متعددة اللغات).

## 2. LangGraph

**المستودع**: [langchain-ai/langgraph](https://github.com/langchain-ai/langgraph)
**اللغة**: Python (أيضًا JS/TS عبر `langgraphjs`)
**الترخيص**: MIT

### البنية

- **آلة حالة قائمة على الرسم البياني**: العقد (وكلاء/دوال) والحواف (انتقالات) تشكل رسمًا بيانيًا موجهًا مستوحى من Pregel/Beam.
- **الوكلاء**: يمكن أن تكون العقد استدعاءات LLM، عمليات تنفيذ أدوات، أو أي دالة Python. غير مصنّفة بقوة كـ "وكلاء" — أشبه بدوال في رسم بياني.
- **التنسيق**: `StateGraph` مع مخطط حالة محدد النوع. العقد تقرأ/تكتب الحالة. حواف شرطية للتفرع. رسوم فرعية للتركيب.
- **الاتصال**: كائن الحالة هو مصدر الحقيقة الوحيد. الرسائل مُلحقة بقوائم الحالة.

### تعريض الأدوات

- الأدوات هي أدوات LangChain أو callables عشوائية مرتبطة بعقد الرسم البياني.
- كل الأدوات المتاحة في عقدة معروضة على LLM في تلك الخطوة.
- لا خنق أدوات مدمج؛ يدير المطورون ما الأدوات الممررة لكل عقدة.

### نموذج الأمان

- **لا عزل**. تنفيذ الكود مسؤولية المطور.
- إنسان في الحلقة عبر `interrupt()` — يوقف تنفيذ الرسم البياني، يسمح بفحص/تعديل الحالة.
- تنفيذ متين: الحالة مستمرة، يمكن الاستئناف بعد الفشل (نقاط تفتيش).
- لا بدائيات عزل لتنفيذ الأدوات أو الوصول لنظام الملفات.

### الذاكرة/السياق

- **قصيرة المدى**: ذاكرة عاملة عبر الحالة (قوائم الرسائل).
- **طويلة المدى**: ذاكرة مستمرة عبر الجلسات عبر تجريد `Store` (مفتاح-قيمة مع تضمينات).
- دمج السياق غير مدمج — يدير المطورون حجم الحالة.
- نقاط التفتيش عبر `MemorySaver` أو `SqliteSaver`.

### ميزات فريدة

- **التنفيذ المتين**: يستأنف تلقائيًا من نقطة التفتيش بعد الفشل/المهلة — مثالي للوكلاء طويلي التشغيل.
- **إنسان في الحلقة عبر المقاطعات**: نمط قوي لتدفقات العمل المعتمدة.
- **تكامل LangSmith**: قابلية ملاحظة عميقة، تتبع، وتقييم.
- **نشر LangSmith**: منصة نشر إنتاج مع تصميم أولي بصري.
- **Deep Agents**: مشروع فرعي جديد لوكلاء يخططون، يستخدمون وكلاء فرعيين، ويستفيدون من أنظمة الملفات.
- **منظومة LangChain**: تكامل سلس مع أدوات ونماذج ومكونات LangChain.

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- مقترن بإحكام بمنظومة LangChain (رغم "يمكن استخدامه بدون LangChain").
- لا نموذج أمان — لا عزل، لا نظام صلاحيات.
- إطار منخفض المستوى — يتطلب boilerplate كبير لتفاعلات الوكلاء.
- لا بروتوكول اتصال أصلي متعدد الوكلاء (A2A).
- نهج رسم الحالة يمكن أن يصبح مرهقًا لتفاعلات الوكلاء المعقدة.
- لا دعم مدمج لعزل تنفيذ الكود.

## 3. MetaGPT

**المستودع**: [`FoundationAgents`/MetaGPT](https://github.com/`FoundationAgents`/MetaGPT)
**اللغة**: Python
**الترخيص**: MIT
**البحث**: منشور في ICLR 2024

### البنية

- **متعدد الوكلاء قائم على SOP**: يصمم شركة برمجيات بأدوار معرّفة مسبقًا (مدير المنتج، المهندس المعماري، المهندس، إلخ).
- **الوكلاء (الأدوار)**: كل `Role` لديه ملف، هدف، قيود، ومجموعة `Action`s. تستخدم الأدوار حلقات ReAct (فكر ← تصرف) بثلاثة أوضاع: `REACT`، `BY_ORDER`، `PLAN_AND_ACT`.
- **التنسيق**: صنف `Team` يوظف الأدوار، يستثمر الميزانية، يشغل الجولات. يدير `Environment` تمرير الرسائل بين الأدوار عبر النشر-الاشتراك.
- **الاتصال**: قائم على الرسائل pub/sub عبر Environment. تشترك الأدوار في وسوم رسائل محددة.
- **الفلسفة الأساسية**: `Code = SOP(Team)` — إجراءات تشغيل معيارية مجسدة.

### تعريض الأدوات

- الإجراءات أصناف Python معرّفة مسبقًا (`WriteCode`، `DesignAPI`، `DebugError`، إلخ) — ~40+ نوع إجراء.
- كل دور يحصل على إجراءات محددة معيّنة وقت الإنشاء.
- الأدوات تشمل: محركات بحث الويب (Serper، SerpAPI، DuckDuckGo، Google، Bing)، متصفحات الويب (Playwright، Selenium)، توليد الصور (DALL-E)، مخازن المستندات (Chroma، FAISS، Milvus، LanceDB، Qdrant).
- يرى LLM فقط مخططات الإجراءات لإجراءاته المعيّنة حاليًا، وليس مجموعة الأدوات الكاملة.

### نموذج الأمان

- **لا عزل**. الكود المُولّد وغير المنفّذ في نفس البيئة.
- Dockerfile مقدم لكن للنشر، وليس لعزل لكل مهمة.
- تتبع الميزانية: معامل `investment` يحد تكلفة API الإجمالية، يطلق استثناء عند التجاوز.
- لا إنسان في الحلقة أثناء التنفيذ.
- لا عزل تنفيذ الأدوات أو نموذج صلاحيات.

### الذاكرة/السياق

- **قصيرة المدى**: صنف `Memory` في `RoleContext` — قائمة مرتبة من الرسائل لكل دور.
- **طويلة المدى**: أصناف `LongTermMemory` و`BrainMemory` للمعرفة المستمرة.
- **الذاكرة العاملة**: ذاكرة عاملة منفصلة لعمليات المخطط.
- **مخزن الرسائل مؤقت**: قائمة رسائل غير متزامنة مع تصفية حسب الوسوم المشترك بها.
- دمج السياق غير معالج صراحةً — تلاحظ الأدوار مجموعة فرعية مُرشّحة من الرسائل.

### ميزات فريدة

- **محاكاة SDLC كاملة**: تصمم شركة برمجيات كاملة بـ SOPs — قصص المستخدم، المتطلبات، وثائق التصميم، الكود، الاختبارات.
- **مخازن مستندات متعددة**: 5+ خيارات قاعدة بيانات متجهة.
- **دعم موسع للمزودين**: 12+ مزود LLM (OpenAI، Azure، Anthropic، Gemini، Ollama، Bedrock، إلخ).
- **مُفسّر البيانات**: وكيل متخصص لمهام علم البيانات.
- **مخرجات بحثية**: أوراق منشورة متعددة (AFlow، SPO، SELA، FACT، Data Interpreter).
- **MGX**: منتج برمجة باللغة الطبيعية مبني فوقه.

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- SOP صارم — الأدوار والإجراءات معرّفة مسبقًا؛ الأدوار المخصصة تتطلب برمجة.
- لا عزل أمان إطلاقًا.
- بنية آلة واحدة — لا نشر موزع للوكلاء.
- لا تحكم وكلاء قائم على المتصفح (CLI/API فقط).
- نموذج الذاكرة أساسي إلى حد ما لكل دور.
- لا دعم بروتوكول MCP/A2A بين الوكلاء.
- تتبع الميزانية تكلفة فقط، وليس قائمًا على الموارد/الأمان.

## 4. ChatDev 2.0 (DevAll)

**المستودع**: [OpenBMB/ChatDev](https://github.com/OpenBMB/ChatDev)
**اللغة**: Python (خلفية)، Vue 3 (واجهة أمامية)
**الترخيص**: Apache 2.0
**البحث**: أوراق NeurIPS/arxiv متعددة

### البنية

- **منصة متعددة الوكلاء بدون كود**: الوكلاء وتدفقات العمل معرّفة بالكامل في تهيئة YAML. لا برمجة مطلوبة.
- **DAG تدفق عمل مدفوع بـ YAML**: العقد تعرّف الوكلاء، الحواف تعرّف تدفق الرسائل. يدعم الرسوم الفرعية. لوحة سحب وإفلات بصرية في واجهة الويب.
- **الوحدات الأساسية**: `runtime/` (تنفيذ الوكيل)، `workflow/` (تنسيق DAG)، `entity/` (التهيئة)، `server/` (FastAPI + WebSocket)، `frontend/` (وحدة تحكم ويب Vue 3).
- **الوكلاء**: معرّفون في تهيئات عقد YAML مع توجيهات، تهيئة LLM، أدوات، وإعدادات ذاكرة.
- **التنسيق**: أنواع منفّذين متعددة: تسلسلي، DAG، متوازي، دورة، حافة ديناميكية. باني الطوبولوجيا يحول تهيئات YAML إلى رسوم بيانية قابلة للتنفيذ.

### تعريض الأدوات

- **نظام استدعاء الدوال**: يحتوي `functions/function_calling/` على أدوات مدمجة (`code_executor`، ملف، طقس، ويب، فيديو، `deep_research`، uv، مستخدم).
- **تسجيل أدوات مخصصة**: الدوال Python في دليل `functions/` تُكتشف تلقائيًا.
- **دعم MCP**: `mcp_example/mcp_server.py` يوضح تكامل MCP.
- الأدوات معيّنة لكل عقدة في تهيئة YAML.

### نموذج الأمان

- **تنفيذ الكود**: `code_executor.py` مخصص مع معاملات تنفيذ قابلة للتهيئة.
- نشر Docker Compose متاح.
- **إنسان في الحلقة**: تدفق عمل `demo_human.yaml`، عقد إدخال المستخدم، تدفقات التأكيد.
- **بث WebSocket**: مراقبة سجل فورية وفحص المنتجات.
- لا نموذج عزل/تحديد صريح للوكلاء.

### الذاكرة/السياق

- **واجهات ذاكرة خلفية متعددة**: ذاكرة بسيطة، ذاكرة `mem0` (مستمرة، قابلة للتعلم)، ذاكرة قائمة على الملفات.
- **تهيئة الذاكرة في YAML**: `store`، `context_window_size`، نوع الذاكرة لكل عقدة.
- **عقد إعادة تعيين السياق**: عقد `context_reset` صريحة في تدفق العمل.
- **اتساق تضمين الذاكرة**: اختبارات لاتساق التضمين عبر الجلسات.
- **مسح مساحة العمل**: `workspace_scanner.py` لحقن السياق القائم على الملفات.

### ميزات فريدة

- **بدون كود**: ابنِ أنظمة متعددة الوكلاء دون كتابة كود Python — YAML + واجهة ويب.
- **وحدة تحكم ويب سحب وإفلات**: مصمم تدفق عمل بصري، مراقبة إطلاق فورية.
- **قوالب تدفق عمل غنية**: تصور البيانات، توليد 3D (Blender)، تطوير الألعاب، بحث عميق، فيديو تعليمي.
- **دعم الرسم الفرعي**: تدفقات عمل فرعية قابلة لإعادة الاستخدام (`react_agent.yaml`، `reflexion_loop.yaml`).
- **تكامل OpenClaw**: يمكن استدعاؤه بواسطة وكلاء برمجة OpenClaw لإنشاء فرق وكلاء ديناميكيًا.
- **Python SDK**: حزمة `chatdev` PyPI لتنفيذ تدفق العمل البرمجي.
- **سلالة بحثية**: MacNet، Puppeteer، IER، التعلم التجريبي المشترك — كلها مبنية على ChatDev.
- **كتاب إلكتروني متعدد الوكلاء**: مجموعة منظّمة من بحث الوكلاء المتعددين.

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- YAML-مركزية تحد القابلية للتعبير عن المنطق المعقد.
- لا عزل أمان لتنفيذ الكود.
- وحدة تحكم الويب هي الواجهة الرئيسية — أقل ملاءمة للاستخدام headless/المضمّن.
- Django-like: بنية مشروع ذات رأي، أقل مرونة من أطر Python الأصلية.
- لا دعم وكلاء متعددي اللغات.
- المجتمع موجه للأبحاث وليس المؤسسات.

## 5. Google ADK (Agent Development Kit)

**المستودع**: [google/adk-python](https://github.com/google/adk-python)
**اللغة**: Python (أيضًا إصدارات Java، Go)
**الترخيص**: Apache 2.0

### البنية

- **كود أولاً**: الوكلاء، الأدوات، والتنسيق معرّفون في كود Python.
- **تجريدات أساسية**: Agent (مخطط)، Tool (قدرة)، Runner (محرك)، Session (حالة المحادثة)، Memory (استدعاء عبر الجلسات)، Artifact Service (ملفات).
- **أنواع الوكلاء**: `LlmAgent` (مدفوع بـ LLM)، `LoopAgent`، `SequentialAgent`، `ParallelAgent`، `RemoteA2aAgent`.
- **تركيب متعدد الوكلاء**: الوكيل الأصل لديه قائمة `sub_agents`. يعالج Runner التوجيه بين الوكلاء.
- **بروتوكول A2A**: دعم أصلي لبروتوكول اتصال الوكيل-إلى-الوكيل للوكلاء البعيدين.
- **تكامل LangGraph**: `langgraph_agent.py` لتضمين رسوم LangGraph البيانية كوكلاء.

### تعريض الأدوات

- **منظومة أدوات غنية**: 50+ أداة مدمجة — Google Search، BigQuery، Bigtable، Spanner، PubSub، Vertex AI Search، أدوات MCP، أدوات OpenAPI، أدوات LangChain، أدوات CrewAI، استخدام الكمبيوتر، bash، تنفيذ الكود، Google APIs.
- **أنواع الأدوات**: `FunctionTool`، `AgentTool` (تغليف وكيل كأداة)، `MCPTool`، `OpenAPITool`، `LangChainTool`، `CrewAiTool`، `SkillToolset`.
- **تأكيد الأداة (HITL)**: تدفق تأكيد صريح قبل تنفيذ الأداة مع إدخال مخصص.
- **نمط Toolbox**: `toolbox_toolset.py` لحزم الأدوات.

### نموذج الأمان

- **عزل تنفيذ الكود**: منفذون متعددون — `container_code_executor.py`، `unsafe_local_code_executor.py`، `vertex_ai_code_executor.py`، `agent_engine_sandbox_code_executor.py`، `gke_code_executor.py`.
- **نظام مصادقة**: تدفق OAuth2 كامل، إدارة بيانات الاعتماد، معالجات مسبقة للمصادقة، `authenticated_function_tool.py`.
- **إنسان في الحلقة**: تأكيد الأداة، دعم المقاطعة.
- **المهارات**: قدرات وكلاء قابلة للحزم والتحكم بالإصدار مع سياق تنفيذ منفصل.
- **هوية الوكيل**: تكامل `agent_identity/` لحسابات الخدمة.

### الذاكرة/السياق

- **الجلسة**: سجل محادثة كامل لكل جلسة، مستمر عبر `SessionService` (في الذاكرة، SQLite، PostgreSQL، Vertex AI).
- **الذاكرة**: استدعاء عبر الجلسات عبر `MemoryService` — في الذاكرة، Vertex AI Memory Bank، Vertex AI RAG.
- **دمج السياق**: `compaction.py` و`llm_event_summarizer.py` للتلخيص التلقائي للسياق.
- **خدمة المنتجات**: معالجة منفصلة للبيانات غير النصية (ملفات، صور).
- **تخزين السياق مؤقتًا**: `gemini_context_cache_manager.py` لتخزين السياق المؤقت الخاص بـ Gemini.
- **التراجع**: القدرة على تراجع الجلسة قبل استدعاء سابق.

### ميزات فريدة

- **دعم متعدد اللغات**: إصدارات Python، Java، Go من ADK.
- **النشر الإنتاجي**: `adk deploy` إلى Cloud Run، Vertex AI Agent Engine، GKE.
- **بروتوكول A2A**: مولود من Google — اتصال متعدد البائعين أصلي للوكلاء.
- **ADK Web**: واجهة تطوير/تصحيح مدمجة مع تتبع الأحداث.
- **إطار التقييم**: اختبار متعدد الطبقات (وحدة، تكامل، تقييمات) مع تسجيل المسار.
- **نظام الإضافات**: تسجيل تصحيح، تصفية السياق، نتائج متعددة الوسائط، حفظ المنتجات، تحليلات BigQuery.
- **نظام المخطط**: مخطط مدمج مع نمط خطط-ثم-نفّذ.
- **دعم vibe coding**: `llms.txt` و`llms-full.txt` لسياق LLM حول ADK.
- **نظام المهارات**: قدرات وكلاء قابلة لإعادة الاستخدام والحزم.
- **تحسين الوكيل**: `agent_optimizer.py` ومحسّن توجيهات GEPA.

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- اعتماد قوي على Google Cloud لميزات الإنتاج.
- مُحسّن لـ Gemini (رغم أنه غير محدود بالنموذج).
- قاعدة كود معقدة مع طبقات تجريد كثيرة.

- معالجة الذاكرة/السياق واعية بـ Gemini-cache، أقل عمومية للنماذج الأخرى.

## 6. OpenAI Swarm (تجريبي، متقادم الآن)

**المستودع**: [openai/swarm](https://github.com/openai/swarm)
**اللغة**: Python
**الترخيص**: MIT
**الحالة**: متجاوز بـ [OpenAI Agents SDK](https://github.com/openai/openai-agents-python)

### البنية

- **بدائيات مينيمالية**: `Agent` (توجيهات + دوال) والتسليمات (إرجاع وكيل آخر من دالة).
- **الحلقة الأساسية** (~300 سطر في `swarm/core.py`):

  1. احصل على إكمال من الوكيل الحالي
  1. نفّذ استدعاءات الأدوات، ألحق النتائج
  1. بدّل الوكيل إذا أرجعت دالة وكيلًا
  1. حدّث متغيرات السياق
  1. كرر حتى لا مزيد من استدعاءات الأدوات أو الوصول لـ `max_turns`

- **الوكلاء**: مجرد اسم، نموذج، توجيهات (نص أو قابل للاستدعاء)، قائمة دوال. لا هرمية وكلاء — تفويض مسطح عبر التسليمات.
- **الاتصال**: رسائل Chat Completions API. بدون حالة بين استدعاءات `client.run()`.

### تعريض الأدوات

- الأدوات هي دوال Python عادية. مخطط تلقائي من تلميحات النوع وسلاسل التوثيق.
- كل الدوال المعيّنة لوكيل معروضة لكل استدعاء LLM.
- إذا أرجعت دالة `Agent`، ينتقل التنفيذ (تسليم).
- معامل `context_variables` يُملأ تلقائيًا إذا كان معرّفًا في توقيع الدالة.

### نموذج الأمان

- **لا شيء**. لا عزل، لا تحديد، لا نموذج صلاحيات. الأدوات تعمل في عملية المستدعي.
- مشروع تعليمي/تجريبي صراحةً ليس للإنتاج.

### الذاكرة/السياق

- **بدون حالة**: لا حالة بين استدعاءات `client.run()`. يجب على المستخدم تمرير `messages` وإعادتها.
- **متغيرات السياق**: dict بسيط يمر عبر استدعاءات الدوال — يمكن قراءته/كتابته بواسطة دوال الأدوات.
- لا ذاكرة، لا استمرارية الجلسة، لا دمج السياق.

### ميزات فريدة

- **بساطة قصوى**: الإطار كامل ~4 ملفات مصدرية. رائع لتعلم تنسيق الوكلاء.
- **نمط التسليم**: أنيق — الوكيل مجرد "توجيهات + أدوات" والوكلاء يفوضون بإرجاع وكيل آخر من دالة أداة.
- **دعم التدفق**: تدفق مدمج مع علامات `{"delim":"start"}` / `{"delim":"end"}` لحدود الوكلاء.

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- متقادم (خلفه OpenAI Agents SDK).
- لا نموذج أمان إطلاقًا.
- لا حالة/استمرارية — بدون حالة تمامًا.
- OpenAI فقط (Chat Completions API).
- لا اتصال متعدد الوكلاء بعد التسليمات.
- تجريبي/تعليمي — ليس إطار إنتاج.

## 7. Cline

**المستودع**: [cline/cline](https://github.com/cline/cline)
**اللغة**: TypeScript (إضافة VS Code) + Go (CLI)
**الترخيص**: Apache 2.0

### البنية

- **إضافة VS Code** + CLI مستقل. الإضافة هي الواجهة الرئيسية؛ CLI أحدث.
- **الحلقة الأساسية** (في `src/core/`):

  1. حلل مهمة المستخدم (نص + صور)
  1. حلل مساحة العمل (ASTs، بحث regex، قراءات ملفات)
  1. نفّذ الأدوات في حلقة: إنشاء/تحرير ملف، أوامر طرفية، إجراءات متصفح
  1. راقب المخرجات (أخطاء linter، مخرجات طرفية، لقطات متصفح)
  1. أصلح المشكلات تلقائيًا، كرر حتى اكتمال المهمة

- **مجموعة الأدوات**: عمليات الملفات (إنشاء، تحرير، diff)، أوامر طرفية (مع تكامل قشرة)، متصفح (headless، نقر/كتابة/تمرير)، أدوات MCP.
- **توجيه النظام**: هندسة توجيهات مفصلة مع تعليمات إدارة السياق.

### تعريض الأدوات

- مجموعة ثابتة من الأدوات المدمجة: `read_file`، `write_to_file`، `replace_in_file`، `execute_command`، `browser_action`، `use_mcp_tool`، إلخ.
- **امتداد MCP**: يمكن إنشاء/تثبيت خوادم MCP جديدة عند الطلب ("أضف أداة تفعل..."). خوادم MCP المجتمعية مدعومة أيضًا.
- **@-إشارات**: `@url`، `@problems`، `@file`، `@folder` لحقن السياق (تقليل تكاليف API).
- عدد محدود من الأدوات لكل استدعاء — عادةً 8-12 مدمجة + أدوات MCP.

### نموذج الأمان

- **إنسان في الحلقة لكل الإجراءات**: كل تغيير ملف وأمر طرفية يجب أن يوافق عليه المستخدم في الواجهة الرسومية.
- **نظام نقاط التفتيش**: لقطات مساحة العمل قبل كل خطوة. يمكن عمل diff/استعادة في أي نقطة.
- **نظام الصلاحيات**: `CommandPermissionController` مع قوائم سماح/رفض.
- **Cline Ignore**: ملف `.clineignore` لاستبعاد الملفات الحساسة.
- **لا عزل**: أوامر الطرفية تعمل في بيئة المستخدم الفعلية — قوة ومخاطرة.
- **المؤسسة**: SSO، مسارات التدقيق، VPC/private link، مُستضاف ذاتيًا/on-prem.

### الذاكرة/السياق

- **إدارة السياق**: تحليل AST للمشروع، بحث regex للملفات ذات الصلة، اختيار دقيق لما يدخل نافذة السياق.
- **دمج السياق**: عندما يمتلئ السياق، تُولّد الملخصات وتُضغط المحادثة القديمة.
- **نقاط التفتيش**: لقطات مساحة عمل كاملة للتراجع.
- **حالة المهمة**: `TaskState.ts` يدير ذاكرة المحادثة وحالة مساحة العمل.
- **كشف الحلقة**: `loop-detection.ts` يمنع حلقات استدعاء الأدوات اللانهائية.

### ميزات فريدة

- **تكامل IDE كامل**: يعيش داخل VS Code، يرى مساحة عملك بالكامل.
- **أتمتة المتصفح**: قدرة Claude Computer Use لاختبار/تصحيح الويب.
- **حلقة الإصلاح التلقائي**: تراقب أخطاء linter/المترجم وتُصلحها تلقائيًا دون تدخل المستخدم.
- **إنشاء أداة MCP**: يمكن إنشاء خوادم MCP جديدة على الطاوية بناءً على طلبات المستخدم.
- **المتابعة أثناء التشغيل**: تنفيذ أوامر الطرفية غير الحاجب.
- **متعدد النماذج**: OpenRouter، Anthropic، OpenAI، Gemini، Bedrock، Azure، Vertex، Cerebras، Groq، Ollama، LM Studio.
- **Go CLI**: خيار CLI مستقل للبيئات بدون VS Code.
- **إطار التقييم**: اختبار 3 طبقات (عقد، دخان، E2E/مقاعد).

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- مقيد بـ VS Code للتجربة الكاملة (CLI أحدث وأقل نضجًا).
- بنية وكيل واحد — لا تعاون متعدد الوكلاء.
- لا أدوار/تخصص وكلاء معرّفة.
- موافقة بشرية مطلوبة لكل إجراء (رغم إمكانية الأتمتة).
- لا تشغيل مستقل طويل الأمد بدون تفاعل.
- ليس إطارًا — تطبيق. لا يمكن تضمينه كمكتبة.
- TypeScript/Go فقط، لا Python SDK.

## 8. Aider

**المستودع**: [Aider-AI/aider](https://github.com/Aider-AI/aider)
**اللغة**: Python
**الترخيص**: Apache 2.0

### البنية

- **برمجة زوجية قائمة على الطرفية**: أداة CLI تحرر الكود في مستودعك.
- **الحلقة الأساسية** (`base_coder.py`):

  1. ابنِ خريطة المستودع (ملخص قاعدة الكود قائم على AST لـ Tree-sitter)
  1. أرسل التوجيه + خريطة المستودع + الملفات إلى LLM
  1. حلل استجابة LLM لتعليمات التحرير (diffs موحدة، كتل بحث/استبدال، إعادة كتابة ملف كامل)
  1. طبق التحريرات، lint، شغل الاختبارات، أصلح الفشل تلقائيًا
  1. Git commit برسائل معقولة

- **تنسيقات تحرير متعددة**: `udiff`، `editblock`، `wholefile`، `search_replace`، `diff_fenced`، `editor_editblock`، `editor_whole`، `patch`، `architect`. كل واحد صنف coder منفصل مع قالب توجيه خاص به.
- **وضع المهندس المعماري/المحرر**: نمط وكيلين — المهندس يخطط، المحرر ينفذ. أثبت فعالية عالية في المعايير القياسية.

### تعريض الأدوات

- **لا استدعاء أدوات تقليدي**. يستخدم Aider استجابات نصية منظّمة (وليس استدعاء الدوال).
- يُخرج LLM تعليمات التحرير في تنسيقات محددة (كتل diff، بحث/استبدال) يحللها Aider ويطبقها.
- أوامر القشرة: يمكن لـ LLM طلب تشغيل أوامر قشرة (مؤكدة من قبل المستخدم).
- كشط الويب: متصفح قائم على Playwright لقراءة صفحات الويب.
- إدخال صوتي: كلام-إلى-كود عبر الميكروفون.
- صور: يمكن إضافة لقطات شاشة/صور للدردشة للسياق البصري.

### نموذج الأمان

- **لا عزل**. التحريرات مطبقة مباشرة على ملفاتك. Git يوفر شبكة أمان.
- **تأكيد المستخدم**: أوامر القشرة تتطلب موافقة صريحة من المستخدم.
- **تكامل Git**: كل التغييرات ملتزمة تلقائيًا — تراجع سهل.
- **وضع المراقبة**: يمكن مراقبة تغييرات الملفات وإعادة التطبيق التلقائي إذا فشلت الاختبارات.
- **Lint/اختبار**: يشغل تلقائيًا linters وأجنبة الاختبار المُهيّأة بعد التغييرات.
- لا نظام صلاحيات، لا عزل، لا وصول قائم على الأدوار.

### الذاكرة/السياق

- **خريطة المستودع**: تحليل AST لـ Tree-sitter يبني خريطة موجزة لقاعدة الكود بالكامل (تواقيع الدوال، تعريفات الأصناف، علاقات الاستيراد). هذا يضع بنية قاعدة الكود في السياق دون تضمين كل المصدر.
- **سجل الدردشة**: محادثة كاملة في السياق.
- **اختيار الملفات**: يطلب LLM ملفات محددة عبر بناء جملة — فقط محتويات تلك الملفات تُضاف للسياق.
- **إدارة نافذة السياق**: عند الاقتراب من حد السياق، يُسقط Aider أدوار المحادثة الأقدم ويضمّن الملخصات.
- **لا ذاكرة طويلة المدى**: بدون حالة بين الجلسات. كل إطلاق `aider` يبدأ جديدًا.

### ميزات فريدة

- **خريطة المستودع**: فهم قاعدة الكود قائم على AST يتفوق على RAG القائم على التضمين لمهام الكود.
- **تنسيقات تحرير متعددة**: يتكيف مع ما يعمل أفضل لكل نموذج (بعض النماذج تفعل أفضل مع udiff، أخرى مع بحث/استبدال، إلخ).
- **وضع المهندس المعماري/المحرر**: عملية من خطوتين باستدعاءات LLM منفصلة للتخطيط والتنفيذ.
- **متعدد اللغات**: 100+ لغة برمجة عبر Tree-sitter.
- **لوائح المتصدرين LLM**: يحتفظ بمعايير قياسية عامة لتحرير الكود عبر النماذج.
- **برمجة صوتية**: كلام-إلى-كود مباشرة في الطرفية.
- **وضع النسخ-اللصق**: يعمل مع واجهات دردشة الويب للنماذج دون وصول API.
- **مدرك لـ Git**: التزامات تلقائية، رسائل التزام معقولة، يعمل مع مستودعات git الموجودة.
- **الكتابة الذاتية**: 88% من كود Aider نفسه كُتب بواسطة Aider.

### فجوات محتملة مقابل أهداف تصميم Entelecheia

- **تركيز على ملف واحد**: يحرر أساسًا ملفًا واحدًا في كل مرة (رغم أن خريطة المستودع توفر السياق).
- **لا نظام متعدد الوكلاء**: فقط زوج المهندس المعماري/المحرر. لا أدوار وكلاء قابلة للتخصيص.
- **لا منظومة أدوات**: لا يمكن استخدام APIs، قواعد البيانات، خدمات الويب — فقط تحرير الملفات والقشرة.
- **لا عزل**: وصول مباشر لنظام الملفات — قوي لكن محفوف بالمخاطر.
- **لا حالة مستمرة**: كل جلسة مستقلة. لا تعلم عبر الجلسات.
- **لا بدائيس تنسيق**: ليس إطارًا — أداة مستقلة.
- **هشاشة تحليل النص**: يعتمد على LLM لإخراج تنسيقات تحرير دقيقة — يمكن أن يفشل عند انحرافات النموذج.
- **طرفية فقط**: غير قابل للتضمين كمكتبة في معظم تدفقات العمل (رغم وجود Python API).

## النتائج الرئيسية لخارطة طريق Entelecheia

1. **الأمان/العزل**: تقريبًا كل الأطر تفتقر للتنفيذ المعزول. Google ADK هو الاستثناء البارز بعزل قائم على الحاويات/Kubernetes. هذه فرصة تمييز كبيرة.

1. **اتصال متعدد الوكلاء**: فقط Google ADK لديه بروتوكول رسمي بين الوكلاء (A2A). معظم الأطر تستخدم تمرير رسائل مخصص. البروتوكول الموحد (مثل A2A) فجوة.

1. **بنى الذاكرة**: معظم الأطر لديها ذاكرة قصيرة المدى أساسية. القليل لديها ذاكرة هرمية متطورة (عاملة، قصيرة المدى، طويلة المدى) مع إدارة سياق مؤتمتة. تصفية MetaGPT ودمج ADK أفضل الأمثلة.

1. **إدارة تعريض الأدوات**: كل الأطر تعرض كل الأدوات لـ LLM لكل استدعاء. لا يوجد إطار يقوم بتحديد مجموعة فرعية من الأدوات ديناميكيًا بناءً على السياق/الحالة/مستوى الأمان. هذه فجوة بنيوية.

1. **تنفيذ الكود**: فقط ADK لديه تنفيذ معزول بجودة إنتاج. ChatDev لديه منفذ كود أساسي. Cline/Aider يعتمدان على البيئة الأصلية. هذه فجوة أمان حرجة عبر المنظومة.

1. **التقييم**: ADK و Cline لديهما أطر تقييم رسمية. البعض الآخر يعتمد على اختبار مخصص أو معايير بحثية. التقييم المضمّن عامل تمييز.

1. **تقادم OpenAI Swarm** إلى OpenAI Agents SDK يشير إلى اتجاه السوق نحو أطر جاهزة للإنتاج على التجارب التعليمية.

1. **التنفيذ المتين لـ LangGraph** قوي بشكل فريد للوكلاء طويلي التشغيل — معظم الأطر تفترض مهام قصيرة العمر.

1. **نهج ChatDev 2.0 بدون كود** يستهدف شخصية مستخدم مختلفة جوهريًا (غير المطورين) عن معظم الأطر. هذا متعامد مع تصميم Entelecheia المطور-أولاً.

1. **Cline و Aider** تطبيقات، وليست أطر. توضح قوة التكامل المحكم للأدوات (IDE، git، طرفية، متصفح) لكن لا يمكن تركيبها في أنظمة وكلاء أكبر.

## الملحق: تذكير الحالة الحالية لـ Entelecheia

هذا الملحق هو طبقة التصحيح الموثوقة للمقارنات أعلاه.

### ما هو نشط اليوم

- 12 وكيل Layer1 مُترجم في مساحة العمل.
- صندوق Layer1 واحد لأتمتة الويب نشط.
- الوكلاء المتخصصون الإضافيون مؤرشفون كمخططات أو وثائق جزئية ولا يجب قراءتهم كوحدات وقت تشغيل مشحونة بالكامل.

### ما هو ناضج نسبيًا

- `packages/scepter`، `packages/shared`، و`packages/tui`
- نموذج تعريض الأدوات تنفيذ فقط
- مسارات التنفيذ المدعومة بالحاويات
- تخزين مفاتيح المزودين المشفّر وأنابيب المصادقة المتعلقة بـ RBAC

### ما لا يزال جزئيًا

- WebUI مقارنة بـ TUI
- تغطية أوامر CLI
- تكاملات سطح المكتب/المحمول (نُقلت إلى [shittim-chest](https://github.com/celestia-island/shittim-chest))
- RAG والذاكرة، التي تعتمد حاليًا على مستندات في الذاكرة، تضمينات قائمة على التجزئة، واجتياز الرسم البياني بدلًا من مكدس ONNX + pgvector مدمج بالكامل
- اكتمال التدقيق وتقوية الحاويات
