
# ADR-003：PostgreSQL + PgVector 统一数据存储

**日期**：2026-02
**状态**：已接受

## 背景

Entelecheia 需要一个数据存储层，同时承担**两个不同职责**：

1. **传统关系型数据**：用户账户、API 凭证、Agent 会话、任务元数据、审计日志、RBAC 策略、容器状态、时间线事件——所有受益于关系完整性、事务和 SQL 查询的结构化数据。

1. **向量相似性搜索**：用于 RAG（检索增强生成）、记忆沉淀、知识图谱遍历和文档检索的嵌入——需要高维最近邻搜索的数据。

评估了以下几种存储方案：

| 方案 | 关系型数据 | 向量搜索 | 双重职责 | 成熟度 |
| --- | --- | --- | --- | --- |
| **PostgreSQL + PgVector** | 完整 SQL、ACID 事务 | HNSW/IVFFlat 索引 | 单一数据库、单一查询语言 | PostgreSQL：35+ 年；PgVector：稳定、广泛部署 |
| **Qdrant** | 无（仅向量） | 优秀（专为此设计） | 需要单独的关系型数据库 | 中等 |
| **Milvus** | 无（仅向量） | 优秀 | 需要单独的关系型数据库 | 中等 |
| **Weaviate** | 有限（内置 CRUD） | 良好 | 两者都妥协 | 中等 |
| **SQLite + 向量扩展** | 完整 SQL | 实验性 | 单一文件、并发有限 | 低（向量扩展不成熟） |
| **MongoDB + Atlas Vector Search** | 文档存储，无 SQL | 良好 | 查询模型妥协 | 高（但向量搜索是专有的） |

## 决策

我们选择 **PostgreSQL 配合 PgVector 扩展** 作为统一存储后端。

**主要原因：**

1. **稳定性和经过验证的可靠性。** PostgreSQL 是经过最充分实战检验的开源关系型数据库引擎，拥有 35+ 年的开发历史、成熟的查询优化器、坚如磐石的 ACID 事务以及庞大的工具生态。对于处理认证凭证、RBAC 策略和审计日志的系统而言，这种稳定性不是可选项——它是基本要求。为这些负载选择一个更新、未经验证的数据库会引入不必要的运行风险。

1. **统一的二元存储。** PgVector 以向量相似性搜索（HNSW 和 IVFFlat 索引）扩展了 PostgreSQL，无需单独的数据库。这意味着：

   - **一个数据库来管理**，一个连接池，一个备份策略，一个迁移系统。
   - **跨越关系数据和向量数据的 JOIN 查询**——例如，"查找与此嵌入相似且用户有权访问的文档"可以表达为单个 SQL 查询。
   - **元数据更新和嵌入插入之间的事务一致性。**
   - **SQL 作为统一的查询语言**——开发者无需学习单独的向量查询 DSL。

1. **PgVector 在我们规模下相对稳定。** 虽然 PgVector 在十亿级部署中不如专为向量设计的数据库优化得好，但它能胜任 Entelecheia 的工作负载（Agent 记忆、知识文档、RAG 上下文）。嵌入维度（768-3072）和数据集大小（数千到低百万级向量）完全在 PgVector 的舒适区域之内。

1. **团队熟悉度和生态系统。** PostgreSQL 是世界上部署最广泛的数据库引擎。团队对 SQL、PostgreSQL 管理和 Rust ORM 生态（SeaORM、SQLx）有深厚熟悉度。选择不熟悉的向量数据库将需要为边际收益付出巨大的学习投入。

1. **不妥协 SQL 兼容性。** 许多较新的向量数据库要么根本不支持 SQL，要么只支持有限方言。这将迫使应用程序维护两种独立的查询模型——一种用于关系数据，一种用于向量搜索——增加代码复杂度和 bug 产生的面。

## 后果

### 积极方面

- **单一运维面**：一个数据库来监控、备份、升级和调试。SeaORM 迁移同时处理 schema 和扩展设置。
- **事务性向量操作**：嵌入插入和元数据更新在同一事务中发生，防止孤立或不一致数据。
- **组合查询的完整 SQL 能力**：带权限感知的向量搜索、按时间过滤的相似性以及多表 JOIN 操作是原生 SQL 操作。
- **SeaORM + PgVector 集成**：Rust 生态拥有成熟的 PostgreSQL 支持。SeaORM 实体可以包含带距离操作符的向量列。
- **生产环境验证**：PostgreSQL 的复制、时间点恢复、连接池（PgBouncer）和监控（`pg_stat_statements`）已是行业标准。

### 消极方面

- **向量搜索性能上限**：对于非常大的嵌入集合（1 亿+ 向量），专用向量数据库（Qdrant、Milvus）显著优于 PgVector。Entelecheia 当前规模并未接近此极限，但这是未来的考量。
- **PgVector 是扩展，非核心**：PgVector 必须与 PostgreSQL 分开安装和维护。容器镜像必须包含该扩展（我们使用 `pgvector/pgvector:pg18`）。升级 PostgreSQL 可能需要重新编译扩展。
- **有限的向量索引类型**：PgVector 支持 HNSW 和 IVFFlat。专用向量数据库提供更多专门索引类型（如 DiskANN、ScaNN），对特定分布可能更高效。
- **资源竞争**：向量索引（尤其是 HNSW 构建）消耗的 CPU 和内存与同一 PostgreSQL 实例上的 OLTP 负载共享。在规模上，将向量负载分离到专用副本可能成为必要。

### 接受的权衡

**以性能上限换取运维简单性。** 双数据库架构（PostgreSQL 用于关系数据 + Qdrant/Milvus 用于向量）在规模下会提供更好的向量搜索性能。然而，它将使运维复杂度翻倍，需要在两个系统间进行数据同步，并引入一致性挑战。对于 Entelecheia 当前和近期的规模（单用户到小团队部署），统一的 PostgreSQL 方案是正确的权衡。如果向量搜索在未来成为瓶颈，可以逐步引入带有 PgVector 的读副本或专用向量缓存层，而无需改变应用程序的查询模型。
