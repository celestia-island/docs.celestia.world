# ADR-002：Boa 作为嵌入式 JavaScript 引擎

**日期**：2026-02
**状态**：已接受

## 背景

IEPL（交互式执行管道层）需要一个 JavaScript 运行时来执行每个 COSMOS 容器内的 LLM 生成代码。此运行时必须：

1. **可嵌入**到 Rust 应用程序中——它将作为轻量容器内的 init 进程（PID 1）运行。
1. **安全且沙箱化**——LLM 生成的代码本质上是不可信的。
1. 支持 **ES 模块导入**以用于工具调度机制（例如 `import { file_read } from 'agent'`）。
1. 具有**最小的启动开销**——容器是短暂的，必须快速就绪。
1. **跨平台**且易于为容器目标编译。

评估了多个 JavaScript/TypeScript 运行时：

| 运行时 | 语言 | 可嵌入 Rust | 沙箱控制 | 启动速度 | ES 模块支持 |
| --- | --- | --- | --- | --- | --- |
| **Boa** | 纯 Rust | 原生（Rust crate） | 完全（host 控制一切） | 快（<10ms） | 部分（对 IEPL 足够） |
| **Deno** | Rust + V8 | 可通过 FFI | 有限（V8 隔离） | 慢（~100ms） | 完全 |
| **QuickJS** | C | 通过 FFI/绑定 | 中等 | 快 | 部分 |
| **V8** | C++ | 通过 FFI（v8 crate） | 有限 | 慢 | 完全 |
| **wasmoon** | C（Lua） | 通过 FFI | 好 | 快 | 不适用（Lua） |
| **rquickjs** | C（QuickJS） | 通过 FFI | 中等 | 快 | 部分 |

## 决策

我们选择 **Boa Engine**（v0.21）作为嵌入式 JavaScript 运行时。

**主要原因：**

1. **纯 Rust——零 FFI 开销。** Boa 完全用 Rust 编写，这意味着它原生编译进 COSMOS 二进制文件，没有 C/C++ 依赖链。这消除了一整类 FFI 相关的安全漏洞、构建复杂性和交叉编译障碍。在 COSMOS 容器上下文中，我们控制 init 进程，这是一个关键优势。

1. **"即调用即用"的简洁性。** Boa 被设计为库优先的引擎。它可以用几行 Rust 代码实例化、配置宿主函数并执行。没有单独进程需要管理，没有 IPC 桥接需要维护，也没有事件循环复杂性。COSMOS 中的 `JsReplHandle` 为 Boa 运行时创建专用 OS 线程，并通过标准 Rust 通道通信——一个干净、可组合的架构。

1. **安全性是最高优先级，而非性能。** 在 COSMOS 沙箱中，每毫秒的 JS 执行时间不在关键路径上——瓶颈始终是 LLM 推理往返。重要的是运行时赋予我们**完全控制**执行代码的能力。Boa 的宿主函数注册 API 让我们精确定义哪些函数可用（仅 MCP 工具调度函数），没有逃逸途径。AST 安全验证器（阻止 `eval`、`require`、`process` 等）在 Boa AST 上操作，为我们提供原生 Rust 强制执行保证。

1. **对 IEPL 足够的 ES 模块支持。** IEPL 管道使用模拟模块系统——ES 模块导入在命名空间构建器级别解析，而非通过真正的模块加载器。Boa 的能力对此模式绰绰有余。我们不需要完整的 Node.js 兼容模块解析算法。

## 后果

### 正面

- **无 C/C++ 构建依赖**——COSMOS 仅需 `cargo build` 即可清晰编译，无需系统级库要求。
- **完全沙箱控制**——执行 JS 代码可用的每个函数都由宿主显式注册。无默认 I/O、网络或文件系统访问。
- **与 Rust 类型紧密集成**——为自定义宿主对象实现 `boa_gc::Trace` trait，原生 `serde_json` 互操作，尽可能零拷贝。
- **崩溃安全**——Boa panic 被 OS 线程边界捕获，防止 COSMOS 进程因格式错误的 JS 代码崩溃。
- **较小的二进制体积**——与基于 V8 的解决方案相比，Boa 显著减少了 COSMOS 二进制大小。

### 负面

- **JavaScript 一致性不完整**——Boa 未完全实现 ECMAScript 2024+。某些高级特性（例如 `WeakRef`、`FinalizationRegistry`、完整的 `Promise` 集成、所有上下文中的 `async/await`）可能存在限制或缺失实现。
- **性能不如 V8/`SpiderMonkey` 有竞争力**——Boa 的解释器比 JIT 编译引擎慢得多。对于 CPU 密集型工作负载（大数据处理、复杂算法），这很重要。然而，在 IEPL 上下文中，JS 代码主要是编排胶水，调用 MCP 工具，而非计算。
- **生态系统较小**——Boa 的贡献者比 V8 或 QuickJS 少，实战测试也较少。上游错误修复可能需要更长时间。
- **无 `eval` 或动态代码生成**——设计上（并由 AST 验证器强制执行），动态代码评估被阻止。这限制了某些元编程模式，但对于安全模型是可接受的。

### 已接受的权衡

**为安全性和可嵌入性牺牲性能。** 如果 IEPL 执行引擎需要运行复杂算法或处理大型数据集，Boa 将是错误的选择。但在 COSMOS 沙箱中，JS 代码是一个薄编排层——其工作是按正确顺序调用 MCP 工具、处理错误和组合结果。实际繁重工作由基于 Rust 的 agent 工具完成。与 V8 相比，Boa 10-100 倍较慢的执行速度在每次工具调用都涉及往返 Scepter 的 50-500ms 网络往返时是无关紧要的。
