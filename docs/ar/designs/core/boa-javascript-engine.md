# ADR-002: Boa كمحرك JavaScript مدمج

**التاريخ**: 2026-02
**الحالة**: مقبول

## السياق

يتطلب IEPL (طبقة خط أنابيب التنفيذ التفاعلي) زمن تشغيل JavaScript لتنفيذ الكود المُولّد بواسطة LLM داخل كل حاوية COSMOS. يجب أن يكون زمن التشغيل هذا:

1. **قابلاً للدمج** داخل تطبيق Rust — سيعمل كعملية init (PID 1) داخل حاويات خفيفة.
1. **آمنًا ومعزولًا** — الكود المُولّد بواسطة LLM غير موثوق بطبيعته.
1. **يدعم استيراد وحدات ES** لآلية توجيه الأدوات (مثل `import { file_read } from 'agent'`).
1. **ببداية تشغيل ضئيلة** — الحاويات مؤقتة ويجب أن تكون جاهزة بسرعة.
1. **عبر منصات** وسهل التجميع للهدف الحاويات.

تم تقييم عدة أزمنة تشغيل JavaScript/TypeScript:

| زمن التشغيل | اللغة | قابل للدمج في Rust | تحكم العزل | سرعة البدء | دعم وحدة ES |
| --- | --- | --- | --- | --- | --- |
| **Boa** | Rust نقي | أصلي (كرات Rust) | كامل (الخادم يتحكم في كل شيء) | سريع (<10ms) | جزئي (يكفي لـ IEPL) |
| **Deno** | Rust + V8 | ممكن عبر FFI | محدود (عزل V8) | بطيء (~100ms) | كامل |
| **QuickJS** | C | عبر FFI/روابط | متوسط | سريع | جزئي |
| **V8** | C++ | عبر FFI (كرات v8) | محدود | بطيء | كامل |
| **wasmoon** | C (Lua) | عبر FFI | جيد | سريع | غير قابل للتطبيق (Lua) |
| **rquickjs** | C (QuickJS) | عبر FFI | متوسط | سريع | جزئي |

## القرار

اخترنا **Boa Engine** (v0.21) كزمن تشغيل JavaScript مدمج.

**الأسباب الرئيسية:**

1. **Rust نقي — صفر عبء FFI.** Boa مكتوب بالكامل بـ Rust، مما يعني أنه يُجمَّع أصليًا في ثنائي COSMOS بدون سلسلة اعتماد C/C++. هذا يلقي فئة كاملة من الثغرات الأمنية المتعلقة بـ FFI، وتعقيد البناء، ومشاكل التجميع المتقاطع. في سياق حاوية COSMOS حيث نتحكم في عملية init، هذه ميزة حرجة.

1. **بساطة "استدعِ واستخدم".** Boa مصمم كمحرك أولًا كمكتبة. يمكن إنشاء مثيل له، تكوينه بدوال الخادم، وتنفيذه في بضعة أسطر كود Rust. لا توجد عملية منفصلة لإدارتها، ولا جسر IPC للحفاظ عليه، ولا تعقيد حلقة أحداث. ينشئ `JsReplHandle` في COSMOS خيط نظام تشغيل مخصص لزمن تشغيل Boa ويتواصل عبر قنوات Rust القياسية — هندسة نظيفة قابلة للتركيب.

1. **الأمان هو الأولوية القصوى وليس الأداء.** في صندوق رمل COSMOS، كل مللي ثانية من وقت تنفيذ JS ليست على المسار الحرج — الاختناق دائمًا رحلة استدلال LLM ذهابًا وإيابًا. ما يهم هو أن زمن التشغيل يمنحنا **تحكمًا كاملًا** فيما يمكن للكود المنفذ فعله. يتيح لنا API تسجيل دوال الخادم في Boa تعريف الدوال المتاحة بدقة (دوال توجيه أداة MCP فقط)، بدون منافذ هروب. يعمل مدقق أمان AST (الذي يحجب `eval`، `require`، `process`، إلخ.) على Boa AST، مانحًا إيانا ضمان Rust أصلي للإنفاذ.

1. **دعم وحدة ES كافٍ لـ IEPL.** يستخدم خط أنابيب IEPL نظام وحدات محاكى — يُحل استيراد وحدات ES على مستوى باني النطاقات، وليس بواسطة محمل وحدات حقيقي. قدرات Boa أكثر من كافية لهذا النمط. لا نحتاج خوارزمية حل وحدات متوافقة كاملة مع Node.js.

## النتائج

### إيجابية

- **لا اعتماد بناء C/C++** — يُجمَّع COSMOS بنظافة مع `cargo build` فقط، لا متطلبات مكتبات على مستوى النظام.
- **تحكم عزل كامل** — كل دالة متاحة لكود JS المنفذ مسجّلة صراحةً من قبل الخادم. لا إدخال/إخراج افتراضي، ولا شبكة، ولا وصول لنظام الملفات.
- **تكامل محكم مع أنواع Rust** — تنفيذ سمة `boa_gc::Trace` لكائنات الخادم المخصصة، توافق `serde_json` أصلي، نسخ صفر حيثما أمكن.
- **سلامة الانهيار** — تُلتقط ذعرات Boa بحدود خيط نظام التشغيل، مما يمنع تعطل عملية COSMOS بسبب كود JS مشوّه.
- **بصمة ثنائية صغيرة** — مقارنةً بالحلول المبنية على V8، يضيف Boa قدرًا أقل بكثير إلى حجم ثنائي COSMOS.

### سلبية

- **توافق JavaScript غير مكتمل** — Boa لا ينفذ ECMAScript 2024+ بالكامل. بعض الميزات المتقدمة (مثل `WeakRef`، `FinalizationRegistry`، تكامل `Promise` الكامل، `async/await` في كل السياقات) قد تكون لها قيود أو تنفيذات مفقودة.
- **الأداء غير تنافسي مع V8/`SpiderMonkey`** — مفسر Boa أبطأ بكثير من المحركات المجمّعة JIT. لأحمال العمل المكثفة CPU (معالجة بيانات كبيرة، خوارزميات معقدة)، هذا يهم. لكن في سياق IEPL، كود JS هو في الأساس غراء تنسيق يستدعي أدوات MCP وليس حسابًا.
- **النظام البيئي أصغر** — Boa لديه مساهمون أقل واختبار في المعارك أقل من V8 أو QuickJS. قد تستغرق الأخطاء وقتًا أطول للإصلاح في المصدر.
- **لا `eval` أو توليد كود ديناميكي** — بالتصميم (ومنفّذ بواسطة مدقق AST)، تقييم الكود الديناميكي محجوب. هذا يحد من أنماط برمحة ميتا معينة لكنه مقبول لنموذج الأمان.

### التنازل المقبول

**التضحية بالأداء مقابل الأمان والقابلية للدمج.** لو احتاج محرك تنفيذ IEPL لتشغيل خوارزميات معقدة أو معالجة مجموعات بيانات كبيرة، لكان Boa خيارًا خاطئًا. لكن في صندوق رمل COSMOS، كود JS طبقة تنسيق رقيقة — وظيفته استدعاء أدوات MCP بالترتيب الصحيح، والتعامل مع الأخطاء، ودمج النتائج. الرفع الثقيل الفعلي يقوم به أدوات الوكلاء المبنية على Rust. سرعة تنفيذ Boa الأبطأ بـ 10-100 مرة مقارنةً بـ V8 غير ذات صلة عندما كل استدعاء أداة يتضمن رحلة شبكة إلى Scepter تستغرق 50-500ms.
