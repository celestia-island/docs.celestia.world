
# ADR-004: 60以上のクレートによる階層化ワークスペースアーキテクチャ

**日付**: 2026-03
**ステータス**: 承認済み

## 背景

Entelecheiaは、すべての共有インフラストラクチャ（型、MCPプロトコル、LLMプロバイダ、コンテナ管理、データベース、セキュリティ、設定など）を含む単一の `packages/shared` クレート（38K行、187個の `.rs` ファイル）から始まった。プロジェクトが12エージェント + 1ドメインエージェント + 3バイナリパッケージに成長するにつれ、いくつかの問題が発生した：

1. **コンパイル時間**: `shared` への変更は、1つの構造体のみが変更された場合でも、全187ファイルの再コンパイルが必要だった。
1. **依存関係の汚染**: MCP型のみを必要とするエージェントクレートが、データベースドライバ、コンテナランタイム、LLMプロバイダに推移的に依存することを強いられた。
1. **所有権の不明確さ**: 1つのクレートに187ファイルがあると、どのモジュールがどの機能を「所有」しているか不明確で、リファクタリングがリスキーだった。
1. **フィーチャーフラグの爆発**: 不要な依存関係を取り込まないためにCargoフィーチャーによる条件付きコンパイルが使用されたが、テスト構成の組み合わせ爆発を引き起こした。

## 決定

モノリシックな `packages/shared` を**37の焦点を絞ったサブクレート**に分解し、**6つの依存関係レイヤー**（L0からL5）に整理する。厳密な依存方向に従う：

```text
L0（リーフ） → L1 → L2 → L3 → L4 → L5 → コンシューマ（scepter, エージェント, tui）
```

**レイヤー定義:**

| レイヤー | クレート | ルール |
| --- | --- | --- |
| **L0** | core, logging, macros | 他のentelecheiaクレートへの内部依存ゼロ |
| **L1** | domain_enums, mcp_types, text, concurrent | L0のみに依存 |
| **L2** | config, agent_registry, state_types | L0-L1に依存 |
| **L3** | domain_agent, container, agent_lifecycle, agent_runtime, thread_types, toolchain, infra_utils | L0-L2に依存 |
| **L4** | state_sync, domain_skills, hooks, domain_auth, container_runtime, skills_permissions, timeline, iepl | L0-L3に依存 |
| **L5** | llm_provider, prompt, custom_agent, storage, infra_jsonrpc, infra_services, e2e_events, adapter, plugin_host, rag, embedding, security_policy | L0-L4に依存 |

すべての内部依存宣言は `workspace = true` を使用してバージョン一貫性を確保する。薄い集約クレートは存在せず、コンシューマは個々のサブクレートから直接インポートする。

## 結果

### ポジティブ

- **増分コンパイル**: `shared-core`（L0）への変更は依然伝播するが、`shared-security-policy`（L5）への変更はそのクレートと直接のコンシューマのみが再コンパイルされる。ビルド時間は大幅に改善された。
- **明確な所有権境界**: 各クレートは焦点を絞った責任を持つ。コードレビューの範囲はクレート境界によって自然に制限される。
- **依存関係の分離**: エージェントクレートは必要な共有クレートのみをインポートする。SkeMmaはデータベースドライバを取り込まない。EleOsはコンテナランタイムを取り込まない。
- **循環依存の防止**: 階層化アーキテクチャにより、循環依存を作ることが構造的に不可能 — L3クレートはL5クレートに依存できない。
- **分離テストが可能**: 各クレートのテストは独立して実行され、ワークスペース全体の依存関係ツリーを必要としない。

### ネガティブ

- **ワークスペース管理のオーバーヘッド**: 単一ワークスペースに60以上のクレートがあるということは、より多くの `Cargo.toml` ファイルの保守、バージョンアップ時のより多くの `[dependencies]` セクションの更新、より慎重な依存宣言が必要となる。
- **クロスクレートリファクタリングの困難さ**: 型をL2からL3に移動するには、すべてのL2コンシューマを更新し、L3以上のクレートが誤って古い場所を通じて移動された型に依存していないことを検証する必要がある。
- **クレート命名の冗長性**: 内部クレート名は `_shared_*` プレフィックス規約（例：`_shared_domain_skills_permissions`）を使用し、冗長だがワークスペースの明瞭さのために必要である。
- **過剰分解の可能性**: 一部のクレート（例：`shared-text`、約200行）は、独自のクレートオーバーヘッドを正当化しないかもしれない。分解は厳密な必要性ではなく「成長する可能性があれば分離する」という哲学に従った。

### 受け入れたトレードオフ

**コンパイル時間とアーキテクチャの明瞭さのための管理複雑性。** `shared` の37クレート分解は、Rustワークスペース設計の中でも積極的な方である。中間点（10〜15クレート）の方が管理が簡単だっただろう。しかし、プロジェクトの広範な表面積（26のLLMプロバイダ、2つのコンテナランタイム、12エージェント、完全なセキュリティパイプライン、データベース、IEPL）を考慮すると、細粒度の分解により各部分が独立して進化できるようになる。`workspace = true` パターンはバージョン管理のオーバーヘッドを軽減する。
