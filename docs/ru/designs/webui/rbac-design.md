# Документ детального дизайна системы RBAC

## 1. Цели

Реализация полной системы управления доступом на основе ролей для Shittim Chest с поддержкой:

- **Управление пользователями**: Администратор может приглашать/создавать/отключать/удалять пользователей
- **Управление группами**: Поддержка групп учётных записей, пользователи могут принадлежать нескольким группам
- **Детальные права**: Контроль возможности пользователей добавлять/изменять/использовать конкретных провайдеров моделей, инструменты MCP, агентов Layer3, каналы IM и т.д.
- **Функциональные переключатели**: Контроль возможности пользователей использовать продвинутые функции, такие как режим автоматического круиза
- **Гибкие режимы авторизации**: Администратор может выбрать глобальную унифицированную конфигурацию, индивидуальную настройку для каждой учётной записи или общий доступ для групп учётных записей

## 2. Основные концепции

### 2.1 Роль (Role)

| Роль | Описание |
| --- | --- |
| `admin` | Суперадминистратор, имеет все права, может управлять самой RBAC |
| `operator` | Операционный персонал, может управлять большинством ресурсов (провайдеры, каналы, агенты и т.д.) |
| `member` | Обычный участник, может использовать авторизованные ресурсы |
| `viewer` | Только чтение, может просматривать, но не изменять |

Роли **предустановлены**, пользовательские роли не предоставляются (упрощённая реализация). Каждый пользователь может иметь одну основную роль.

### 2.2 Права (Permission)

Формат прав: `<ресурс>.<действие>`

| Категория | Право | Описание |
| --- | --- | --- |

### 2.4 Режимы авторизации

Для ресурсов, таких как провайдеры, MCP, агенты, каналы, поддерживаются три режима авторизации:

| Режим | Описание | Подходящий сценарий |
| --- | --- | --- |
| **Глобальная конфигурация** | Все пользователи разделяют одинаковые права | Маленькие команды, личное использование |
| **Конфигурация по пользователям** | Каждый пользователь имеет независимые права на ресурсы | Сценарии, требующие тонкого контроля |
| **Конфигурация по группам** | Пользователи одной группы разделяют права | Разделение по отделам/командам |

Администратор выбирает режим авторизации на странице «Матрица прав», затем настраивает конкретные правила разрешения/запрета.

**Приоритет**: Конфигурация по пользователям > Конфигурация по группам > Глобальная конфигурация > Права роли по умолчанию

## 3. Схема базы данных

### 3.1 Новые таблицы

#### `rbac_groups` — Группы пользователей

```sql
CREATE TABLE rbac_groups (
    id          UUID PRIMARY KEY,
    name        VARCHAR(64) NOT NULL UNIQUE,
    description TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `rbac_user_groups` — Связь пользователь-группа

```sql
CREATE TABLE rbac_user_groups (
    id         UUID PRIMARY KEY,
    user_id    UUID NOT NULL REFERENCES auth_users(id) ON DELETE CASCADE,
    group_id   UUID NOT NULL REFERENCES rbac_groups(id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(user_id, group_id)
);
```

#### `rbac_grants` — Предоставление прав (унифицированная таблица)

```sql
CREATE TABLE rbac_grants (
    id           UUID PRIMARY KEY,
    -- Цель авторизации (выбор одного из трёх)
    scope        VARCHAR(16) NOT NULL, -- 'global' | 'group' | 'user'
    user_id      UUID REFERENCES auth_users(id) ON DELETE CASCADE,
    group_id     UUID REFERENCES rbac_groups(id) ON DELETE CASCADE,
    -- Право
    permission   VARCHAR(64) NOT NULL, -- напр. 'provider.use', 'yolo.use'
    resource_id  VARCHAR(128),         -- Опционально: ограничение конкретным ресурсом (имя провайдера, id канала и т.д.), NULL означает все ресурсы категории
    -- Тип предоставления
    granted      BOOLEAN NOT NULL DEFAULT TRUE, -- TRUE=разрешить, FALSE=запретить
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    -- Ограничение: scope и соответствующий FK должны совпадать
    CONSTRAINT rbac_grants_scope_check CHECK (
        (scope = 'global' AND user_id IS NULL AND group_id IS NULL) OR
        (scope = 'user'   AND user_id IS NOT NULL AND group_id IS NULL) OR
        (scope = 'group'  AND user_id IS NULL AND group_id IS NOT NULL)
    )
);
CREATE INDEX idx_rbac_grants_user ON rbac_grants(user_id);
CREATE INDEX idx_rbac_grants_group ON rbac_grants(group_id);
CREATE INDEX idx_rbac_grants_permission ON rbac_grants(permission);
```

### 3.2 Изменение существующих таблиц

#### Добавление поля роли в `auth_users`

```sql
ALTER TABLE auth_users ADD COLUMN role VARCHAR(16) NOT NULL DEFAULT 'member';
-- Миграция: пользователи с is_admin=true устанавливаются как 'admin'
UPDATE auth_users SET role = 'admin' WHERE is_admin = TRUE;
```

Поле `is_admin` сохраняется для совместимости, но новый код предпочитает использовать `role`.

### 3.3 Логика проверки прав (псевдокод)

```rust
fn has_permission(user, permission, resource_id=None) -> bool {
    // 1. Роль admin проходит напрямую
    if user.role == "admin" { return true; }

    // 2. Сбор всех совпадающих grants, сортировка по приоритету
    let grants = [];

    // 2a. Права роли по умолчанию (самый низкий приоритет)
    grants.push(role_defaults(user.role, permission));

    // 2b. Глобальная конфигурация
    grants.extend(query_grants(scope="global", permission, resource_id));

    // 2c. Конфигурация групп (все группы пользователя)
    for group in user.groups:
        grants.extend(query_grants(scope="group", group_id=group.id, permission, resource_id));

    // 2d. Конфигурация уровня пользователя (самый высокий приоритет)
    grants.extend(query_grants(scope="user", user_id=user.id, permission, resource_id));

    // 3. По приоритету: user > group > global > role_default
    // Внутри одного scope, denied имеет приоритет над granted
    // Любой denied в user scope → отказ
    // Любой denied в group scope → отказ (если нет granted в user scope)
    // Финальный результат
    resolve_grants(grants)
}
```

## 4. Дизайн API

### 4.1 Управление пользователями (`/api/rbac/users`)

| Метод | Путь | Право | Описание |
| --- | --- | --- | --- |
| GET | `/api/rbac/users` | `rbac.manage` | Список всех пользователей (включая роль, группы) |
| POST | `/api/rbac/users` | `rbac.manage` | Пригласить пользователя (отправить email или создать учётную запись) |
| PUT | `/api/rbac/users/:id` | `rbac.manage` | Обновить роль пользователя, включить/отключить |
| DELETE | `/api/rbac/users/:id` | `rbac.manage` | Удалить пользователя |

### 4.2 Управление группами (`/api/rbac/groups`)

| Метод | Путь | Право | Описание |
| --- | --- | --- | --- |
| GET | `/api/rbac/groups` | `rbac.manage` | Список всех групп |
| POST | `/api/rbac/groups` | `rbac.manage` | Создать группу |
| PUT | `/api/rbac/groups/:id` | `rbac.manage` | Обновить группу (название, описание) |
| DELETE | `/api/rbac/groups/:id` | `rbac.manage` | Удалить группу |
| POST | `/api/rbac/groups/:id/members` | `rbac.manage` | Добавить участника |
| DELETE | `/api/rbac/groups/:id/members/:userId` | `rbac.manage` | Удалить участника |

### 4.3 Управление правами (`/api/rbac/grants`)

| Метод | Путь | Право | Описание |
| --- | --- | --- | --- |
| GET | `/api/rbac/grants` | `rbac.manage` | Список всех правил прав (поддержка фильтрации ?scope=&permission=) |
| PUT | `/api/rbac/grants` | `rbac.manage` | Пакетная установка прав (передача полного списка правил, перезапись правил соответствующего scope) |
| DELETE | `/api/rbac/grants/:id` | `rbac.manage` | Удалить одно правило |

### 4.4 Проверка прав (`/api/rbac/check`)

| Метод | Путь | Право | Описание |
| --- | --- | --- | --- |
| GET | `/api/rbac/check?permission=xxx&resource_id=yyy` | (любой аутентифицированный пользователь) | Проверить, имеет ли текущий пользователь указанное право |
| GET | `/api/rbac/my-permissions` | (любой аутентифицированный пользователь) | Вернуть список всех действующих прав текущего пользователя |

### 4.5 Модификация видимости ресурсов

Существующие API ресурсов требуют добавления фильтрации прав:

- `GET /api/chat/providers` → возвращать только провайдеров, на которых у текущего пользователя есть право `provider.list`, и показывать только модели с правом `provider.use`
- `GET /api/channel` → возвращать только каналы с правом `channel.list`
- Перед запуском режима круиза → проверять право `yolo.use`

## 5. Дизайн фронтенда (Malkuth)

### 5.1 Рефакторинг RbacView

Разделён на три вкладки:

#### Вкладка 1: Управление пользователями

- Таблица списка пользователей: аватар, имя пользователя, email, роль (выпадающий переключатель), теги групп, статус (активен/отключён), действия
- Кнопка приглашения пользователя → всплывающее модальное окно (ввод имени пользователя/email/пароля, выбор роли)
- Действия со строками: редактирование роли, отключение/включение, удаление

#### Вкладка 2: Управление группами

- Таблица списка групп: название, описание, количество участников, действия
- Создание группы → всплывающее модальное окно
- Нажатие на группу → развернуть список участников, можно добавлять/удалять участников

#### Вкладка 3: Матрица прав

- Выбор режима авторизации в левом верхнем углу: Глобальный / По группам / По пользователям
- После выбора группы или пользователя отображается таблица матрицы прав:
  - Строки: категории ресурсов (провайдеры, MCP, агенты, каналы, режим круиза...)
  - Столбцы: действия (список, создание, изменение, удаление, использование)
  - Ячейки: трёхпозиционный переключатель (✅ Разрешить / ❌ Запретить / ➖ Наследовать по умолчанию)
- Детальный контроль конкретных ID ресурсов (например, разрешить использование только определённого провайдера)

### 5.2 Контроль прав навигации

- Элементы боковой панели динамически показываются/скрываются в соответствии с правами текущего пользователя
- Защита маршрутов добавляет проверку прав, при отсутствии прав перенаправляет на страницу 403
- Кнопки действий (например, «Добавить провайдера») показываются/скрываются в соответствии с правами

## 7. Соображения безопасности

- Права роли `admin` не могут быть переопределены через `rbac_grants` (жёстко закодированный пропуск)
- Проверка прав выполняется единообразно на уровне промежуточного ПО, не полагаясь на ручную проверку в бизнес-коде
- Чувствительные операции (удаление пользователя, изменение прав) записываются в аудиторский журнал
- JWT содержит только роль, конкретные права запрашиваются из БД в реальном времени каждый раз (избегая ситуации, когда токен не обновлён после изменения прав)
